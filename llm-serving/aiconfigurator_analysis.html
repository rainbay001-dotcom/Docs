AIConfigurator Deep Dive

How NVIDIA's configuration-optimization tool turns kernel microbenchmarks into a searchable LLM-serving cost model — with architecture, data organization, and granularity analysis

AIConfigurator (ai-dynamo/aiconfigurator, paper arXiv 2601.06288, NVIDIA, January 2026) is an offline tool that searches the LLM-serving configuration space — parallelism degrees, worker counts, batch sizes, framework-specific flags — and picks a deployment config optimized for your model, GPU, and latency SLA. It's marketed as "lightning-fast" because it completes a search in ~30 seconds, reporting up to 40% throughput gains on dense models (Qwen3-32B) and 50% on MoE (DeepSeek-V3) over manual tuning.

The trick that makes 30-second search possible is a carefully-built kernel-level performance database, collected once per hardware/framework pair on real GPUs, and queried by a cost model at search time. No GPU is touched during search. This document analyzes the design of that database and the profiling pipeline that produces it, based on direct reading of github.com/ai-dynamo/aiconfigurator at commit 3a1612e.

1. Paper in One Screen

Title: AIConfigurator: Lightning-Fast Configuration Optimization for Multi-Framework LLM Serving  —  Xu et al., arXiv 2601.06288, 2026-01-09.

Authors: Tianhao Xu, Yiming Liu, Xianglong Lu, Yijia Zhao, Xuting Zhou, Aichen Feng, Yiyi Chen, Yi Shen, Qin Zhou, Xumeng Chen, Ilya Sherstyuk, Haorui Li, Rishi Thakkar, Ben Hamm, Yuanzhe Li, Xue Huang, Wenpeng Wu, Anish Shanbhag, Harry Kim, Chuan Chen, Junjie Lai. NVIDIA.

Three stated contributions:

  1. A methodology that decomposes inference into analytically modelable primitives — GEMM, attention, communication, memory — while capturing framework-specific scheduling dynamics.
  2. A calibrated kernel-level performance database for these primitives across hardware (H100, H200, B200) and popular open-weight models (GPT-OSS, Qwen, DeepSeek, Llama, Mistral).
  3. An abstraction layer that resolves optimal launch parameters for the target backend (TRT-LLM, vLLM, SGLang), seamlessly integrating into production orchestration.

Headline numbers:

  • Up to +40% performance for dense models (Qwen3-32B).
  • Up to +50% for MoE (DeepSeek-V3).
  • Search completes in ~30 seconds on average.
  • Accuracy stays within ~10–15% of real-machine measurements.

2. End-to-End Pipeline

Three temporally separate phases. The real-hardware phase runs once when a new GPU or framework version lands; the CSV artifacts ship in the repo via Git LFS (which is why a shallow clone still weighs 1.4 GB); the configurator only ever reads the CSVs.

3. Role of Real-Machine Profiling

Real-machine profiling plays three specific roles:

3.1 One-time calibration input

The CSVs are built once per (GPU model, framework, framework version) and shipped with the tool. End users never run profiling at search time. The repo README explicitly encourages reusing a nearby version (e.g. TRT-LLM 1.0.0rc3 data for deploying 1.0.0rc4) rather than re-profiling — small version drift doesn't move the cost model significantly.

3.2 Per-kernel primitive measurement

Each kernel category has a purpose-built collector that invokes the kernel directly, in isolation, with representative problem sizes:

  • GEMM across all quantization modes: fp16 / bf16 / fp8 / fp8_block / nvfp4 / int8_wo / int4_wo
  • Attention for both phases: context FMHA (prefill), generation FMHA (decode); MHA / GQA / MLA
  • MoE compute + dispatch/combine as separate ops
  • Communication: custom AllReduce, NCCL AllReduce/AllGather/ReduceScatter/AllToAll, P2P send/recv
  • Memory: scale-matrix ops, element-wise (embedding, layer-norm)

Each row is indexed by problem-size dimensions: GEMM by (m, n, k), attention by (batch, seq_len, num_heads, head_size), communication by (message_size, num_gpus). The DB is the closure of these primitives under a realistic problem-size grid.

3.3 Search-time composition — the cost model

Given a candidate configuration, the configurator:

  1. Decomposes the model's forward pass into its per-layer op sequence (forward = attention + MLP + norm + residual, MoE = per-expert routing + expert compute).
  2. Queries the profile DB for each op with the candidate config's problem dimensions.
  3. Interpolates with scipy.interpolate.griddata (linear/cubic) when a specific (m, n, k) or (batch, seq_len) wasn't directly profiled.
  4. Sums op latencies, overlaying scheduling logic: in-flight batching for aggregated serving, prefill/decode split timing for disaggregated serving.
  5. Falls back to a roofline bound from the per-GPU YAML spec when the DB is too sparse at the query point.

This is why the search is seconds rather than hours — the "execution" of a candidate config is a lookup-and-interpolate walk over the DB, not a launch-and-measure cycle on a real GPU.

4. How the Profile Data Is Organized

4.1 Directory layout

src/aiconfigurator/systems/
├── h100_sxm.yaml                   ← hardware speed-of-light spec
├── h200_sxm.yaml
├── b200_sxm.yaml
└── data/
    ├── h100_sxm/
    │   ├── trtllm/1.0.0rc3/
    │   │   ├── gemm_perf.txt
    │   │   ├── context_attention_perf.txt      (prefill attention)
    │   │   ├── generation_attention_perf.txt   (decode attention)
    │   │   ├── context_mla_perf.txt            (DeepSeek MLA prefill)
    │   │   ├── generation_mla_perf.txt         (DeepSeek MLA decode)
    │   │   ├── mla_bmm_perf.txt
    │   │   ├── moe_perf.txt
    │   │   ├── custom_allreduce_perf.txt
    │   │   └── scale_matrix_perf.txt
    │   ├── vllm/{0.12.0, 0.14.0, 0.19.0, ...}/
    │   └── sglang/0.5.6.post2/
    ├── nccl/2.27.3/                  ← framework-agnostic comms
    ├── h200_sxm/, b200_sxm/, ...     ← other systems

Every leaf is a version-stamped directory so that framework updates don't invalidate prior profiles. The nccl/ tree holds framework-independent communication measurements.

4.2 File format

Plain CSV text — no binary, no parquet, no database. Each row is one kernel measurement.

Representative GEMM row:

framework,version,device,op_name,kernel_source,gemm_dtype,m,n,k,latency[,power,power_limit]
TRTLLM,1.0.0rc3,NVIDIA H100 80GB HBM3,gemm,trt_flow_/weight_only_quant_matmul_L257/...,int8_wo,8192,65536,4096,4.875

Representative attention row (context):

framework,version,device,op_name,attention_type,gemm_dtype,kv_cache_dtype,quant_mode,
    num_heads,head_size,seq_len,batch_size,latency[,power,power_limit]

Representative communication row (custom all-reduce):

framework,version,device,op_name,comm_quant_mode,num_gpus,size_in_bytes,latency

Row counts per file: O(100)–O(500) for GEMM; O(50)–O(200) for attention; O(10)–O(50) for communication.

4.3 In-memory indexing

After CSV load, the code (src/aiconfigurator/perf_database.py) builds nested-dict lookup tables:

GEMM:          quant_mode → m → n → k → latency
Attention:     quant_mode → kv_dtype → num_kv_heads → head_size →
               window_size → seq_len → batch_size → latency
Comm:          comm_quant_mode → num_gpus → message_size → latency
MoE:           quant_mode → num_experts → top_k → num_tokens → latency

Multi-dimensional queries that fall between grid points go through scipy.interpolate.griddata. The nested-dict structure makes exact hits O(1) and interpolation lookups O(log n) in the dimension space.

4.4 Per-GPU YAML spec

Alongside the measurement data, each GPU has a YAML of hardware properties used for roofline bounds when the DB is sparse at a query point:

# src/aiconfigurator/systems/h100_sxm.yaml
gpu:
  mem_bw: 3350000000000            # 3.35 TB/s
  float16_tc_flops: 989000000000000  # 989 TFLOPS (fp16 TC)
  power: 700                        # Watts max
  sm_version: 90
node:
  num_gpus_per_node: 8
  intra_node_bw: 450000000000       # bytes/s per GPU
  inter_node_bw:  50000000000       # bytes/s per GPU (multi-node)
misc:
  nccl_version: "2.27.3"

4.5 Version management

Each collector module declares __compat__ metadata (e.g. "trtllm>=1.1.0"), and the configurator at search time resolves the best matching dataset via version_resolver.py using packaging.version. When frameworks ship new kernels, a new sibling collector file collect_<op>_v2.py is added alongside the old one, each with its own version range.

5. Profiling Granularity: Kernel vs Module

The default is kernel-level microbenchmarking — every primitive is invoked in isolation with dummy weights and activations. One exception exists: MLA attention (DeepSeek), which is profiled at the module level because separating it into independent GEMMs would miss the projection/attention choreography.

Operation Granularity Collector file How it's profiled
GEMM kernel collect_gemm.py Direct tensorrt_llm._torch.modules.linear.Linear call per (m, n, k, quant_mode)
Attention (MHA/GQA) kernel collect_attn.py Standalone FMHA context / generation kernel per (batch, seq_len, num_heads, head_dim)
MoE kernel collect_moe_v*.py Expert compute + dispatch/combine as separate primitives
MLA / DSA (DeepSeek-V3) module collect_mla_module.py Mock single-layer DeepseekV3DecoderLayer with dummy weights; extract attention module; time forward pass
Communication micro-bench collect_comm.sh + NCCL tests Standard nccl-tests + custom collective over (msg_size, num_gpus, quant_mode). Not run in model context.
Memory / element-wise kernel collect_computescale.py, collect_gdn.py Per-op direct invocation

5.1 Why mostly kernel-level

The cost model assumes you can sum per-kernel times (with a scheduling overlay). That holds if and only if each kernel's latency is reproducible independent of model context. Microbenchmarking guarantees independence: no neighboring kernel's cache state or DRAM contention contaminates the measurement. Trace-based profiling, by contrast, couples kernel time to surrounding state and breaks the sum-and-interpolate abstraction.

5.2 Why MLA is the exception

From collect_mla_module.py: "Profiles the complete attention module forward pass (projections + attention + output), not just the bare attention kernel."

MLA has multi-stage down/up projections fused with FMHA; separating them into independent GEMMs loses accuracy because projection math + attention math share reshape/transpose choreography whose cost is not captured by the bare kernels. The MLA collector builds a mock single-layer DeepSeek decoder with dummy weights, extracts the attention module, and times .forward(). Same output format, coarser granularity.

5.3 Never end-to-end

No collector ever runs the full model and attributes time from traces. End-to-end profiling would be significantly more expensive, model-specific, and brittle to production variance. Every DB row is a standalone primitive.

6. Measurement Mechanics

What makes the kernel microbenchmarks trustworthy enough to drive production config decisions:

6.1 Hardware preparation (from collector/README.md)

  • Persistent mode: sudo nvidia-smi -pm 1
  • Clock lock: sudo nvidia-smi -ac <mem_freq>,<sm_freq> to prevent frequency drift
  • Single-tenant node — no other workloads sharing the GPU
  • Adequate cooling — collector also detects throttle events post-hoc and flags the row

6.2 Runtime mechanics (from helper.py)

  • benchmark_with_power wraps every measurement: 3 warmup iterations, 6 timed runs with repeat_n=5 replays for GEMM.
  • CUDA graph capture: torch.cuda.graph captures the launch sequence once and replays it, eliminating launch overhead from each measurement.
  • Graceful fallback: allow_graph_fail=True for kernels that can't be graph-captured (e.g. dynamic-shape ops).
  • Event-based timing: torch.cuda.Event with elapsed_time for GPU-side precision (not CPU wall clock).
  • Optional power measurement: driven by COLLECTOR_MEASURE_POWER=1, reads via NVML; enables adaptive num_runs to guarantee min measurement duration.
  • Weight caching: _weight_cache keyed by (gemm_type, n, k) so all batch sizes for the same weight shape reuse the same torch.randn allocation — cold-cache noise removed, and the test cases are grouped by (n, k) and sorted largest-first for cache reuse.

6.3 Concurrency

  • File-level locks on CSV outputs (perf_filename + ".lock") allow parallel collector workers to write concurrently.
  • One row at a time per worker; no partial-write corruption.

7. At Search Time: Composing Predictions

The configurator's search loop for a given (model, GPU, SLA) target:

  1. Enumerate candidate configs. Parallelism combinations (TP/PP/EP), worker counts, batch sizes, KV-cache allocations, CUDA-graph on/off, prefill/decode split ratios. The space is pruned by feasibility checks (memory fit, head divisibility).
  2. Decompose each candidate into op calls. For a given model architecture, the per-layer op sequence is known statically: [rmsnorm, qkv_gemm, rope, attention, o_gemm, rmsnorm, gate_gemm, up_gemm, activation, down_gemm, allreduce, residual] × num_layers.
  3. Look up each op in the DB. Build the problem-size key for this candidate, hit the nested dict, interpolate with scipy.interpolate.griddata if off-grid.
  4. Compose total latency. Sum per-op times; layer in-flight batching overlap (comp-comm or comp-dispatch); for disaggregated serving, compute prefill / decode / KV-transfer components separately.
  5. Apply roofline bound. Clamp predictions to the speed-of-light number derived from the YAML spec (memory BW / flops / power-limited).
  6. Score against the SLA. Evaluate throughput at TTFT/TPOT targets; check latency percentile feasibility.
  7. Return best config. Output YAML / JSON fed directly to Dynamo orchestrator.

Published accuracy vs real measurement: ~10–15% for the configurations the paper evaluated. Within the search, the ranking among candidates matters more than absolute accuracy, and ranking is more robust than point predictions.

8. Comparison to Related Work

System Input signal Cost model style Serving scope
AIConfigurator Kernel-level microbench CSVs per (GPU, framework, version) Nested-dict lookup + scipy griddata interpolation + roofline fallback Full disaggregated / aggregated inference config
SimAI (NSDI '25) ASTRA-sim + ns-3 discrete-event network simulation Event-driven simulator, NCCL latency floors + calbusbw formulas Training + (1.5) inference with Vidur scheduler
xPU-simulator ml_predictor.py Optional ProfilingDB, RandomForest per op category ML regression (matmul / attention / comm / elementwise) Single-node inference cost model
Vidur (MLSys '24) Profile + predictor for LLM operator timings Similar spirit: profile-driven, per-op prediction Inference request scheduling
Roofline Peak FLOPS + memory BW only Analytical min(compute, memory) bound Coarse capacity planning

AIConfigurator sits between roofline (too coarse) and full simulation (too slow). Its distinguishing move is the bet that a dense measured grid + interpolation beats both ML regression (needs training) and event-driven simulation (needs a simulator).

9. Takeaways for xPU-simulator and SMAP

9.1 For xPU-simulator's ML predictor

The existing backends/ml_predictor.py (absorbed from SimAI in 2026-04-14) trains RandomForests per op category. AIConfigurator's CSV + griddata design is a strong alternative:

  • Interpretable — each prediction traces back to nearby measured points.
  • No training step — the table is the model.
  • Framework version as a first-class key — captures kernel improvements across versions that RandomForest trained on pooled data would smooth over.
  • Cost: requires a disciplined collection pipeline (see §6), which ML-from-historical-traces avoids.

Hybrid worth considering: use AIConfigurator-style CSV lookup when the profile DB covers the query point, fall back to the ML predictor when out-of-distribution.

9.2 For SMAP's memory-tier decisions

Less directly applicable (AIConfigurator is a compute/latency tool, SMAP is a memory placement tool), but the design principle transfers: profile the primitives once, compose predictions cheaply. SMAP already does this in spirit with ACTC counter collection; the profile-DB idea could apply to page-migration cost (per-node, per-page-size, per-hotness-class) instead of re-running experiments every time a migration decision is needed.

9.3 Reusable design patterns

  • Version-stamped data directories that survive framework upgrades.
  • YAML hardware specs alongside measurements for roofline fallback.
  • Plain CSV for human inspectability — debug and diff trivially.
  • One-time calibration + runtime composition as the core speedup.
  • Strict measurement hygiene: clock lock, CUDA graphs, weight cache, event-based timing.
