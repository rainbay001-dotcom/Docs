SimAI Deep Dive

How Alibaba's LLM training simulator uses framework-hijacking profiling plus NCCL interception to achieve 98% simulation fidelity — and how this differs sharply from NVIDIA AIConfigurator

SimAI (aliyun/SimAI, NSDI '25 Spring) is a unified simulator built by Alibaba Cloud to predict LLM training performance end-to-end — compute, collective communication, and network — at scales from a single host to thousands of GPUs. Its published accuracy is 98.1% on average vs real cluster runs, a number achieved by two design choices that are unusual compared to other simulators: hijacking training frameworks to generate workloads on a single host, and hijacking NCCL to reproduce its exact algorithm selection.

This document analyzes how real-machine profiling feeds into SimAI, where the profile data lives, how it's exploited at simulation time, and compares the approach head-to-head with NVIDIA AIConfigurator. Source: direct reading of /Users/ray/Documents/Repo/SimAI and cross-check with the NSDI'25 paper.

1. Paper in One Screen

Title: SimAI: Unifying Architecture Design and Performance Tuning for Large-Scale Large Language Model Training with Scalability and Precision.

Authors: Xizheng Wang, Qingxu Li, Yichi Xu, Gang Lu, Dan Li, Li Chen, Heyang Zhou, Linkang Zheng, Sen Zhang, Yikai Zhu, Yang Liu, Pengcheng Zhang, Kun Qian, Kunling He, Jiaqi Gao, Ennan Zhai, Dennis Cai, Binzhang Fu. Alibaba Cloud + Tsinghua + Zhongguancun Lab + SCUT. NSDI '25 Spring.

Core claim: existing simulators operate at one granularity (flow-level for capacity planning, packet-level for performance tuning), which creates fidelity gaps. SimAI unifies both in one stack by combining high-fidelity sub-components:

  1. SimAI-WG (workload generator): hijacks Megatron / DeepSpeed on a single host to generate fine-grained workload files for the target model.
  2. SimAI-CP (compute simulator): records submodule execution times on real GPUs into an operation database; replays them at simulation time.
  3. SimAI-CM (communication simulator): integrates a modified NCCL (SimCCL) to reproduce algorithm selection; drives either NS3 packet-level or analytical bus-BW network simulation.

Headline numbers from the paper:

  • Average fidelity: 98.1% (deviation ~1.9%) vs real cluster runs.
  • Submodule-level simulation precision: 96.9%–99.5%.
  • Scales from lab (<10 GPUs) to production (>10k GPUs).
  • GPU-scaling baseline (SimAI-CP-Model): ~25% deviation when extrapolating to unreleased GPUs from A100 specs.

2. SimAI Architecture

Real-hardware profiling happens once per (model, config, GPU) in AICB's workload generator. The resulting timings fill in the workload file. At simulation time, the astra-sim core replays those times for compute, defers to SimCCL for collective communication, and optionally goes down to NS3 for packet-level network simulation.

3. The "Hijacking" Idea

SimAI's design is built around a concept the authors call "hijacking" — intercepting real framework/library calls on a single host so that the code path matches a real multi-GPU cluster run, except you only need one GPU (or none) to capture it.

3.1 Framework hijacking (workload generation)

Instead of writing an LLM trainer inside the simulator, SimAI runs an actual Megatron-LM or DeepSpeed forward/backward pass — but with parallel group objects mocked so it all executes on one host. The call graph through torch.nn matches what would happen on 1024 GPUs, just with fake peers on the communication calls. This eliminates modeling error from "our simulator's notion of the forward pass" being different from "what Megatron actually does."

See /Users/ray/Documents/Repo/SimAI/aicb/workload_generator/SimAI_training_workload_generator.py for the generator that captures the per-submodule sequence and timings.

3.2 NCCL hijacking (collective simulation)

Similar trick for communication. libhacknccl.so intercepts ncclCommInitAll and the core collective entry points. It constructs virtual communicators that pretend to live in a multi-GPU cluster, drives NCCL's real algorithm-selection logic (which protocol, which tree/ring, how many channels), and emits peer-to-peer operation lists. Those lists are then consumed by the simulator.

The upside of hijacking: your simulation is algorithmically identical to production, to the extent that NCCL versions match. The downside: upgrading NCCL requires re-characterizing the interception surface.

4. Compute Profiling: Submodule-Level via AICB

4.1 What gets measured

AICB instruments the hijacked framework with torch.cuda.Event timing wrappers (/Users/ray/Documents/Repo/SimAI/aicb/utils/utils.py:499-512). For each submodule invocation during a mocked training iteration, it records:

  • attention_forward
  • attention_backward
  • mlp_forward
  • mlp_backward
  • grad_forward, grad_backward
  • Plus per-model add-ons (e.g., DeepSeek's MLA components, router/expert timings for MoE).

These are submodule-level measurements — an entire attention block, not individual kernels inside it. The resulting file (results/aiob_outputs/{model}-{config}.txt) is section-based:

attention_forward:
    time_gpu_avg: 12345.67
mlp_forward:
    time_gpu_avg: 23456.78
attention_backward:
    time_gpu_avg: 45678.90
...

Values in microseconds, averaged over multiple iterations.

4.2 How the data is keyed

Unlike AIConfigurator (which is keyed on problem shape (m, n, k)), SimAI's data is keyed on submodule name and the enclosing workload configuration: model hyperparameters, parallelism degrees, micro-batch size are all baked into the file name, not lookup keys. Each text file is a specific (model, config, GPU) fingerprint.

4.3 How a lookup happens

extract_averages() in aicb/utils/utils.py:373-447 parses the file and returns a dict. Then _get_aiob_compute_time() (SimAI_training_workload_generator.py:55-79) looks up by stage name:

if stage in compute_time_map.keys():
    return compute_time_map[stage]
# Fallback: prefix match (e.g., "attention_forward" for stage="forward")
for key, value in compute_time_map.items():
    if prefix == key:
        return compute_time_map.get(key)

Exact match or prefix match. No interpolation. If you need a config that wasn't profiled, you must run AICB again on that config.

4.4 The fine-grained fallback and unreleased-GPU scaling

The paper's §3.3 describes two supplementary mechanisms:

  • Fine-grained kernel simulation. When new parallelization or kernel-fusion optimizations reorganize submodules, the per-submodule database is stale. A module-kernel converter breaks each submodule down into smaller kernels (GEMM, softmax, layernorm) that are individually measured on the target GPU and summed. This is kernel-level — closer to AIConfigurator's granularity, used as a fallback.
  • SimAI-CP-Model for unreleased GPUs. Uses A100 measurements as baseline, scales by announced spec sheet (compute / bandwidth). Paper reports ~25.1% deviation — explicitly less accurate than true profiling, but the only option when hardware doesn't exist.

So the overall compute story is a three-tier strategy:

Tier Method Accuracy Cost to add a config
Primary Submodule profile via AICB on the actual GPU 96.9–99.5% One actual single-host training run
Secondary Module-kernel converter + per-kernel measurement High (paper doesn't separate) Per-kernel microbenchmark sweep
Fallback SimAI-CP-Model: scale from A100 baseline ~75% (25.1% deviation) Just spec sheet needed

5. Communication Modeling: SimCCL + calbusbw

Communication is where SimAI differs most dramatically from AIConfigurator. AIConfigurator has no network modeling at all; SimAI has two complementary pieces.

5.1 Algorithm selection via SimCCL (hijacked NCCL)

SimCCL (a git branch of SimAI at /Users/ray/Documents/Repo/SimCCL/SimCCL/) is a fork of NCCL patched to emit peer-to-peer operation sequences instead of actually running them. It preserves NCCL's real algorithm-selection table:

  • Chooses tree vs ring vs double-binary-tree per collective.
  • Picks LL vs LL128 vs Simple protocol per message size.
  • Determines number of channels, chunk sizes, connection topology.

The authors argue this is irreplaceable: any guessed collective algorithm in a simulator would be wrong for specific (NCCL version, GPU count, message size) points. Hijacking bypasses the guess.

5.2 Latency modeling: hardcoded calibration constants

Once SimCCL has decomposed a collective into peer-to-peer steps, per-step latency comes from hardcoded analytical constants — analytical, not measured. Two files:

/Users/ray/Documents/Repo/SimAI/astra-sim-alibabacloud/astra-sim/system/MockNccl.h:

static const float baseLat[NCCL_NUM_ALGORITHMS][NCCL_NUM_PROTOCOLS] = {
    {6.8, 14.0, 0},      // Tree:  LL, LL128, Simple
    {6.6, 14.0, 8.4},    // Ring
    ...
};
static const double perChMaxRingLL128Bws[3][3] = {
    {20.0, 20.0, 20.0},  // Volta
    {20.0, 20.0, 20.0},  // Ampere
    {36.7, 36.7, 36.7}   // Hopper (36 GB/s per channel)
};

/Users/ray/Documents/Repo/SimAI/astra-sim-alibabacloud/astra-sim/system/calbusbw.cc:

float getNvlinkBw(GPUType node_type) {
    if (node_type == GPUType::H100)  return 370.8 * 12;  // SM90_NVLINK_BW × H100_NVLINKS
    if (node_type == GPUType::A100)  return 30.0  * 12;  // 360 GB/s
    ...
}
float calcTreeBusBw(int gpus_per_node, int node_count, float nvlink_bw,
                    float nic_bw, float nics_per_node, float all_gather_bus_bw) {
    if (node_count == 1)
        return all_gather_bus_bw * (gpus_per_node-1) / gpus_per_node;
    // else: blend intra-node NVLink and inter-node NIC bandwidths
    ...
}

These values are sourced from vendor specifications and are treated as constants of the simulator — they are not a profile DB. Updating to a new GPU requires editing these files.

5.3 Efficiency correction via ratio tables

Raw peak BW doesn't match real behavior across message sizes. SimAI has CSV ratio tables that map (message_size) to an efficiency fraction:

/Users/ray/Documents/Repo/SimAI/astra-sim-alibabacloud/inputs/ratio/nvlink_ratio.csv:

Size,1 Node(NVLinkx),2Node-16GPU(NIC),4Node-32GPU(NIC),...
16777216,0.45,0.55,0.34,...
33554432,0.55,0.68,0.60,...
67108864,0.68,0.78,0.72,...

These are empirically derived — partial real-machine calibration grafted onto the analytical model. The simulator multiplies peak BW by the ratio at the relevant message size during collective latency computation.

5.4 Optional packet-level simulation via NS3

For workloads where the analytical + ratio model is insufficient (e.g., tail latency or queue behavior under congestion), SimAI routes the peer-to-peer operation stream into ns-3-alibabacloud for full discrete-event packet simulation. Expensive but maximally accurate.

6. Data Organization on Disk

Path Format What it holds Source
results/aiob_outputs/{model}-{config}.txt Section-based text (stage: time_gpu_avg) Per-submodule compute times in µs Real GPU run via AICB
aicb/workload/physical/model_workload/*.csv CSV NCCL communication trace from real training runs; msg_size, stage, count Real cluster run
aicb/workload/simAI/model_workload/*.txt Text Compiled workload with compute times injected AICB → simulator consumer
astra-sim-alibabacloud/inputs/ratio/*.csv CSV (message_size) → BW utilization ratio per topology Empirical calibration
astra-sim-alibabacloud/astra-sim/system/MockNccl.h C++ constants Base NCCL latencies + per-channel max BW per GPU generation Vendor specs
astra-sim-alibabacloud/astra-sim/system/calbusbw.cc C++ formulas NVLink/NIC BW per GPU; tree/ring bus-BW formulas Vendor specs
example/busbw.yaml YAML Per-collective bus-BW overrides (optional manual calibration) User-provided

Note the mix of file formats. SimAI is not a single-source-of-truth DB like AIConfigurator — it's a layered stack where each layer uses whichever format fits the data type: text for time-per-submodule, CSV for communication traces and ratios, C++ headers for physical constants.

7. Simulation-Time Flow

  1. Load workload file (aicb/workload/simAI/*/.txt). This already has compute times filled in (by AICB during generation) plus the communication op sequence (from the framework hijacking).
  2. Parse into Layer objects (astra-sim-alibabacloud/astra-sim/workload/Layer.cc). Each layer carries fwd_pass_compute_time, input_gradient_compute_time, weight_gradient_compute_time — numbers lifted directly from the profile.
  3. For each collective op: hand off to SimCCL to get the peer-to-peer step list. For each step, compute latency from calbusbw.cc BW formula × message size / ratio-table efficiency.
  4. Optionally route peer-to-peer sends/receives into NS3 for packet-level timing. Otherwise use the analytical bus-BW result.
  5. Compute PP bubble via (PP_size - 1) / (GA * vpp) (Layer.cc formula); interleave forward / input-gradient / weight-gradient with communication per the training state machine.
  6. Accumulate total step time → report.

Crucially, compute times are looked up, not recomputed. The simulation is mostly bookkeeping on top of profile-derived numbers. This is why SimAI is fast (single-host simulation of thousand-GPU training runs in minutes) despite claiming packet-level fidelity.

9. When to Use Which

Use SimAI when

  • You need end-to-end training simulation: TPU utilization, pipeline bubble, DP/TP/PP overlap, collective congestion.
  • You need capacity planning for cluster purchases — what fabric topology achieves what throughput.
  • You're willing to pay for a real-hardware profile run per target model/config.
  • You have a specific set of models to benchmark (GPT, DeepSeek, Qwen, Llama variants in AICB); don't need to sweep arbitrary problem shapes.
  • You care about communication / network behavior: SimCCL + NS3 gives packet-level insight AIConfigurator doesn't have.

Use AIConfigurator when

  • You're doing inference serving config search: pick TP/PP/batch/KV-fraction for a Dynamo deployment.
  • You need to explore thousands of configurations in seconds.
  • You care about kernel-shape variation across configs (different sequence lengths, batch sizes produce different GEMM shapes).
  • You're happy with 10–15% absolute error since ranking among configs is what matters.
  • You don't need network modeling (single-node or fabric is already well-understood).

Use both when

  • You're designing a new cluster and want to plan capacity with SimAI then tune each deployment's config with AIConfigurator.
  • Inference vs training: AIConfigurator for serving config, SimAI for training budget / fabric sizing.

10. Takeaways

  1. Profile-driven simulators trade collection cost for accuracy, differently. SimAI collects at submodule granularity per (model, config, GPU) — heavy per-config, but then replays trivially. AIConfigurator collects at kernel granularity per (GPU, framework, version) once — heavy one-time, then covers many configs via interpolation.
  2. Hijacking beats analytical modeling for code paths with many parameters. NCCL is the extreme case — its algorithm selection is so state-dependent that any re-implementation would drift. Hijacking forces algorithmic identity at the cost of coupling to a specific library version.
  3. Granularity matters more than method. Both SimAI and AIConfigurator do real-machine profiling; the difference in how (full framework hijack vs isolated kernel microbench) derives from the difference in what (submodule vs kernel). The "right" granularity is whichever one the consumer's query space operates at.
  4. For xPU-simulator, both approaches offer patterns worth borrowing. The existing backends/ml_predictor.py (a RandomForest predictor) sits between them: more flexible than SimAI's exact lookup, less accurate than AIConfigurator's dense-grid interpolation. A natural next step is a two-tier cost model — SimAI-style submodule lookup when available, AIConfigurator-style kernel interpolation as fallback.
  5. Updating profiles is the operational pain. SimAI needs new AICB runs for each model variant and GPU. AIConfigurator needs new microbench runs for each framework version and GPU. Both are easier if the collection pipeline is robust (clock lock, CUDA graph capture, file-locked CSV outputs — see AIConfigurator's hygiene playbook in that document's §6).
