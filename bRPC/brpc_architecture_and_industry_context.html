bRPC Architecture and Industry Context

bRPC is Apache's C++ RPC framework that came from Baidu's production infrastructure. It combines a multi-protocol RPC layer, a high-concurrency M:N threading runtime, non-blocking network IO, zero-copy buffers, client-side service governance, and built-in observability.

1. Overall Architecture

bRPC should be read as a complete service communication stack, not just a protobuf stub generator. The public API exposes brpc::Server, brpc::Channel, brpc::Controller, protobuf services, HTTP handlers, and protocol-specific adapters. Under that API, bRPC owns connection management, protocol detection, message parsing, request scheduling, output queuing, load balancing, retry/timeout behavior, statistics, and built-in debugging pages.

I did not find a bRPC-specific academic paper that publishes a canonical architecture diagram. The closest canonical architecture source is the official bRPC IO document's "full picture" image. Academic papers are still useful for context: "The Tail at Scale" explains why long-tail isolation matters in fan-out services, and "Dapper" explains why RPC libraries are a natural instrumentation point for distributed tracing.

How bRPC connects to TCP sockets or RDMA

TCP mode uses a native non-blocking fd under brpc::Socket. RDMA mode still uses brpc::Socket, but attaches RdmaEndpoint; payload transfer is posted to RDMA QPs through verbs. A TCP connection is still used for RDMA handshake and liveness, but not for payload after RDMA setup.

How bRPC uses epoll on Linux

bRPC uses epoll as a readiness notification layer, not as a place to run all IO. On Linux, EventDispatcher registers fds with edge-triggered events (EPOLLIN | EPOLLET and, when needed, EPOLLOUT | EPOLLET). epoll_wait returns SocketIds, input events call Socket::StartInputEvent, and output events call Socket::HandleEpollOut. The actual read, message cutting, protocol parsing, and service execution happen in bthreads and InputMessenger.

Architectural thesis: bRPC trades a larger C++ runtime for lower per-request overhead, protocol compatibility, and operational visibility in high-QPS internal services. It is closest to an infrastructure runtime plus RPC framework, while gRPC and Thrift are usually introduced as RPC/interface frameworks.

2. What bRPC Is

The official bRPC documentation describes bRPC as an industrial-grade C++ RPC framework used in high-performance systems such as search, storage, machine learning, advertising, and recommendation. The GitHub README also says that "bRPC" means "better RPC" and that the framework was known inside Baidu as "baidu-rpc".

Its central idea is practical unification. Instead of forcing all services to migrate to one wire protocol, bRPC can host and call multiple protocols while keeping one common runtime, observability model, client API, timeout/retry logic, and load-balancing machinery. This is why bRPC can be used as a native RPC framework, an HTTP/gRPC server, a client for Redis/Memcached-like protocols, and a compatibility bridge for older Baidu protocols.

Server side

A brpc::Server owns services, listens on a port, detects the incoming protocol automatically, dispatches each bounded message to protocol-specific parsing and processing callbacks, and runs user service logic in bthreads.

Client side

A brpc::Channel is the client abstraction. It is thread-safe after initialization and can represent a direct server, a discovered cluster, or a composition of channels used for sharded or parallel access.

Operational side

bRPC exposes built-in services for status, metrics, connections, flags, rpcz, CPU profiling, heap profiling, and lock contention profiling. This was a major design goal, not an add-on.

3. Why bRPC Was Created

bRPC was created because Baidu had a large C++ service fleet with multiple old RPC frameworks and protocols. The official benchmark page discusses UB, hulu-pbrpc, sofa-pbrpc, nova_pbrpc, public_pbrpc, Thrift, and gRPC as comparison points, and states that bRPC was developed by Baidu's infrastructure team around the end of 2014 to unify major distributed systems and business lines.

Problems the project was trying to solve

Problem Why it mattered bRPC response
Many wire protocols in production Large companies accumulate old protocols. Rewriting every service at once is unrealistic. Multi-protocol server and client support, with a documented path for adding new protocol callbacks.
Poor observability in older frameworks When RPC internals are a black box, teams debug with logs and framework maintainers become bottlenecks. Built-in HTTP debug pages, bvar metrics, rpcz, profiling, connection views, runtime flags.
Tail latency in fan-out services Search, recommendation, ads, and storage frontends often call many downstream services. Small slowdowns amplify into user-visible tail latency. Concurrent message processing, bthreads, low-contention Socket design, backup requests, load balancers, and per-call timeout/retry controls.
Event-loop maintainability Pure callback/event-loop code is fast when handlers are tiny, but complex service code often blocks or fans out to downstream RPCs. M:N bthread model lets service code look synchronous while still mapping many logical tasks onto fewer pthread workers.
Payload-copy overhead RPC attachments, HTTP bodies, and serialized protobuf messages can be large enough that flattening buffers becomes visible. IOBuf stores non-contiguous block references, supports zero-copy append/cut, and writes through scatter/gather IO when possible.

This motivation aligns with broader distributed-systems literature. Dean and Barroso's "The Tail at Scale" explains why large fan-out services must actively reduce and tolerate latency variability. Google's Dapper paper explains why RPC libraries are a natural place to put low-overhead tracing and observability, because RPC calls connect the causal path across services. Thrift's Facebook white paper shows the older industry pressure for language-neutral IDL and code generation. bRPC accepts those pressures, but focuses more specifically on C++ infrastructure performance, protocol coexistence, and production operability.

4. Runtime Internals

bthread

bthread is bRPC's M:N user-level threading library. Many bthreads map onto fewer pthread workers. The goal is to preserve a synchronous programming style while increasing concurrency and keeping better cache locality than a design that creates many OS threads. The official bthread documentation emphasizes work stealing and butex as key mechanisms: work stealing moves runnable bthreads across workers, and butex lets bthreads and pthreads wait/wake each other.

EventDispatcher and InputMessenger

The IO documentation distinguishes bRPC from ordinary "IO thread reads and parses" designs. bRPC's EventDispatcher waits for fd events, but the dispatcher itself is not supposed to do all reading and writing. On input readiness, bRPC starts a bthread to read from the fd. InputMessenger cuts bounded messages from bytes, uses the protocol's Parse callback, and then runs Process callbacks. When several messages are read from the same fd, bRPC can process them concurrently instead of letting a large or slow message block following messages.

Socket and SocketId

Socket is the central object around a file descriptor and related resources. bRPC uses a 64-bit SocketId to address sockets safely in a concurrent environment. The IO documentation compares this design loosely to strong and weak references: Socket::Address obtains a strong handle when the socket is still valid, while SetFailed prevents future addressing and lets references drain.

Memory pools

The memory-management documentation describes ResourcePool and ObjectPool as specialized allocators for same-sized internal objects. These reduce global allocation contention in frequently allocated framework structures. The docs also warn that this pattern is not a general-purpose recommendation for application code, because isolated pools can waste memory if overused.

5. Send and Receive Flow

Client send path

1. User call Application invokes generated protobuf stub or Channel::CallMethod with a Controller, request, response, and optional callback.
2. Select endpoint Channel consults naming service, health status, and load balancer, then chooses a socket or connection policy.
3. Pack protocol Protocol code serializes metadata and body. Attachments and protobuf streams are represented as IOBuf where possible.
4. Queue write Socket::Write puts a write request into a wait-free MPSC list. The first contender writes in-place; later contenders link requests.
5. Flush to fd bRPC batches pending buffers and writes them with scatter/gather IO when possible. Partial writes keep the remaining IOBuf slices queued.
6. Complete RPC Response parsing finds the outstanding RPC context, fills response/controller state, and runs the user's callback or wakes the waiting caller.

Server receive path

1. fd event EventDispatcher receives readiness from epoll and schedules input handling.
2. Read bytes A bthread reads from the socket into IOBuf-backed input storage.
3. Detect protocol The server tries protocol parsers. Once a connection's protocol is known, the remembered protocol is tried first.
4. Cut messages InputMessenger cuts complete bounded messages from the byte stream.
5. Run service Protocol Process code parses higher-level message state and calls the user's service method, usually in a bthread.
6. Send response Server-side done->Run() causes the framework to pack and write the response through the same Socket/IOBuf write path.
In older designs, a single IO thread often owns an fd's read, parse, process, and sometimes write path. bRPC intentionally splits event notification, message cutting, user processing, and output flushing so that slow user work does not unnecessarily block unrelated fd work or later messages on the same fd.

6. IOBuf Role

butil::IOBuf is bRPC's non-contiguous, reference-counted byte buffer. It is used for attachments in several protocols and for HTTP bodies. Internally, an IOBuf is a sequence of references to block slices rather than a single flat std::string. 

Logical payload:
  [RPC header][protobuf body][attachment bytes]

Physical IOBuf:
  BlockRef(offset, len) -> 8 KB block
  BlockRef(offset, len) -> 8 KB block
  BlockRef(offset, len) -> user/protobuf/attachment data
Operation IOBuf behavior Why it matters in bRPC
Append another IOBuf Copies buffer references, not payload bytes. Request or response attachments can be joined with protocol headers cheaply.
Cut from front Adjusts offsets and moves refs. Protocol parsers can remove headers/bodies without memmove.
Protobuf serialize/parse Uses zero-copy stream wrappers. Large protobuf messages avoid temporary flat buffers.
Socket write Builds iovec-like slices and writes multiple blocks. Output does not need to be flattened before kernel IO.
Partial write Sent bytes are popped from the front. Remaining unsent data stays as slice refs for the next flush.

The official IOBuf documentation warns that IOBuf is not a universal long-lived string replacement. A small retained slice may keep an 8 KB backing block alive. That tradeoff is acceptable for short-lived RPC buffers, but can waste memory if application code stores tiny slices for a long time.

7. Service Governance

bRPC puts several service-governance features in the C++ client stack instead of expecting a separate sidecar or proxy layer.

Naming service

A Channel can be initialized from schemes such as list://, file://, http://, https://, bns://, and consul://. The naming-service abstraction publishes server changes into the framework, which updates watched server lists.

Load balancing

The client can use round-robin, weighted round-robin, random, weighted random, locality-aware, and consistent-hash strategies. Health checking and isolation remove bad nodes and probe them for recovery.

Reliability controls

Controller and Channel options cover timeout, retry, backup request, cancellation, connection type, compression, SSL, authentication, and message-size limits.

Protocol coexistence

Server-side protocol auto-detection allows several protocols on the same port. Client side knows the requested protocol, so response parsing avoids server-side protocol probing overhead.

8. Observability

bRPC's observability model is one of its strongest differentiators. The framework exposes operational data over built-in HTTP services. This is important because RPC frameworks sit on the causal path of nearly every distributed request.

Feature Purpose
/vars and /brpc_metrics bvar metrics for framework and application counters, including Prometheus export.
/status Service-level status and version information.
/connections Connection inspection, including SSL information when enabled.
/flags Runtime inspection and modification of gflags where supported.
/rpcz RPC-level tracing/debugging similar in spirit to putting low-overhead instrumentation inside the RPC library.
CPU, heap, contention profilers On-demand diagnosis of hot functions, allocations, and lock contention in live services.

The design philosophy is close to the Dapper paper's central lesson: in large service graphs, tracing and diagnostics become much more effective when they are integrated into common RPC and threading libraries rather than manually recreated by every service team.

9. Use Scenarios

Strong fits

  • Large C++ service fleets that need high QPS, low latency, and robust tail behavior.
  • Search, recommendation, advertising, storage, machine-learning serving, and other fan-out systems where one request may issue many downstream RPCs.
  • Organizations with multiple existing protocols that need a unifying runtime and a migration bridge rather than a clean-slate protocol migration.
  • Systems that need built-in operational debugging on every process without deploying a separate observability agent first.
  • Internal APIs that need protobuf RPC, HTTP/JSON access, gRPC/h2 compatibility, or protocol-specific clients such as Redis/Memcached/Thrift under one API style.
  • Distributed storage and consensus systems using braft or C++ infrastructure where RPC overhead and diagnostic clarity are important.

Weaker fits

  • A polyglot public API where standard gRPC ecosystem support across many languages matters more than C++ runtime performance.
  • A Java-centered enterprise microservice platform already standardized on Dubbo, Spring Cloud, or service-mesh governance.
  • Small CRUD services where ordinary REST plus HTTP server/client libraries are sufficient and operational complexity should stay minimal.
  • Browser-facing APIs that need direct browser support. bRPC can serve HTTP, but the full RPC value is mostly for service-to-service traffic.

10. Industry Alternatives

Framework Primary identity Strengths Compared with bRPC
bRPC C++ production RPC runtime and multi-protocol framework. High-performance C++ runtime, bthread, IOBuf, multi-protocol server/client, built-in diagnostics, Baidu protocol compatibility. Best when C++ service performance and protocol coexistence matter.
gRPC HTTP/2 + protobuf RPC framework from Google, now common across cloud-native stacks. Broad language support, standard service definition, streaming, authentication, load-balancing and tracing hooks, strong ecosystem. bRPC can speak gRPC/h2, but gRPC is usually the safer default for public, polyglot APIs.
Apache Thrift IDL, serialization, and RPC framework originally from Facebook. Cross-language code generation, multiple transports/protocols, compact binary encodings, mature IDL model. bRPC can access/serve Thrift protocol while adding its own thread-safe client API, load balancing, timeout/retry, and diagnostics.
Apache Dubbo Microservice RPC and governance framework, historically strong in Java/Alibaba ecosystems. Service discovery, traffic governance, Java/Spring integration, Dubbo2 TCP protocol, Triple protocol compatible with gRPC. Dubbo is more of an application-level microservice governance framework; bRPC is a C++ infrastructure runtime with deep IO/threading control.
Twitter Finagle Scala/JVM protocol-agnostic RPC system. Composable filters, futures, client-side load balancing, protocol abstraction, high-concurrency JVM services. Similar protocol-agnostic ambition, but in the JVM/Scala ecosystem rather than C++ with bthread/IOBuf.
REST over HTTP libraries Resource-oriented web APIs using HTTP/JSON or similar formats. Universal interoperability, browser/tool support, easy debugging, simple deployment. REST is simpler and more public-facing. bRPC is better for internal high-QPS service calls, binary payloads, and controlled service fleets.
The bRPC benchmark page contains direct comparisons against older versions of UB, hulu-pbrpc, sofa-pbrpc, Thrift, and gRPC. Treat those numbers as historical evidence for bRPC's design motivation rather than as current universal benchmark results. Modern gRPC, Dubbo, Netty, Linux kernels, NICs, and compilers have changed substantially.

11. Tradeoffs

Design choice Benefit Cost or risk
M:N bthread model Synchronous-looking code with high concurrency and better scheduling control than many blocking pthreads. Framework-specific runtime semantics; blocking pthread/system calls can still exhaust workers if overused.
Multi-protocol support Migration bridge and compatibility with heterogeneous production fleets. More complex protocol detection, testing matrix, and operational surface.
IOBuf non-contiguous buffers Low-copy attachment/body handling and efficient partial writes. Not a generic string. Long-lived slices can retain full 8 KB blocks.
Built-in debug services Fast field diagnosis and low-friction observability. Must be protected in production. Public exposure can leak internal state.
C++ focus Deep performance control and integration with existing C++ infrastructure. Less natural than gRPC/Thrift/Dubbo for broad multi-language product APIs.
