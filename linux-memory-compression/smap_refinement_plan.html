SMAP Refinement Plan for ubturbo

Source repo: /Users/ray/Documents/Repo/ubturbo

Reviewed areas: src/smap, plugins/smap, plugins/rmrs/src/ubturbo_plugin/common/smap, and related tests.

This document converts the SMAP refinement notes into an implementation-oriented design page. It summarizes the current architecture, risk areas, and a prioritized plan for improving safety, maintainability, observability, and migration policy quality.

Summary

ubturbo contains two SMAP layers:

  • src/smap: the UBTurbo-facing IPC/client/server bridge. It encodes SMAP requests, sends them through UBTurbo IPC, dynamically loads libsmap.so, and dispatches to SMAP APIs.
  • plugins/smap: the actual SMAP package. It contains user-space management, kernel/tiering code, migration wrappers, config handling, unit tests, and design documentation.

SMAP is a multi-tier memory manager. It identifies hot and cold pages, migrates hot data toward local or fast memory, migrates cold data toward remote or slower memory, and adjusts per-process memory allocation. The existing design documentation already describes the major policy ideas: hot/cold sorting, adaptive migration parameters, multi-process scheduling, and hardware HIST sliding-window scanning.

Overall Architecture

The separation is useful: UBTurbo owns service lifecycle and IPC, while libsmap.so owns the actual memory-tiering logic. The refinement plan keeps that boundary and improves the reliability of the bridge, deployment, policy observability, and migration decisions.

Main Findings

1. IPC codec validation should be hardened

Several decode paths read a len field from the incoming IPC buffer, then use it in payload-size arithmetic before validating that the value is positive and within the expected maximum. Key areas include SmapAddProcessTrackingCodec::DecodeRequest, SmapRemoveProcessTrackingCodec::DecodeRequest, and SmapEnableProcessMigrateCodec::DecodeRequest.

Risk:

  • Malformed IPC can provide negative or oversized lengths.
  • Signed int can be promoted into large size_t arithmetic.
  • Copies into fixed arrays depend on later memcpy_s protection rather than clear precondition checks.

Recommended helper layer:

static bool IsCountInRange(int count, int max)
{
    return count > 0 && count <= max;
}

static bool CheckedAdd(size_t a, size_t b, size_t *out)
{
    if (SIZE_MAX - a < b) {
        return false;
    }
    *out = a + b;
    return true;
}

static bool CheckedMul(size_t a, size_t b, size_t *out)
{
    if (a != 0 && b > SIZE_MAX / a) {
        return false;
    }
    *out = a * b;
    return true;
}

Validate len before computing sizeof(pid_t) * len, sizeof(uint32_t) * len, or similar payload sizes.

2. Codec implementation has heavy duplication

The codec file repeats the same patterns: allocate a TurboByteBuffer, copy fields, encode an int response, decode an int response, and reset buffers on error.

Recommended helper functions:

int EncodePlain(TurboByteBuffer &buffer, const void *src, size_t size);
int DecodePlain(const TurboByteBuffer &buffer, void *dst, size_t size);
int EncodeIntResponse(TurboByteBuffer &buffer, int value);
int DecodeIntResponse(const TurboByteBuffer &buffer, int &value);

For compound requests, use a small append writer:

class BufferWriter {
public:
    explicit BufferWriter(TurboByteBuffer &buffer, size_t size);
    int Append(const void *src, size_t size);
};

3. Client buffers should use RAII

The client code manually deletes send.data and recv.data in every function. That makes future edits prone to leaks or inconsistent cleanup. 

class ByteBufferGuard {
public:
    TurboByteBuffer buffer{};

    ~ByteBufferGuard()
    {
        delete[] buffer.data;
    }

    TurboByteBuffer *operator->()
    {
        return &buffer;
    }
};

Then client APIs can use ByteBufferGuard send; and ByteBufferGuard recv;.

4. Hardcoded paths should become configuration

Current hardcoded paths include:

  • /dev/shm/ubturbo_page_type.dat
  • /usr/lib64/libsmap.so
  • /dev/shm/smap_config

Recommended config defaults:

smap.libsmap_path = /usr/lib64/libsmap.so
smap.page_type_path = /dev/shm/ubturbo_page_type.dat
smap.config_path = /dev/shm/smap_config

5. Dynamic symbol loading diagnostics should be explicit

turbo_module_smap.cpp resolves many SMAP symbols with dlsym and checks an aggregate flag. When deployment has an old or mismatched libsmap.so, logs should name the exact missing symbol. 

struct SmapSymbol {
    const char *name;
    void **target;
};

Load each symbol in a loop and log the exact missing symbol name.

6. Build script has a path-check bug

In plugins/smap/build.sh, the bounds-check library test includes a trailing space:

if [[ ! -f "$LIB_BOUNDS_CHECK_FILE " ]]

It should be:

if [[ ! -f "$LIB_BOUNDS_CHECK_FILE" ]]

Also quote paths consistently in shell scripts.

7. Unit test script mutates source files

test/run_ut.sh removes static from production source files before building tests:

find ${dir} -type f -name "*.cpp" | xargs -i sed -i "s/\bstatic\b//g" {}

Do not rewrite source files in-place. Prefer one of:

  • compile test-only copies under test/build/generated;
  • expose narrow test hooks behind #ifdef UNIT_TEST;
  • move private helpers into small internal translation units with test-visible headers;
  • use linker wrapping or mocks where feasible.

8. NUMA probing should avoid shell commands

plugins/smap/src/user/smap_interface.c uses:

popen("numastat -cvm", "r")

Prefer one of:

  • /sys/devices/system/node/nodeN;
  • libnuma APIs;
  • topology cached during SMAP initialization;
  • kernel-provided node masks already visible to the module.

Algorithm-Level Refinements

Migration hysteresis

Use separate promote and demote thresholds so pages do not oscillate between local and remote memory.

Migration ROI scoring

Only migrate when expected latency savings exceed migration cost and budget constraints.

Per-process budgets

Limit pages per cycle, bytes per second, CPU time spent scanning, and concurrent migration tasks.

Anti-thrashing

Track repeat migrations and temporarily pin or de-prioritize unstable pages or regions.

Runtime observability

Expose scan period, migration period, pages scanned, pages migrated, failures, bytes, and hit rates.

Dry-run mode

Compute candidate migrations without executing them, useful for safe policy tuning.

ROI sketch

benefit = expected_latency_saving * expected_future_accesses
cost = migration_bytes / migration_bandwidth + CPU_overhead
score = benefit - cost

Only migrate when score > 0 and budget allows it.

Testing Recommendations

Area Recommended tests
IPC and ABI Negative length, zero length, oversized length, truncated payload, null data, payload overflow, invalid response length, public struct size/alignment/offset checks.
Dynamic loading Missing libsmap.so, missing symbol, old library version, symbol resolved but returns error, unload/reload lifecycle.
Policy simulation All-local hot, all-remote hot, steady hot set, shifting hot set, noisy access pattern, mixed heavy/light workloads.
Build hygiene Scripts do not mutate tracked files, generated files stay under build/output directories, shell paths are quoted, practical shellcheck coverage.

Suggested Implementation Order

  1. Fix plugins/smap/build.sh path check and quoting.
  2. Stop test/run_ut.sh from mutating source files.
  3. Add codec length/range validation helpers.
  4. Add RAII buffer guards in src/smap/client.
  5. Refactor repeated codec response helpers.
  6. Make SMAP paths configurable.
  7. Improve dlsym diagnostics with a symbol table.
  8. Add malformed IPC and ABI tests.
  9. Replace numastat shell probing with sysfs/libnuma/cached topology.
  10. Add runtime observability and dry-run mode.
  11. Add migration hysteresis, ROI scoring, and anti-thrashing.

First Patch Candidate

Low risk Safety No policy behavior change

A practical first patch can include:

  • fix the "$LIB_BOUNDS_CHECK_FILE " typo;
  • add IsCountInRange;
  • validate IPC lengths before payload-size arithmetic;
  • add negative and oversized length unit tests;
  • add a simple ByteBufferGuard for client cleanup.

This improves safety and maintainability without changing SMAP's migration policy behavior.
