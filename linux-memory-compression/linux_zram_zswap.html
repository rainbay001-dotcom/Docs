1. Overview & Motivation

When Linux runs low on free memory, the kernel's reclaim path evicts pages. Anonymous pages (heap, stack, mmap) that cannot be discarded must be swapped out to persistent storage. Traditional swap writes uncompressed 4 KiB pages to an HDD or SSD — incurring millisecond-scale I/O latency and wearing flash cells.

Compressed swap exploits a key insight: most memory pages compress to 40–60% of their original size. By spending a few microseconds of CPU time on compression, the kernel can store 2–3× more evicted pages in RAM (or reduce disk I/O dramatically). Linux provides two complementary mechanisms:

1.1 Overall Architecture

1.2 History & Evolution

Compressed memory in operating systems has a long lineage:

  • 1999 — Paul Wilson & Scott Kaplan publish “Compressed Caching”, the foundational academic work showing that using a fraction of RAM as a compressed cache for evicted pages outperforms simply adding that RAM to the page cache. They demonstrated 2–5× effective memory expansion on workloads with favorable compression ratios.
  • 2009 — Nitin Gupta implements compcache/ramzswap (later renamed zram) as an out-of-tree module, providing a compressed block device for swap.
  • 2010 — zram enters drivers/staging/zram/ in Linux 2.6.33.
  • 2012 — Seth Jennings proposes zcache/zswap as a transparent compressed cache layer using the frontswap API — no user-visible block device needed.
  • 2013 — Apple ships Compressed Memory in OS X Mavericks, validating the approach at scale. zswap merges into Linux 3.11. Android begins defaulting to zram.
  • 2014 — zram moves out of staging into drivers/block/zram/ (Linux 3.14). Microsoft adds memory compression to Windows 10.
  • 2017–2019 — ChromeOS fully adopts zram; Fedora enables zram-swap by default starting with Fedora 33.
  • 2023–2024 — zswap undergoes major refactoring: frontswap dependency removed (replaced with direct xarray lookup), memcg-aware shrinker added, folio API adopted. Multi-compression support added to zram (CONFIG_ZRAM_MULTI_COMP).
  • 2025 — zram adds compressed writeback and async backing-device reads. zswap gains disk-swapin tracking for shrinker calibration. Both subsystems mature significantly.

2. zram — Compressed RAM Block Device

zram creates virtual block devices (/dev/zram0, /dev/zram1, …) backed entirely by compressed RAM. Each device behaves like any block device — you can mkswap + swapon, mkfs + mount, or use it as a raw block device. The most common use is swap.

2.2 Data Structures

struct zram (per-device)

struct zram {
    struct zram_table_entry *table;      // per-page metadata array
    struct zs_pool *mem_pool;             // zsmalloc pool for compressed data
    struct zcomp *comps[ZRAM_MAX_COMPS];  // compression engines (up to 4)
    struct gendisk *disk;                 // block device
    unsigned long limit_pages;            // memory quota
    u64 disksize;                         // logical size in bytes
    s8 num_active_comps;                  // active compression engines

    // CONFIG_ZRAM_WRITEBACK
    struct file *backing_dev;             // backing block device
    unsigned long *bitmap;                // backing device block bitmap
    bool compressed_wb;                   // write compressed data to backing dev
    u32 wb_batch_size;                    // max in-flight writeback ops
};

struct zram_table_entry (per-page, ~16 bytes)

struct zram_table_entry {
    unsigned long handle;        // zsmalloc handle OR backing device block index
    union {
        unsigned long __lock;    // bit-spinlock (ZRAM_ENTRY_LOCK)
        struct {
            u32 flags;           // packed: size (bits 0..PAGE_SHIFT) + status flags
            u32 ac_time;         // access time (CONFIG_ZRAM_TRACK_ENTRY_ACTIME)
        } attr;
    };
};

// Flag bits (above PAGE_SHIFT):
//   ZRAM_SAME          — page is same-filled (zero or repeated word)
//   ZRAM_WB            — page written back to backing device
//   ZRAM_HUGE          — incompressible, stored uncompressed
//   ZRAM_IDLE          — not accessed since last idle mark
//   ZRAM_INCOMPRESSIBLE — no algorithm could compress it
//   ZRAM_COMP_PRIORITY — 2-bit field: which compression algo was used (0–3)
Design insight: The per-page metadata is only ~16 bytes. For a 16 GB zram device (4M pages), the table costs just 64 MB — about 0.4% of the virtual size. The compressed size is encoded in the lower bits of the flags word, avoiding a separate field.

2.3 Write Path

When the swap subsystem writes a page to /dev/zramN:

bio submit → zram_submit_bio()
  → zram_bio_write()              // iterate bio_vecs
    → zram_bvec_write()
      → zram_write_page(page, index)
          1. page_same_filled()?  → ZRAM_SAME flag, store element only (no alloc!)
          2. zcomp_stream_get()   → acquire per-CPU compression stream (mutex)
          3. zcomp_compress()     → compress into 2×PAGE_SIZE buffer
          4. comp_len ≥ huge?     → ZRAM_HUGE, store uncompressed (PAGE_SIZE alloc)
          5. zs_malloc(comp_len)  → allocate from zsmalloc pool
          6. zram_can_store_page? → check limit_pages quota
          7. zs_obj_write()       → write compressed data to zsmalloc object
          8. slot_lock → update table[index] → slot_unlock
          9. update stats (pages_stored, compr_data_size)

Same-filled page optimization: Before compressing, zram checks if the entire page is filled with a single repeated unsigned long value (common for zero pages). If so, only the fill value is stored — no compression, no allocation. This handles the very common case of zero-initialized memory.

2.4 Read Path

page fault → bio submit → zram_submit_bio()
  → zram_bio_read()
    → zram_bvec_read()
      → zram_read_page(page, index)
          slot_lock(index)
          check flags:
            ZRAM_SAME → zram_fill_page() (memset_l)
            ZRAM_WB   → read_from_bdev() (async bio to backing device)
            ZRAM_HUGE → zs_obj_read_begin() → copy_page() (no decompress)
            normal    → read_compressed_page():
                          zcomp_stream_get(comps[priority])
                          zs_obj_read_begin(handle)  → get compressed data ptr
                          zcomp_decompress()          → decompress into page
                          zs_obj_read_end(handle)
          mark_slot_accessed()  → clear ZRAM_IDLE flag
Async backing device reads: When a page has been written back (ZRAM_WB), zram issues an async bio to the backing device. The completion callback zram_async_read_endio() handles decompression in a high-priority workqueue if compressed_wb was enabled — avoiding decompression in interrupt context.

2.5 Writeback to Backing Device

With CONFIG_ZRAM_WRITEBACK, zram can offload cold or incompressible pages to a real block device, freeing RAM. This is user-triggered via sysfs, not automatic:

# Setup
echo /dev/sda2 > /sys/block/zram0/backing_dev

# Trigger writeback of idle pages
echo all > /sys/block/zram0/idle            # mark all pages idle
# ... wait for some time ...
echo "type=idle" > /sys/block/zram0/writeback

# Or writeback huge (incompressible) pages
echo "type=huge" > /sys/block/zram0/writeback

The writeback pipeline uses async bio submission with configurable batch size (wb_batch_size, default 32). With compressed_wb=1, compressed data is written directly to the backing device without decompressing first, saving CPU.

2.6 Idle Recompression

With CONFIG_ZRAM_MULTI_COMP, zram supports up to 4 compression algorithms (1 primary + 3 secondary). Cold pages initially compressed with a fast algorithm (e.g., lzo-rle) can be recompressed with a slower but denser algorithm (e.g., zstd):

# Configure secondary algorithm
echo "algo=zstd priority=1" > /sys/block/zram0/recomp_algorithm

# Mark pages idle, then recompress
echo all > /sys/block/zram0/idle
echo "type=idle algo=zstd max_pages=10000" > /sys/block/zram0/recompress

The recompression logic compares zsmalloc size class indices (not raw byte counts) to decide if recompression actually saves memory — a smaller byte count that maps to the same size class is not worth the effort.

3. zswap — Compressed Swap Cache

zswap is a transparent compressed cache that sits between the swap subsystem and the actual swap device. When the kernel decides to swap out a page, zswap intercepts it, compresses it, and stores the compressed version in RAM. Only when zswap's pool fills up does it evict (writeback) the coldest pages to the real swap device.

Unlike zram, zswap is not a block device — it hooks directly into the MM layer (mm/page_io.c), making it invisible to userspace. A real swap device is still required as a backing store.

Lookup Structure

// One xarray per 64 MB swap granule, per swap device
static struct xarray *zswap_trees[MAX_SWAPFILES];

// Lookup: swap_zswap_tree(swp_entry_t) returns the correct xarray
// Key: swp_offset within that 64M window
Why xarray, not a hash table? Swap offsets are dense integers in a known range. Xarray provides O(log64 n) lookup with excellent cache behavior, automatic memory management, and RCU-safe concurrent access — perfect for this use case. The 64 MB granularity keeps individual xarrays small while supporting very large swap devices.

3.3 Store Path

Page reclaim (vmscan.c / kswapd)
  → swap_writepage() in page_io.c
    → zswap_store(folio)                   // intercept before disk I/O
        1. Check static key (fast path if disabled)
        2. Check memcg zswap.max limit
        3. Check global pool limit (zswap_check_limits)
        4. Get current pool ref (RCU-safe)
        5. For each page in folio:
           → zswap_store_page(page, objcg, pool)
              a. Allocate zswap_entry (kmem_cache, NUMA-local)
              b. zswap_compress(page, entry, pool)
                 - acomp_ctx_get_cpu_lock() → per-CPU context
                 - SG input from page, output to buffer
                 - crypto_acomp_compress() + crypto_wait_req()
                 - If comp_len ≥ PAGE_SIZE → store uncompressed (if WB enabled)
                 - zs_malloc() → zs_obj_write() → store in zsmalloc
              c. xa_store() → insert into xarray (free old entry if redirtied)
              d. obj_cgroup_charge_zswap() → charge memcg
              e. list_lru_add() → add to LRU
              f. entry->referenced = true → second-chance protection
        6. Count ZSWPOUT events
        7. Return true → page_io.c skips disk write!

3.4 Load Path

Page fault (do_swap_page)
  → swap_read_folio() in page_io.c
    → zswap_load(folio)
        1. xa_load(tree, offset) → find entry (RCU-safe, no lock)
        2. If not found → return -ENOENT → fall through to disk read
        3. zswap_decompress(entry, folio)
           - Get per-CPU acomp context
           - If length == PAGE_SIZE → memcpy (stored uncompressed)
           - Else → crypto_acomp_decompress() + wait
        4. Mark folio up-to-date
        5. If swapcache → xa_erase + free entry (swapcache is now authoritative)
        6. Count ZSWPIN event
        7. Return 0 → page_io.c skips disk read!

3.5 Eviction & Writeback

When the compressed pool exceeds max_pool_percent of RAM (default 20%), zswap must evict entries to the real swap device:

Second-Chance LRU Algorithm

The shrinker walks the LRU list. Each entry has a referenced bit:

  • First encounter, referenced=true: Clear the bit, rotate entry to tail of LRU. The page gets a "second chance" before eviction.
  • Second encounter, referenced=false: Evict — decompress and write to real swap.

This prevents thrashing of recently-stored pages and approximates true LRU behavior without the overhead of maintaining access timestamps.

Writeback Pipeline

shrink_memcg_cb() [LRU walker]
  → zswap_writeback_entry(entry, swpentry)
      1. swap_cache_alloc_folio() → allocate new folio
      2. Verify entry still in xarray (concurrent invalidation check)
      3. zswap_decompress(entry, folio) → decompress into folio
      4. xa_erase() + free entry → remove from zswap
      5. __swap_writepage(folio) → write to real swap device
      6. Count ZSWPWB event

Shrinker Calibration

The shrinker count function includes two sophisticated feedback mechanisms:

  • Disk swapin protection: Tracks nr_disk_swapins per lruvec. If zswap overshrinks, pages get read from disk on swapin instead of from zswap — this counter detects that and reduces future shrink aggressiveness.
  • Compression ratio scaling: If compression is good (small backing), fewer evictions are needed per byte freed. If compression is poor, more evictions needed.

4. zsmalloc — The Shared Allocator

Both zram and zswap use zsmalloc (mm/zsmalloc.c) to store compressed page data. zsmalloc is a specialized slab allocator designed for variable-size compressed objects, optimizing for high packing density with minimal fragmentation.

Key Properties

  • 255 size classes with 16-byte granularity, covering 32 B to PAGE_SIZE
  • ZSpages group 1–N physical pages together; all objects within a zspage are the same size class
  • Objects can span page boundaries — zs_obj_read_begin() copies spanning objects to a local_copy buffer transparently
  • Handle indirection — handles encode PFN + object index, allowing zspage migration for compaction without invalidating external references
  • Best-fit allocation — allocates from the most-full zspage first, minimizing fragmentation
  • Compaction — zs_compact() migrates objects from partially-full zspages to consolidate free pages, callable from memory pressure or sysfs
  • Typical overhead — 10–20% internal fragmentation (objects rounded up to nearest size class)

5. Compression Algorithms

The choice of compression algorithm is a fundamental CPU-vs-memory tradeoff. Both zram and zswap compress individual 4 KiB pages independently (no inter-page context), limiting compression ratios but enabling random access.

AlgorithmCompress SpeedDecompress SpeedRatio (typical)Notes
lzo-rle~600 MB/s~800 MB/s2.0–2.5× LZO with RLE pre-pass; fast, low CPU. zram default.
lz4~700 MB/s~1200 MB/s2.0–2.5× Fastest decompression; excellent for latency-sensitive workloads.
zstd~150 MB/s~500 MB/s2.5–3.5× Best ratio/speed tradeoff; ideal for recompression of cold pages.
lz4hc~50 MB/s~1200 MB/s2.5–3.0× Asymmetric: slow compress, fast decompress. Good for write-once data.
deflate~100 MB/s~400 MB/s2.5–3.5× Mature, well-understood. zswap default. Tunable via winbits.

Per-page compression context: Each page is compressed independently with no dictionary or cross-page state. This is necessary for random access but means compression ratios are lower than file-level compression. Memory pages have high entropy variance — code pages compress well (3–4×), while encrypted or pre-compressed data barely compresses at all.

Multi-Compression Strategy (zram)

zram's CONFIG_ZRAM_MULTI_COMP enables a tiered approach:

  1. Hot path (writes): Fast algorithm (lzo-rle or lz4) for minimal latency impact
  2. Cold path (recompression): Dense algorithm (zstd) applied to idle pages in background, recovering additional memory from pages unlikely to be accessed soon

This achieves the best of both worlds: low-latency writes with high compression for cold data.

6. zram vs zswap — When to Use Which

zram Best For

  • No swap device available (embedded, diskless, tmpfs-only)
  • Android / ChromeOS (ships as default)
  • Full control (sysfs knobs for writeback, recompression, idle tracking)
  • Non-swap uses (compressed /tmp, ramfs replacement)
  • Multiple algorithms simultaneously (hot: lz4, cold: zstd)
  • Predictable latency (no disk I/O unless writeback enabled)

zswap Best For

  • Servers with real swap (augments existing swap infrastructure)
  • Transparent deployment (no userspace changes, just enable module)
  • Cgroup-aware workloads (containers, systemd slices)
  • Automatic eviction (shrinker + LRU, no manual writeback needed)
  • Reducing SSD wear (fewer writes to real swap)
  • Large memory servers (20% of RAM as compressed cache is huge)

6.1 NVMe as Backing Store — Tiered Memory Hierarchy

Both zram and zswap can use an NVMe device to store memory pages, but in fundamentally different ways. This creates a three-tier memory hierarchy: DRAM → Compressed RAM → NVMe, where compression adds a new level that bridges the latency gap between DRAM and flash.

How Each Uses NVMe

zswap + NVMe Swap (automatic)

zswap requires a real swap device. An NVMe partition or swap file is the backing store. The kernel handles everything automatically:

# Setup
mkswap /dev/nvme0n1p2
swapon /dev/nvme0n1p2
echo 1 > /sys/module/zswap/parameters/enabled

# Flow (fully automatic):
# page evicted
#   → compressed in RAM (zswap pool)
#   → pool full? LRU shrinker evicts coldest
#     → decompress → __swap_writepage()
#       → written to NVMe swap partition
#
# On swapin:
#   → check zswap first (~1 µs)
#   → miss? read from NVMe (~50 µs)

zswap reduces NVMe write volume by 60–80% and completely eliminates NVMe reads for pages still in the compressed pool.

zram + NVMe Backing (manual)

zram can optionally write back cold pages to an NVMe backing device (CONFIG_ZRAM_WRITEBACK). This is user-triggered:

# Setup
echo /dev/nvme0n1p3 > /sys/block/zram0/backing_dev

# Trigger writeback of idle pages
echo all > /sys/block/zram0/idle
sleep 3600   # let pages go cold
echo "type=idle" > /sys/block/zram0/writeback

# With compressed_wb=1, compressed data
# is written directly — no decompress cycle:
echo 1 > /sys/block/zram0/compressed_wb
echo "type=huge" > /sys/block/zram0/writeback

The compressed_wb option writes already-compressed data to NVMe, saving CPU on the writeback path. On read-back, decompression happens asynchronously in a high-priority workqueue.

Compressed Writeback — Saving CPU on the NVMe Path

Both zram and zswap support writing already-compressed data to the backing device, avoiding the decompress→write→recompress cycle:

Normal WritebackCompressed Writeback
Write path decompress (CPU) → write PAGE_SIZE to NVMe write comp_len to NVMe directly (no CPU decompression)
Read path read PAGE_SIZE from NVMe → done read comp_len from NVMe → decompress (CPU)
NVMe bandwidth Full 4 KiB per page ~1.5–2 KiB per page (compressed)
CPU cost Decompression on write, none on read None on write, decompression on read
Best for Read-heavy workloads (fast swapin) Write-heavy / SSD-wear-sensitive (fewer bytes written)
SSD wear reduction: With compressed writeback, each evicted page writes only ~1.5–2 KiB to NVMe instead of 4 KiB. Combined with the 60–80% traffic reduction from the compressed cache, total NVMe write volume can be reduced by 85–95% compared to raw swap — dramatically extending SSD lifespan.

7. Real-World Deployments

Android (zram)

Android has used zram as its primary swap mechanism since Android 4.4 (KitKat). On memory-constrained devices (2–4 GB RAM), zram typically provides 1–2 GB of effective additional memory through compression. Android's lmkd (Low Memory Killer Daemon) works in concert with zram — kswapd pushes pages to zram, and lmkd kills apps when even compressed swap is exhausted.

ChromeOS (zram)

ChromeOS uses zram extensively. The swap_management service configures zram at boot with LZO compression. ChromeOS also uses zram for its tab discarding heuristics — tabs are compressed to zram before being fully discarded.

Fedora / Ubuntu (zram)

Fedora 33+ enables zram by default via systemd-zram-generator, typically configuring 50% of RAM as zram swap with lzo-rle compression. Ubuntu 24.10+ followed suit. This replaced traditional swap partitions in default desktop installations.

Meta / Facebook (zswap)

Meta uses zswap extensively in their server fleet. Johannes Weiner (Meta engineer) is a primary maintainer. Their workloads benefit from zswap's transparent cgroup integration with their container infrastructure. Much of the recent zswap work (memcg-aware shrinker, improved eviction) was driven by Meta's production experience.

Enterprise Linux (varies)

RHEL/CentOS typically provides both as kernel modules. Default configurations vary by deployment: containerized workloads tend toward zswap; bare-metal databases often disable both in favor of abundant RAM and no swap.

Foundational Work

Paul R. Wilson, Scott F. Kaplan, Yannis Smaragdakis. "The Case for Compressed Caching in Virtual Memory Systems." USENIX Annual Technical Conference, 1999.

The seminal paper establishing that a compressed in-memory cache for evicted pages can outperform using the same RAM as uncompressed page cache. The paper's adaptive model — dynamically resizing the compressed cache based on observed compression ratios and access patterns — directly influenced zswap's design.

[2] zcache & tmem (2010–2012)

Dan Magenheimer (Oracle). "Transcendent Memory" and "zcache" patches. Linux Plumbers Conference, 2010–2012.

Proposed a "transcendent memory" abstraction where the kernel could opportunistically store pages in a compressed cache managed outside the normal page allocator. zcache was the prototype; its lessons (complexity of the cleancache/frontswap APIs, difficulty of integration) directly shaped zswap's simpler design.

[3] Memory Compression in macOS (2013)

Apple Inc. "OS X Mavericks Core Technologies Overview." WWDC 2013.

Apple's implementation validated compressed memory at consumer scale. Their approach uses WKdm (a word-centric dictionary compressor optimized for memory pages) and integrates compression into the Mach VM layer. Unlike Linux's zram/zswap split, macOS uses a single unified compressed memory system.

[4] Memory Compression in Windows 10 (2015)

Microsoft. "Memory compression" in Windows 10 Memory Manager.

Windows 10 introduced a "compression store" managed by the System process. Pages are compressed in-memory before being written to the pagefile, conceptually similar to zswap. The Windows implementation uses XPRESS (Huffman variant) compression.

Compression Algorithm Research

[5] LZ4 — Collet (2011)

Yann Collet. "LZ4: Extremely fast compression algorithm." 2011.

LZ4 achieves >500 MB/s compression with ~1.2 GB/s decompression on modern CPUs, making it ideal for latency-critical compressed swap. Its fixed-overhead design (no Huffman tables) means consistent performance regardless of input entropy.

[6] Zstandard — Collet (2015)

Yann Collet. "Zstandard: Fast real-time compression algorithm." RFC 8478, 2018. Facebook Engineering Blog, 2015.

Zstd provides compression ratios approaching DEFLATE with speeds closer to LZ4. Its dictionary compression mode is particularly effective for memory pages with repetitive structure. zram's multi-comp feature leverages zstd for background recompression of cold pages.

Systems Research

[7] Understanding Memory Compression Trade-offs

Various authors. Proceedings of Linux Plumbers Conference, 2019–2024.

Key findings from production deployments:

  • CPU overhead is not free: On CPU-bound workloads, zram/zswap compression can reduce throughput by 5–15%. The benefit depends on whether the workload is memory-bound or CPU-bound.
  • Compression ratios vary widely: Text/code pages achieve 3–4×; already-compressed or encrypted data achieves ~1× (incompressible). The "huge page" detection in zram avoids wasting CPU on incompressible pages.
  • Fragmentation matters: zsmalloc's 16-byte granularity means 10–20% of allocated memory is wasted on rounding. The z3fold and zbud allocators (used by earlier zswap versions) had worse fragmentation but simpler code.

[8] Zpool Allocator Comparison

Seth Jennings. "zbud: A Low-Latency, Low-Overhead Compressed Page Cache" and related work on z3fold (Vitaly Wool). 2013–2016.

Three allocators have been used for compressed pages in Linux:

AllocatorDesignMax Objects/PageFragmentationStatus
zbud2 objects per page (buddy)2High (~50%)Deprecated
z3fold3 objects per page3Medium (~33%)Deprecated (6.1.112)

Both zram and zswap now use zsmalloc exclusively. The zpool abstraction layer was removed from zswap in kernel 6.18 (Dec 2025) — zswap now calls zsmalloc directly.

[9] Cgroup-Aware Memory Compression

Nhat Pham, Johannes Weiner. "zswap: memcg-aware compressed swap cache." LKML patches, 2023–2024.

The addition of memcg awareness to zswap's shrinker was driven by container workloads at Meta. Without cgroup-aware eviction, a single container's compressed pages could monopolize the zswap pool, degrading performance for all containers. The memcg-aware shrinker enables per-cgroup zswap limits and fair eviction.

Mobile / Android Research

[10] SWAM — Samsung Research (MobiCom 2023)

Samsung Research. "SWAM: Revisiting Swap and OOMK for Improving Application Responsiveness on Mobile Devices." ACM MobiCom 2023. arXiv:2306.08345

Integrates adaptive swap (memory or storage), OOM cleaner, and enhanced OOM killer. 6.5× fewer killed apps, 36% faster launch, 41% faster response. Implemented on Pixel 6 with Linux 5.10 + AOSP 12 (7,250 LoC modified).

[11] Ariadne — CMU-SAFARI (HPCA 2025)

Yu Liang et al. "Ariadne: A Hotness-Aware and Size-Adaptive Compressed Swap Technique for Fast Application Relaunch." IEEE HPCA 2025. arXiv:2502.12826, GitHub

Hotness-aware data organization + size-adaptive compression (small chunks for hot data, large for cold) + proactive decompression. On Pixel 7: 50% reduction in relaunch latency, 15% less CPU for compression/decompression.

[12] ASAP — SNU-ARC (USENIX ATC 2021)

Sam Son et al. "ASAP: Fast Mobile Application Switch via Adaptive Prepaging." USENIX ATC 2021. PDF

Addresses decompression latency during app switch by combining footprint estimators for file-backed and anonymous pages with efficient prepaging. 22–28% reduction in switch time.

[13] oaRAM (ACM TACO 2025)

"Object-Aware Memory Compression for Smartphones." ACM TACO, 2025. DOI

Proposes an object-aware compressed memory format that leaves reference information intact so GC threads can interpret compressed pages — addressing the fundamental incompatibility between zram's LZ4 format and garbage collection in managed runtimes (Java/Android).

Hardware Compression

[14] IBM Memory Expansion Technology (MXT)

Tremaine et al. "IBM Memory Expansion Technology (MXT)." IBM J. Res. Dev. 45(2), 2001. IEEE

First commercial hardware memory compression. A large low-latency cache between the processor bus and compressed main memory, with dedicated hardware performing real-time compression/decompression. Achieved >2× effective memory with negligible performance penalty.

[15] QZRAM (MDPI Applied Sciences 2023)

"QZRAM: A Transparent Kernel Memory Compression System with QAT Accelerator." MDPI Applied Sciences 13(18), 2023.

Offloads zram compression to Intel QAT hardware. Reduces CPU usage by 49.2%, increases effective memory by 1.66× vs software zram.

[16] Compresso (MICRO 2018)

Choukse et al. "Compresso: Pragmatic Main Memory Compression." IEEE/ACM MICRO 2018. PDF

First main-memory compression architecture running unmodified OSes. 1.85× average compression, 24% speedup for single-core, 27% for multi-core.

Earlier Linux Work

[17] Cervera, Cortes, Becerra (USENIX ATC 1999)

"Improving Application Performance through Swap Compression." USENIX ATC 1999. PDF

One of the earliest Linux-specific implementations of compressed swap. With ~1 MB cache sizes, performance improved by up to 2× vs uncompressed swap.

[18] Tuduce & Gross (USENIX ATC 2005)

Irina Chihaia Tuduce, Thomas Gross. "Adaptive Main Memory Compression." USENIX ATC 2005. PDF

Dynamically adapts the split between compressed and uncompressed memory, managing fragmentation without user involvement. Directly influenced the adaptive pool management in modern zswap.

Future Directions

Virtual Swap Spaces (2026, in progress)

Led by Nhat Pham (based on earlier proposals by Yosry Ahmed), virtual swap spaces replace per-device swap tables with a single virtual swap table. This would decouple zswap from the requirement of a backing swap device, allowing it to operate like zram (purely in-memory) while retaining all kernel-managed benefits (cgroup awareness, LRU, shrinker). The swap table refactoring merged in kernel 6.18.

Compression Batching for Hardware Accelerators (2024)

The zswap_batch_compress() API and acomp_has_async_batching() crypto interface enable hardware compression engines (e.g., Intel IAA/QAT) to process multiple pages in a single batch operation, significantly improving throughput on supported hardware.

zpdesc Memory Descriptor (2025)

A new zpdesc type replaces raw struct page usage in zsmalloc, currently overlaying struct folio. This is a foundation for future memory descriptor optimizations as the kernel moves away from the monolithic struct page.
