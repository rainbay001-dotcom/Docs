SMAP 集成路线图

现代 Linux 内存管理特性盘点——按价值、工作量与战略契合度排序

SMAP 目前是一个以物理地址为驱动的分层守护进程：ACTC + AF-bit 访问跟踪、按进程的策略引擎、通过 migrate_pages() 实现基于 NUMA 节点的 L1/L2 迁移，以及（截至 2026-04-17）通过 reclaim_pages() 实现的 NVMe 换出。这覆盖了三层放置的基本工作。但自 SMAP 设计以来，Linux 内存子系统已经有了长足的发展；若干内核特性可以让 SMAP 更智能、更具响应性、更少定制。

本文档梳理了十五个集成方向，说明每个方向填补的差距，勾勒具体的集成方案，并提出分阶段的路线图。这是一份盘点，不是承诺——目的是把各种权衡摊开讲清楚，以便我们做出正确的选择。

1. 问题框架：SMAP 当前看得到与看不到的

在盘点特性之前，先给各类差距命名会有帮助。下面每一项集成方向都落在以下某个类别中：

缺失的信号（输入）

  • 压力 当前工作负载实际承受了多少痛苦。SMAP 看得到内存水平（供给侧），但看不到停顿（需求侧）。→ PSI。
  • 代 在内核自身视角下每一页有多老/多冷。SMAP 的 AF-bit 扫描器重复了内核已经在做的工作。→ MGLRU。
  • 拓扑 哪些 NUMA 节点是 DRAM、哪些是 CXL、哪些是 PMEM，以及它们的相对速度。SMAP 将其硬编码。→ HMAT / CDAT。
  • 模式 实际被访问的地址，以及延迟。AF-bit 是二值的；SPE 采样是连续的。→ ARM SPE / x86 IBS。
  • 设备 由硬件暴露的按 CXL 设备计的访问次数。→ CXL 3.0 DPA 计数器。

缺失的手段（输出）

  • 压缩层 在落盘之前先走内存内的压缩存储。→ zswap / zram。
  • 作用域 按 cgroup 回收，而不是按进程或全局回收。→ cgroup memory.reclaim。
  • 限流 防止单一租户失控消耗 swap。→ memory.swap.{max,high}。
  • 异步 I/O 对 swap 设备 I/O 管线的显式控制。→ io_uring。
  • 缺页时刻 在缺页时刻做放置决策，而非事后处理。→ userfaultfd。

缺失的框架集成

  • 框架 基于 kprobe 的自定义符号查找在跨内核版本时很脆弱。→ eBPF 内省。
  • 框架 策略是编译进去的 C 代码；难以做实验。→ eBPF 可加载策略。
  • 框架 SMAP 重新实现了访问跟踪、过滤、配额——DAMON 已经全都具备。→ DAMON 操作集。

研究 / 长期

  • 研究 静态阈值对大多数工作负载都是错误的。→ 强化学习自动调优。
  • 研究 通过 CXL 互联或 RDMA 在多台机器间共享内存池。→ RDMA 跨节点。

2. 当前架构快照

SMAP 今天据以行动的信号：ACTC 计数器、AF-bit 扫描、L2 水位线、按进程的冷窗口计数。手段：migrate_pages()、reclaim_pages()。下文所有其他内容都是对这两个集合的潜在扩展。

3. 十五个方向

每个方向以统一模板呈现：填补的差距、具体集成方案、工作量、风险 / 依赖。工作量按单个工程师估算：天、周、月。

3.1 信号 PSI —— 压力停顿信息 天

差距： SMAP 看得到内存水平（空闲、已用、水位线），但看不到停顿。一个系统可能空闲内存充足却仍在抖动；也可能空闲内存很低但系统空闲。PSI 上报任务因内存、IO 或 CPU 阻塞所占的墙钟时间比例——这才是实际的"痛苦信号"。

openEuler 可用性： 已确认。kernel/sched/psi.c + CONFIG_PSI，以及 openEuler 扩展 CONFIG_PSI_FINE_GRAINED（按原因细分）与 CONFIG_PSI_CGROUP_V1（v1 层级）。

文件 / API：

  • /proc/pressure/{cpu,memory,io,irq} —— 标准的 2 行格式（some/full × avg10/60/300/total）。
  • /proc/pressure/stat —— openEuler 细粒度：global_memory_reclaim、cgroup_memory_reclaim、swap、compact、cgroup_async_memory_reclaim、cpu_cfs_bandwidth、cpu_qos。
  • /sys/fs/cgroup/<path>/memory.pressure + pressure.stat —— 同样的文件，按 cgroup 提供。
  • 触发通道： 向文件写入 "some 150000 1000000"，然后 poll(fd, POLLPRI)——内核会在阈值触发（任意 1 秒窗口内停顿达到 150 ms）时唤醒你。

集成：

  1. 新增模块 plugins/smap/src/user/sense/psi_monitor.{h,c}——每个 tick 读取全局及按 cgroup 的 PSI，外加可选的触发 fd。
  2. 扩展 SceneContext，增加 PSIMetrics。场景分类器优先使用 PSI，再回退到当前的启发式。
  3. 策略调优：HEAVY（some_avg10 > 10%）→ 降低冷阈值、增大批大小。UNSTABLE（剧烈导数）→ 本轮跳过换出。
  4. 闭环（拓展目标）：在换出前后采样 PSI 变化量，验证驱逐是否真的缓解了痛苦。

风险： PSI 必须在构建时启用（CONFIG_PSI=y，可选 psi=1 命令行参数）。细粒度文件仅在 CONFIG_PSI_FINE_GRAINED=y 下存在。需要优雅回退。另外：cgroup PSI 需要 v2（或 v1 + 扩展）。

为何特别适合 SMAP： 差异化卖点。DAMON 和 damon_reclaim 目前都不使用 PSI——它们是水位线驱动的。SMAP + PSI = 可度量地比现成工具更聪明。

3.2 信号 MGLRU —— 多代 LRU 周

差距： SMAP 的冷窗口跟踪器重复了 MGLRU 已经在做的工作。MGLRU 通过调度器集成的采样将页面按代（gen 0 = 最新，gen N = 最老）老化；这是内核关于"这一页有多老"的权威答案。并行运行两个老化器既浪费 CPU，又可能得出矛盾结论。

openEuler 可用性： MGLRU 于 6.1 合并到上游；OLK-6.6 应已包含。检查 /sys/kernel/mm/lru_gen/enabled。

文件 / API：

  • /sys/kernel/mm/lru_gen/enabled —— 开关。
  • /sys/kernel/mm/lru_gen/admin —— 查询按 memcg 的代统计。
  • /sys/kernel/debug/lru_gen —— 读：按 memcg/node/zone 的各代计数表。写：强制老化命令。
  • tracepoint：trace_mglru_*，位于 tracefs/events/mglru/。
  • 内核内部：struct lru_gen_folio、folio_lru_gen(folio)。

集成：

  1. 用户态：读取目标 memcg 的 /sys/kernel/debug/lru_gen，从 gen N（最老）页面中筛选换出候选。跳过 SMAP 自身的扫描。
  2. 内核侧（进阶）：在 do_swap_out 的隔离检查期间读取 folio_lru_gen(folio)——仅当代 ≥ 阈值时才驱逐。便宜的内联过滤器。
  3. 对齐：SMAP 的 Scene 模型可以参考代分布——如果大多数页面都集中在最新的 2 代，就没什么可驱逐的。

风险： MGLRU 的内部 ABI 不稳定；读取 debug 文件虽受认可，但解析脆弱。内核侧使用意味着要通过 kallsyms 查找 folio_lru_gen 相邻符号。

收益： 有可能彻底移除冷窗口跟踪器，或至少以更低频率运行。长期回报是降低 SMAP 的每 tick CPU 开销。

3.3 信号 HMAT / CDAT —— 层拓扑发现 周

差距： SMAP 硬编码 NUMA0-3 = 本地 DRAM、NUMA4 = 远端 CXL。移植时不成立：不同的板子有不同的拓扑。ACPI HMAT（Heterogeneous Memory Attribute Table）及其后继 CDAT（Coherent Device Attribute Table）直接暴露每个节点的性能特征。

文件 / API：

  • /sys/devices/system/node/nodeN/access0/initiators/ —— 哪些 CPU 与此内存"相近"。
  • /sys/devices/system/node/nodeN/access0/{read,write}_{latency,bandwidth} —— 定量性能指标。
  • /sys/bus/cxl/devices/*/ —— CXL 设备清单。
  • 内核：include/linux/node.h —— 内核内访问等级查询。

集成：

  1. 新增模块 plugins/smap/src/user/topology/tier_discovery.{h,c}——启动时遍历 sysfs，按延迟/带宽的四分位把节点归入各层。
  2. 用拓扑驱动的谓词替换硬编码的 GetNrLocalNuma()、is_numa_remote()、NotInAttrL2()。
  3. 加入 3 层及以上模式：如果 HMAT 显示 L1 DRAM → L2 CXL → L3 CXL（更远/更慢），SMAP 应规划多跳降级。

风险： 较老的平台可能不填充 HMAT。sysfs 返回空时需要回退到现有硬编码布局。CXL 设备发现随内核版本而异。

下游收益： 此工作的整洁性同时为 #5（CXL.mem 计数器）奠基——二者都依赖可靠的 CXL 设备枚举。

3.4 信号 SPE / PMU —— 硬件访问采样 月

差距： AF-bit 扫描是二值的（在某个窗口内是否访问），且需要完整的 PTE 遍历。ARM Statistical Profiling Extension（SPE）连续采样 load/store 指令，附带物理和虚拟地址、延迟及目标层——在更低 CPU 开销下提供更丰富的信号。

平台： ARMv8.2+ / Neoverse 带 SPE。Kunpeng 920 具备（检查：/sys/bus/event_source/devices/arm_spe_0）。x86 类似物：IBS（AMD）或 PEBS+PRECDIST（Intel）。

文件 / API：

  • perf_event_open(2)，配合 PERF_TYPE_RAW、arm_spe_0 PMU 类型——安装 SPE 采样。
  • 采样记录包含 PA（PHYS_ADDR）、VA（DATA_SRC）、延迟、目标内存层级。
  • 集成先例：HeMem（OSDI '21）、Memtis（SOSP '23）。

集成：

  1. 新增内核侧采样模块 plugins/smap/src/kernel/spe_sampler.c——配置 SPE、环形缓冲区消费者，在 ACTC 之外维护一个按 PA 的访问计数表。
  2. 用户态融合：ACTC（硬件计数器、按页、稀疏）+ SPE（统计的、高粒度、跨节点）→ 更丰富的热度分数。
  3. 延迟感知的层决策：如果 SPE 显示某页的 load 有 90 ns 延迟（L2 CXL），但访问率足以放到 DRAM——执行升级。

风险： SPE 输出量大；需要高效过滤。仅部分芯片具备。采样率过高会带来开销。aarch64 专属。

研究角度： SMAP + SPE + PSI 是一个可发表的组合——三路独立信号融合用于分层决策。

3.5 信号 CXL.mem 硬件计数器 周

差距： CXL 3.0 设备暴露设备物理地址（DPA）访问计数器——CXL 硬件直接告诉你设备上哪些页面正被访问，无需软件采样开销。与 ACTC（在 CPU 侧计数）形成互补。

文件 / API：

  • /sys/bus/cxl/devices/memX/ —— 设备句柄。
  • /dev/cxl/memX —— 通过 CXL_MEM_QUERY_COMMANDS / CXL_MEM_SEND_COMMAND 发送 mailbox 命令。
  • 感兴趣的命令：GET_POISON_LIST，以及厂商特定的 DPA 热度图（依设备而异）。
  • 内核：drivers/cxl/ 子系统。

集成：

  1. 按 SMAP 的扫描间隔轮询 DPA 计数器；通过 CXL 区域映射把 DPA → 主机 PA。
  2. 与 ACTC 一起送入现有按 PA 的热度表。
  3. 特殊情形：CXL 上热但 CPU 从未访问的页面——仅加速器访问；保持在 CXL。

风险： 高度硬件相关。仅在特定 CXL 3.0+ 厂商上可用。需要厂商特定的 mailbox 命令——在不同 CXL 设备间不可移植。

3.6 动作 zswap / zram —— 内存内压缩层 天

差距： 今天的层级结构是 DRAM → CXL → NVMe。CXL 与 NVMe 之间存在约 100× 的延迟鸿沟。zswap（对后备 swap 的写回缓存）或 zram（纯压缩 RAM 块设备）能填补这个鸿沟：2–4× 压缩比、微秒级延迟。是落盘前的一份廉价保险。

openEuler 可用性： 内置。CONFIG_ZSWAP、CONFIG_ZRAM 是标配。

集成：

  1. 部署期：modprobe zram; echo $SIZE > /sys/block/zram0/disksize; swapon -p 10 /dev/zram0，并将 zswap 配置为 NVMe swap 的前端缓存（echo 1 > /sys/module/zswap/parameters/enabled）。
  2. SMAP 代码：无需改动——reclaim_pages() 会写入优先级最高的 swap 设备，也就是 zram。NVMe 成为溢出。
  3. 可选：通过 SDK 暴露 zswap 命中率以便观察。

风险： 很小。zram 占用 RAM，因此过于激进的驱逐反而可能加剧内存压力（压缩页仍消耗 RAM）。要谨慎地调整 zram 大小（典型为 RAM 的 10–20%）。

为何尽早： 最容易的收益。无需改动代码。立即改善 swap 延迟曲线。

3.7 动作 cgroup memory.reclaim —— 作用域回收 天

差距： SMAP 基于 ioctl 的 do_swap_out 以进程为单位回收。但生产工作负载以 cgroup 组织。Kernel 6.0+ 暴露了 memory.reclaim：写入 "1G"，内核便同步从该 cgroup 回收至多 1 GB。

文件 / API：

  • /sys/fs/cgroup/<path>/memory.reclaim（cgroup v2）。
  • 输入格式："1G"、"100M"、"1G type=anon"、"1G type=file"。
  • 同步：返回值 = 实际回收字节数。

集成：

  1. 新增策略变体 reclaim_by_cgroup：不走 ioctl，而是写 cgroup memory.reclaim。
  2. 驱动大部分回收，无需改动任何内核模块。保留自定义 do_swap_out 以应对需要按页控制的情形。
  3. 与 PSI 配合：当 memcg 的 PSI 飙升时，对该 memcg 写入 memory.reclaim。

风险： 由内核挑选要回收哪些页——失去了 SMAP 冷跟踪器的选择性。这条路径比自定义 ioctl 粗糙，但零维护成本。

架构备注： 这是一个回退手段，不是替代。两条路径都保留；这条用于批量回收，ioctl 用于外科手术式回收。

3.8 动作 cgroup memory.swap —— 按容器限流 天

差距： 没有机制防止单一租户消耗全部 swap 带宽。memory.swap.max 与 memory.swap.high 限制按 cgroup 的 swap 使用，并对越过 high 阈值之后的写入进行限流。

文件 / API：

  • /sys/fs/cgroup/<path>/memory.swap.max —— 硬上限。
  • /sys/fs/cgroup/<path>/memory.swap.high —— 软上限，越过后触发限流。
  • /sys/fs/cgroup/<path>/memory.swap.events —— 事件计数器（max、fail、high）。

集成：

  1. 通过 SMAP SDK 暴露：smap_set_swap_limit(pid, high_bytes, max_bytes)。
  2. 管理员配置文件把场景映射到限额：LIGHT → RAM 的 50%，HEAVY → 200%，以此类推。
  3. 监控 memory.swap.events 中的限流事件——作为输入反馈给场景分类器。

风险： 无实质风险。纯粹是对既有内核特性的配置。

3.9 动作 io_uring —— 异步 swap I/O 月

差距： reclaim_pages() 通过内核的通用 swap 路径提交 I/O，该路径近似同步且不透明。对于高吞吐换出（数百 MB/s），带完成转发的显式异步 I/O 在已发表的结果中比默认路径快 2–5×（Option A 设计文档引用过此点）。

文件 / API：

  • io_uring_setup/submit/wait —— 用户态提交 NVMe I/O。
  • IORING_OP_WRITE / IORING_OP_READ，配合固定缓冲区。
  • 需要 custom-swap（Option A）路径：SMAP 内核模块分配 swap 槽并提交 bio，用户态通过 ring 进行调度。

集成：

  1. 重启 feat/nvme-swap-kernel（Option A）：内核模块拥有槽分配器 + bio 引擎。在用户态加一层 io_uring shim 按批喂给它。
  2. 用户守护进程与内核之间的环形缓冲接口：用户通过 ring 提交"把这些 PA[] 换出"，内核执行零拷贝 bio。
  3. 完成环返回已完成/失败列表。

风险： 工作量大。重复了内核的 swap 路径。仅当 profiling 显示当前 swap I/O 是瓶颈时才值得做。

决策门槛： 仅在第 1–2 阶段交付且有吞吐量数据后再考虑。若 NVMe 未饱和，io_uring 帮不上忙。

3.10 动作 userfaultfd —— 用户态缺页处理 月

差距： SMAP 的所有决策都是事后决策：观测访问 → 决策 → 迁移。对于可预测的工作负载（已知访问模式），在缺页时刻做决策更高效——在页首次缺页被装入时就决定放置层。

文件 / API：

  • userfaultfd(2) 系统调用——获取代表给定地址范围内缺页的 fd。
  • UFFDIO_REGISTER，配合 UFFDIO_REGISTER_MODE_MISSING（缺页）或 _WP（写保护）。
  • UFFDIO_COPY / UFFDIO_ZEROPAGE / UFFDIO_CONTINUE 解决缺页。

集成：

  1. SMAP 守护进程提供"opt-in"模式：进程注册地址范围；SMAP 安装 uffd。
  2. 遇到缺页：守护进程基于访问历史，从 NVMe / CXL / 计算端在正确的层上物化该页。
  3. 遇到写保护缺页（被跟踪的冷区域）：记录为脏，重置冷计数器。

风险： 缺页驱动路径必须足够快（微秒级）。带系统调用开销的用户态守护进程很边缘。只有在能让 SMAP 提前参与决策的细分工作负载上才有真实收益。

结论： 除非有特定租户要求，否则跳过。工作量大，适用面窄。

3.11 框架 DAMON 操作集 —— 面向上游的重构 月

差距： SMAP 重新实现了：访问跟踪、冷度评分、过滤器框架、配额、水位线、动作分发。DAMON 已经具备全部这些。架构上正确的答案是让 SMAP 成为 DAMON 的一个操作集——DAMON_OPS_ACTC——从 ACTC 获取访问数据，其余交给 DAMOS scheme 处理。

文件 / API：

  • struct damon_operations —— init / update / prepare_access_checks / check_accesses / apply_scheme / cleanup。
  • damon_register_ops() —— 注册新的操作集。
  • 参考中的现有操作集：DAMON_OPS_VADDR、DAMON_OPS_PADDR、DAMON_OPS_FVADDR。

集成：

  1. 实现 struct damon_operations smap_ops。check_accesses 读取 ACTC 热度计数器；其余复用 DAMON 核心。
  2. 删除：冷窗口跟踪器（由 DAMOS 的 min_age 替代）、自定义策略引擎（由 DAMOS scheme 替代）、自定义 ioctl（由 DAMOS_PAGEOUT + DAMOS_DEMOTION 替代）。
  3. 保留：ACTC 驱动、进程管理、场景模型（作为自定义的 DAMOS 配额/水位线插件）。
  4. 上游候选：DAMON_OPS_ACTC 模块可提交到 linux-mm。

风险： 大型重构。会破坏 SDK 的 API 表面（SMAP 的 IPC 变为 DAMON sysfs）。场景分类逻辑需要新的归宿。openEuler 特有的 DAMOS_DEMOTION 动作非上游——可能并非在所有 DAMON 版本中存在。

回报： 删除数千行重复的胶水代码。免费获得 DAMON 的过滤器框架、配额、水位线与上游 bug 修复。这是长期可维护性的最强理由。

3.12 框架 eBPF —— 内省替换 周

差距： SMAP 通过 kallsyms_lookup_name() + kprobe 解析未导出符号（migrate_pages、folio_isolate_lru、reclaim_pages）。这在跨内核版本、KABI 破坏以及 CONFIG 差异下都很脆弱。eBPF 迭代器与 kfunc 为大多数相同需求提供了更稳定的 API。

API：

  • bpf_iter_task_vma、bpf_iter_folio_lru、bpf_iter_pfn —— 安全枚举。
  • 内存 kfunc：bpf_folio_*、bpf_migrate_*（集合在扩充中）。
  • BTF 驱动访问内核结构体——可在结构体布局变更下存活。

集成：

  1. 新增组件 plugins/smap/src/bpf/*.bpf.c —— 用于访问扫描与 folio 枚举的 eBPF 程序。
  2. 通过 libbpf 在守护进程启动时加载。
  3. 在存在 kfunc 的地方，逐步用 BPF 调用替换 kprobe 查找。

风险： kfunc 覆盖尚不完整——SMAP 的一些操作（例如直接调用 migrate_pages）还没有 kfunc 等价物。需要混合方案。

收益： 可移植性。一份 SMAP 构建能跨若干内核版本工作，而不必为每个版本维护符号表。

3.13 框架 eBPF —— 可加载策略 月

差距： separate_strategy.c 是编译进来的 C 代码，常量是硬写死的。改策略需要重新构建 + 重新部署。BPF 可加载策略允许管理员在运行时加载 .bpf.o——不用重建即可实验。

集成：

  1. 把策略接口定义为 BPF struct_ops：classify_scene(metrics) → scene、pick_threshold(scene, age_dist) → threshold、decide_action(folio, context) → action。
  2. 硬编码的 C 策略保留作为默认；BPF 覆盖可选。
  3. 与 BCC/libbpf 工具链打通：smapctl load-policy my_policy.bpf.o。

风险： BPF verifier 的约束限制表达力。需要精心设计 API，让运维人员可以写出有意义的策略而不触碰 verifier 上限。

结论： 只有在我们期望第三方调策略时才值得做。对单一内部产品，保留 C 即可。

3.14 研究 基于 RL 的自动调优 月

差距： SMAP 中的每个常量（冷窗口阈值、批大小、扫描间隔、场景切换迟滞）目前都是手工调的。已发表的工作（Pond、Memtis、HeMem）表明，对某一工作负载有效的阈值在其他负载上会相差 >2×。学习型控制器优于静态调参。

方法：

  • 状态：PSI 指标、访问密度直方图、swap 延迟分布、各层占用、场景。
  • 动作：阈值、批大小、扫描间隔（离散化）。
  • 奖励：-alpha * p95_latency - beta * swap_write_bw + gamma * free_dram。
  • 算法：PPO 或上下文 bandit；基于离线 trace 训练，在线微调。

集成：

  1. 遥测路径：把 state/action/reward 元组转储到文件或 sqlite。
  2. 离线训练：Python + PyTorch / RLlib。
  3. 推理：导出为 onnx 小模型，在 SMAP 守护进程每个 tick 运行（亚毫秒级）。

风险： 训练数据需要多样的工作负载。运维人员很少信任黑盒控制器——需要可解释性（为什么提高了阈值？）。

可发表性： 是，只要能证明胜过静态调参。与 SPE 信号（#4）结合效果最好。

3.15 研究 RDMA 跨节点迁移 月

差距： SMAP 在单台主机内管理。在 CXL 互联集群或 RDMA 联网的数据中心中，冷页可以迁移到另一台机器的空闲 RAM，而不是本地 swap。吞吐量远高于 NVMe。

集成：

  1. 新增层 L2.5（通过 RDMA 接入的远端 RAM）。需要集群级内存分配器（例如基于 Carbink、AIFM）。
  2. 换出路径：对于冷到足以进入 NVMe、但又热到不愿接受 100 µs 延迟的页 → 通过 RDMA 发送到远端对等节点。
  3. 换入路径：低延迟缺页处理发起 RDMA 读。

风险： 需要集群基础设施。一致性（主机 A 的冷页被主机 B 的内存压力驱逐）。失败模式大量增加。

结论： 远期研究方向。不是近期 SMAP 特性。

5. 分阶段路线图

第 1 阶段 —— 轻松收益（2026 Q2，约 3 周）

  • #6 zswap/zram —— 部署期配置；SMAP 零代码改动。
  • #7 memory.reclaim —— 作为策略中的替代回收路径加入。
  • #8 memory.swap 限流 —— SDK 表面；管理员配置。
  • #1 PSI（Level 1 & 2） —— 感知 + 触发模式，驱动场景分类器。

结果：SMAP 用上了现代 cgroup/swap/压力 基础设施；立即可以测量到延迟与租户隔离的改善。

第 2 阶段 —— 信号丰富化（2026 Q3，约 6 周）

  • #2 MGLRU 集成 —— 替换或补充冷窗口跟踪器。
  • #3 HMAT/CDAT 拓扑发现 —— 移除硬编码的 NUMA 布局。
  • #12 eBPF 内省 —— 在可能之处替换基于 kprobe 的查找。
  • #5 CXL.mem 计数器（若硬件可用）。

结果：每 tick 开销更低、在不同板子上可移植、信号融合更好。

第 3 阶段 —— 框架成熟化（2026 Q4，约 10 周）

  • #9 io_uring（仅当第 1–2 阶段的 profile 显示 swap I/O 是瓶颈时）。
  • #11 DAMON 操作集重构 —— 大规模清理；为 SMAP 开辟上游贡献路径。

结果：架构清爽。维护成本下降。上游贡献路径打开。

第 4 阶段 —— 研究（2027+）

  • #4 基于 SPE 的访问采样 —— 与 RL 结合可成为论文级工作。
  • #14 RL 自动调优 —— 基于第 1–3 阶段的遥测训练。
  • #13 eBPF 策略 —— 仅当有产品化路径要求时。
  • #15 RDMA / #10 userfaultfd —— 仅在有具体工作负载需求时评估。

明确的非目标

看起来诱人，但我建议不要追求的条目，除非出现具体的驱动力：

  • #10 userfaultfd —— 用例窄、工作量大，用户态缺页路径慢。
  • #13 eBPF 策略 —— 增加表面积但没有明确的客户。
  • #15 RDMA 跨节点 —— 集群级问题，不是主机级特性。
