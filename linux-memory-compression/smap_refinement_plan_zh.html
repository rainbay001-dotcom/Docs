ubturbo SMAP 优化与重构计划

源码仓库：/Users/ray/Documents/Repo/ubturbo

审阅范围：src/smap、plugins/smap、plugins/rmrs/src/ubturbo_plugin/common/smap 以及相关测试。

本文将 SMAP 审阅结果整理为面向实现的优化计划，重点覆盖当前架构、主要风险点、 安全加固、可维护性、可观测性以及迁移策略优化。

总览

ubturbo 中有两层 SMAP 相关代码：

  • src/smap：面向 UBTurbo 的 IPC 客户端与服务端桥接层。它负责编码 SMAP 请求、通过 UBTurbo IPC 发送请求、动态加载 libsmap.so，并分发到 SMAP API。
  • plugins/smap：真正的 SMAP 实现包，包含用户态管理、内核 tiering 代码、迁移 wrapper、配置管理、单元测试和设计文档。

SMAP 是一个多级内存管理组件。它识别冷热页，将热数据迁移到本地或快速内存， 将冷数据迁移到远端或慢速内存，并调整进程级内存分配。现有设计文档已经覆盖了 冷热排序、自适应迁移参数、多进程调度、硬件 HIST 滑窗扫描等核心策略。

整体架构

这个分层边界是合理的：UBTurbo 负责服务生命周期和 IPC，libsmap.so 负责实际的内存分级逻辑。优化计划会保留这个边界，重点提升桥接层可靠性、 部署可配置性、策略可观测性和迁移决策质量。

主要发现

1. IPC 编解码需要加强输入校验

多个 decode 路径会先从 IPC buffer 中读取 len 字段，然后在确认 len 为正且不超过上限之前，就参与 payload 大小计算。重点区域包括 SmapAddProcessTrackingCodec::DecodeRequest、 SmapRemoveProcessTrackingCodec::DecodeRequest 和 SmapEnableProcessMigrateCodec::DecodeRequest。

风险：

  • 异常 IPC 消息可以传入负数或超大长度。
  • 有符号 int 在和 size_t 混合计算时可能被提升为很大的无符号值。
  • 对固定数组的拷贝过度依赖后续 memcpy_s 兜底，而不是在前置条件阶段明确拒绝非法输入。

建议增加公共校验工具：

static bool IsCountInRange(int count, int max)
{
    return count > 0 && count <= max;
}

static bool CheckedAdd(size_t a, size_t b, size_t *out)
{
    if (SIZE_MAX - a < b) {
        return false;
    }
    *out = a + b;
    return true;
}

static bool CheckedMul(size_t a, size_t b, size_t *out)
{
    if (a != 0 && b > SIZE_MAX / a) {
        return false;
    }
    *out = a * b;
    return true;
}

在计算 sizeof(pid_t) * len、sizeof(uint32_t) * len 等 payload 大小之前，先校验 len。

2. Codec 实现重复较多

codec 文件反复出现相同模式：分配 TurboByteBuffer、拷贝字段、 编码 int 响应、解码 int 响应、错误时重置 buffer。

建议抽出公共函数：

int EncodePlain(TurboByteBuffer &buffer, const void *src, size_t size);
int DecodePlain(const TurboByteBuffer &buffer, void *dst, size_t size);
int EncodeIntResponse(TurboByteBuffer &buffer, int value);
int DecodeIntResponse(const TurboByteBuffer &buffer, int &value);

对于复合请求，可以使用小型 append writer：

class BufferWriter {
public:
    explicit BufferWriter(TurboByteBuffer &buffer, size_t size);
    int Append(const void *src, size_t size);
};

3. 客户端 buffer 应使用 RAII

客户端代码在每个函数中手动 delete[] send.data 和 delete[] recv.data。随着 API 增长，这种写法容易在新增早退路径时遗漏释放。 

class ByteBufferGuard {
public:
    TurboByteBuffer buffer{};

    ~ByteBufferGuard()
    {
        delete[] buffer.data;
    }

    TurboByteBuffer *operator->()
    {
        return &buffer;
    }
};

之后客户端 API 可以使用 ByteBufferGuard send; 和 ByteBufferGuard recv;。

4. 硬编码路径应改为配置项

当前硬编码路径包括：

  • /dev/shm/ubturbo_page_type.dat
  • /usr/lib64/libsmap.so
  • /dev/shm/smap_config

建议在配置中提供默认值：

smap.libsmap_path = /usr/lib64/libsmap.so
smap.page_type_path = /dev/shm/ubturbo_page_type.dat
smap.config_path = /dev/shm/smap_config

5. 动态符号加载日志应更明确

turbo_module_smap.cpp 使用 dlsym 解析很多 SMAP 符号， 但主要做聚合失败判断。当部署了旧版本或不匹配的 libsmap.so 时， 日志应明确指出缺失的符号名。 

struct SmapSymbol {
    const char *name;
    void **target;
};

使用表驱动循环加载，并对每个缺失符号单独打日志。

6. 构建脚本存在路径判断问题

plugins/smap/build.sh 中，boundscheck 库判断包含了多余空格：

if [[ ! -f "$LIB_BOUNDS_CHECK_FILE " ]]

应修改为：

if [[ ! -f "$LIB_BOUNDS_CHECK_FILE" ]]

同时建议统一为 shell 脚本中的路径变量加双引号。

7. 单元测试脚本不应修改源码

test/run_ut.sh 在构建测试前会移除生产源码中的 static：

find ${dir} -type f -name "*.cpp" | xargs -i sed -i "s/\bstatic\b//g" {}

不要原地改写源码。更合适的方案包括：

  • 在 test/build/generated 下编译测试专用源码副本；
  • 通过 #ifdef UNIT_TEST 暴露少量测试 hook；
  • 将私有 helper 拆到 internal translation unit，并提供测试可见头文件；
  • 在可行处使用 linker wrapping 或 mock。

8. NUMA 探测应避免 shell 命令

plugins/smap/src/user/smap_interface.c 使用：

popen("numastat -cvm", "r")

建议替换为：

  • /sys/devices/system/node/nodeN；
  • libnuma API；
  • SMAP 初始化阶段缓存的拓扑信息；
  • 内核模块已经可见的 node mask。

算法层优化

迁移滞回

使用不同的 promote/demote 阈值，避免页面在本地和远端内存之间来回震荡。

迁移 ROI 评分

只有当预期时延收益大于迁移成本，并且预算允许时才执行迁移。

进程级预算

限制每周期页数、每秒迁移字节、扫描 CPU 时间和并发迁移任务数。

反抖动机制

记录重复迁移，对不稳定页面或区域进行临时 pin、降权或提高滞回阈值。

运行时可观测性

暴露扫描周期、迁移周期、扫描页数、迁移页数、失败次数、迁移字节数和命中率。

Dry-run 模式

只计算候选迁移集合，不真正执行迁移，用于安全调参和策略验证。

ROI 计算草图

benefit = expected_latency_saving * expected_future_accesses
cost = migration_bytes / migration_bandwidth + CPU_overhead
score = benefit - cost

仅当 score > 0 且预算允许时才执行迁移。

测试建议

测试领域 建议覆盖内容
IPC 与 ABI 负数长度、零长度、超大长度、截断 payload、空 data、payload 溢出、非法 response 长度，以及公共 struct 的 size/alignment/offset 检查。
动态加载 缺失 libsmap.so、缺失单个符号、旧版本动态库、符号解析成功但函数返回错误、卸载/重载生命周期。
策略仿真 本地全热、远端全热、稳定热集、变化热集、噪声访问模式、重载与轻载混部。
构建卫生 脚本不修改 tracked 文件、生成物只进入 build/output 目录、shell 路径加引号、尽量通过 shellcheck。

建议实施顺序

  1. 修复 plugins/smap/build.sh 路径判断和 shell 引号问题。
  2. 停止 test/run_ut.sh 对源码做原地修改。
  3. 增加 codec 长度和范围校验 helper。
  4. 在 src/smap/client 增加 RAII buffer guard。
  5. 重构重复的 codec response helper。
  6. 将 SMAP 路径改为配置项。
  7. 使用符号表改善 dlsym 诊断日志。
  8. 增加异常 IPC 和 ABI 测试。
  9. 用 sysfs/libnuma/缓存拓扑替代 numastat shell 探测。
  10. 增加运行时可观测性和 dry-run 模式。
  11. 增加迁移滞回、ROI 评分和反抖动机制。

首个补丁建议

低风险 安全加固 不改变策略行为

一个合适的首个补丁可以包含：

  • 修复 "$LIB_BOUNDS_CHECK_FILE " 这个 typo；
  • 增加 IsCountInRange；
  • 在 payload-size 计算前校验 IPC 长度；
  • 增加负数长度和超大长度的单元测试；
  • 为客户端 cleanup 增加简单的 ByteBufferGuard。

这组改动可以提升安全性和可维护性，同时不改变 SMAP 的迁移策略行为。
