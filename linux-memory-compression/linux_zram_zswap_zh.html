1. 概述与动机

当 Linux 可用内存不足时，内核的回收路径会淘汰页面。匿名页面（堆、栈、mmap）无法被丢弃，必须被换出到持久化存储。传统交换将未压缩的 4 KiB 页面写入 HDD 或 SSD — 产生毫秒级 I/O 延迟并磨损闪存芯片。

压缩交换利用了一个关键洞察：大多数内存页面可以压缩到原始大小的 40–60%。通过花费几微秒的 CPU 时间进行压缩，内核可以在 RAM 中存储 2–3 倍以上的淘汰页面（或大幅减少磁盘 I/O）。Linux 提供了两种互补的机制：

zram 压缩RAM块设备zswap 压缩交换缓存
层次块设备驱动 (drivers/block/zram/)内存管理子系统钩子 (mm/zswap.c)
机制创建 /dev/zramN — 完全由压缩RAM支持的虚拟块设备位于真实交换设备前的透明缓存；在页面写入磁盘前拦截
是否需要真实交换？否 — 本身就是交换设备是 — 位于真实交换分区/文件前端
淘汰机制可选的回写到后端块设备自动 LRU 回写到底层交换设备
引入版本2.6.33 (staging), 3.14 (主线)3.11 (2013)

1.2 历史与演进

操作系统中的内存压缩技术有着悠久的历史：

  • 1999 — Paul Wilson 与 Scott Kaplan 发表 “Compressed Caching”，这是奠基性的学术工作，证明了使用一部分 RAM 作为淘汰页面的压缩缓存，性能优于直接将该部分 RAM 添加到页面缓存。他们在具有良好压缩率的工作负载上展示了 2–5 倍的有效内存扩展。
  • 2009 — Nitin Gupta 实现了 compcache/ramzswap（后更名为 zram），作为树外模块提供用于交换的压缩块设备。
  • 2010 — zram 进入 Linux 2.6.33 的 drivers/staging/zram/ 目录。
  • 2012 — Seth Jennings 提出 zcache/zswap 作为使用 frontswap API 的透明压缩缓存层 — 无需用户可见的块设备。
  • 2013 — Apple 在 OS X Mavericks 中推出压缩内存，在大规模上验证了此方案。zswap 合入 Linux 3.11。Android 开始默认使用 zram。
  • 2014 — zram 从 staging 移出到 drivers/block/zram/ (Linux 3.14)。微软在 Windows 10 中添加内存压缩。
  • 2017–2019 — ChromeOS 全面采用 zram；Fedora 从 Fedora 33 起默认启用 zram-swap。
  • 2023–2024 — zswap 经历重大重构：移除 frontswap 依赖（替换为直接 xarray 查找），添加 memcg 感知的 shrinker，采用 folio API。zram 添加多重压缩支持 (CONFIG_ZRAM_MULTI_COMP)。
  • 2025 — zram 添加压缩回写和异步后端设备读取。zswap 获得磁盘换入跟踪以校准 shrinker。两个子系统均显著成熟。

2. zram — 压缩RAM块设备

zram 创建虚拟块设备（/dev/zram0、/dev/zram1…），完全由压缩 RAM 支持。每个设备的行为与普通块设备相同 — 可以 mkswap + swapon，mkfs + mount，或作为原始块设备使用。最常见的用途是交换。

struct zram（每设备）

struct zram {
    struct zram_table_entry *table;      // 每页元数据数组
    struct zs_pool *mem_pool;             // zsmalloc 压缩数据池
    struct zcomp *comps[ZRAM_MAX_COMPS];  // 压缩引擎 (最多4个)
    struct gendisk *disk;                 // 块设备

    s8 num_active_comps;                  // 活跃的压缩引擎数

    // CONFIG_ZRAM_WRITEBACK
    struct file *backing_dev;             // 后端块设备
    unsigned long *bitmap;                // 后端设备块位图
    bool compressed_wb;                   // 是否将压缩数据写入后端
    u32 wb_batch_size;                    // 最大并发回写操作数
};

struct zram_table_entry（每页, ~16 字节）

struct zram_table_entry {
    unsigned long handle;        // zsmalloc 句柄 或 后端设备块索引
    union {
        unsigned long __lock;    // 位自旋锁 (ZRAM_ENTRY_LOCK)
        struct {
            u32 flags;           // 打包: size (位 0..PAGE_SHIFT) + 状态标志
            u32 ac_time;         // 访问时间 (CONFIG_ZRAM_TRACK_ENTRY_ACTIME)
        } attr;
    };
};

// 标志位 (PAGE_SHIFT 以上):
//   ZRAM_SAME          — 页面为同值填充 (全零或重复字)
//   ZRAM_WB            — 页面已回写到后端设备
//   ZRAM_HUGE          — 不可压缩, 以未压缩方式存储
//   ZRAM_IDLE          — 自上次标记空闲以来未被访问
//   ZRAM_INCOMPRESSIBLE — 无算法能压缩该页面
//   ZRAM_COMP_PRIORITY — 2位字段: 使用了哪个压缩算法 (0–3)
设计洞察：每页元数据仅约 16 字节。对于 16 GB 的 zram 设备（4M 页），元数据表仅占 64 MB — 约为虚拟大小的 0.4%。压缩后大小编码在 flags 字的低位中，避免了额外的字段。

2.3 写入路径

当交换子系统向 /dev/zramN 写入一个页面时：

bio submit → zram_submit_bio()
  → zram_bio_write()              // 遍历 bio_vecs
    → zram_bvec_write()
      → zram_write_page(page, index)
          1. page_same_filled()?  → ZRAM_SAME 标志, 仅存储元素值 (无需分配!)
          2. zcomp_stream_get()   → 获取每CPU压缩流 (互斥锁)
          3. zcomp_compress()     → 压缩到 2×PAGE_SIZE 缓冲区
          4. comp_len ≥ huge?     → ZRAM_HUGE, 以未压缩方式存储 (PAGE_SIZE 分配)
          5. zs_malloc(comp_len)  → 从 zsmalloc 池分配
          6. zram_can_store_page? → 检查 limit_pages 配额
          7. zs_obj_write()       → 将压缩数据写入 zsmalloc 对象
          8. slot_lock → 更新 table[index] → slot_unlock
          9. 更新统计 (pages_stored, compr_data_size)

同值填充页面优化：在压缩之前，zram 会检查整个页面是否由单个重复的 unsigned long 值填充（常见于零页面）。如果是，则仅存储该填充值 — 无需压缩，无需分配。这处理了非常常见的零初始化内存情况。

2.4 读取路径

page fault → bio submit → zram_submit_bio()
  → zram_bio_read()
    → zram_bvec_read()
      → zram_read_page(page, index)
          slot_lock(index)
          检查标志:
            ZRAM_SAME → zram_fill_page() (memset_l)
            ZRAM_WB   → read_from_bdev() (异步 bio 到后端设备)
            ZRAM_HUGE → zs_obj_read_begin() → copy_page() (无需解压缩)
            正常      → read_compressed_page():
                          zcomp_stream_get(comps[priority])
                          zs_obj_read_begin(handle)  → 获取压缩数据指针
                          zcomp_decompress()          → 解压缩到页面
                          zs_obj_read_end(handle)
          mark_slot_accessed()  → 清除 ZRAM_IDLE 标志
异步后端设备读取：当页面已被回写 (ZRAM_WB) 时，zram 向后端设备发起异步 bio。完成回调函数 zram_async_read_endio() 在高优先级工作队列中处理解压缩（如果启用了 compressed_wb）— 避免在中断上下文中执行解压缩。

2.5 回写到后端设备

启用 CONFIG_ZRAM_WRITEBACK 后，zram 可以将冷页面或不可压缩页面卸载到真实块设备，释放 RAM。这是通过 sysfs 由用户触发的，而非自动执行：

# 设置
echo /dev/sda2 > /sys/block/zram0/backing_dev

# 触发空闲页面回写
echo all > /sys/block/zram0/idle            # 标记所有页面为空闲
# ... 等待一段时间 ...
echo "type=idle" > /sys/block/zram0/writeback

# 或回写大页（不可压缩）页面
echo "type=huge" > /sys/block/zram0/writeback

回写管道使用异步 bio 提交，并具有可配置的批处理大小（wb_batch_size，默认 32）。启用 compressed_wb=1 时，压缩数据会直接写入后端设备而无需先解压缩，从而节省 CPU。

2.6 空闲页面重压缩

启用 CONFIG_ZRAM_MULTI_COMP 后，zram 支持最多 4 种压缩算法（1 个主要 + 3 个辅助）。最初使用快速算法（如 lzo-rle）压缩的冷页面，可以使用更慢但压缩率更高的算法（如 zstd）进行重压缩：

# 配置辅助算法
echo "algo=zstd priority=1" > /sys/block/zram0/recomp_algorithm

# 标记页面为空闲，然后重压缩
echo all > /sys/block/zram0/idle
echo "type=idle algo=zstd max_pages=10000" > /sys/block/zram0/recompress

重压缩逻辑会比较 zsmalloc 的大小类索引（而非原始字节数）来判断重压缩是否真的能节省内存 — 如果更小的字节数映射到相同的大小类，则不值得执行重压缩。

lzobackend_lzo.c非常快2.5–3×经典 LZO
lz4backend_lz4.c快2.2–2.8×典型默认值；参数: level

每个后端实现一个简单的虚函数表 (struct zcomp_ops)：compress、decompress、create_ctx、destroy_ctx、setup_params、release_params。CPU 热插拔回调确保每 CPU 流被正确分配/释放。

3. zswap — 压缩交换缓存

zswap 是位于交换子系统和实际交换设备之间的透明压缩缓存。当内核决定换出一个页面时，zswap 会拦截它，压缩后存储在 RAM 中。只有当 zswap 的内存池满时，才会将最冷的页面淘汰（回写）到真实交换设备。

与 zram 不同，zswap 不是块设备 — 它直接挂接到内存管理层（mm/page_io.c），对用户空间完全透明。仍然需要一个真实的交换设备作为后端存储。

3.3 存储路径

页面回收 (vmscan.c / kswapd)
  → swap_writepage() in page_io.c
    → zswap_store(folio)                   // 在磁盘 I/O 前拦截
        1. 检查 static key (禁用时的快速路径)
        2. 检查 memcg zswap.max 限制
        3. 检查全局池限制 (zswap_check_limits)
        4. 获取当前池引用 (RCU安全)
        5. 对 folio 中的每个页面:
           → zswap_store_page(page, objcg, pool)
              a. 分配 zswap_entry (kmem_cache, NUMA本地)
              b. zswap_compress(page, entry, pool)
                 - acomp_ctx_get_cpu_lock() → 每CPU上下文
                 - SG 输入来自 page, 输出到缓冲区
                 - crypto_acomp_compress() + crypto_wait_req()
                 - 如 comp_len ≥ PAGE_SIZE → 存储未压缩 (如启用WB)
                 - zs_malloc() → zs_obj_write() → 存入 zsmalloc
              c. xa_store() → 插入 xarray (如重脏则释放旧条目)
              d. obj_cgroup_charge_zswap() → memcg 计费
              e. list_lru_add() → 添加到 LRU
              f. entry->referenced = true → 二次机会保护
        6. 计数 ZSWPOUT 事件
        7. 返回 true → page_io.c 跳过磁盘写入!

3.4 加载路径

缺页异常 (do_swap_page)
  → swap_read_folio() in page_io.c
    → zswap_load(folio)
        1. xa_load(tree, offset) → 查找条目 (RCU安全, 无锁)
        2. 如未找到 → 返回 -ENOENT → 回退到磁盘读取
        3. zswap_decompress(entry, folio)
           - 获取每CPU acomp 上下文
           - 如 length == PAGE_SIZE → memcpy (以未压缩方式存储)
           - 否则 → crypto_acomp_decompress() + 等待
        4. 标记 folio 为最新
        5. 如 swapcache → xa_erase + 释放条目 (swapcache 此时为权威)
        6. 计数 ZSWPIN 事件
        7. 返回 0 → page_io.c 跳过磁盘读取!

3.5 淘汰与回写

当压缩池超过 RAM 的 max_pool_percent（默认 20%）时，zswap 必须将条目淘汰到真实交换设备：

二次机会 LRU 算法

Shrinker 遍历 LRU 链表。每个条目有一个 referenced 位：

  • 首次遇到，referenced=true：清除该位，将条目移到 LRU 尾部。页面在被淘汰前获得"二次机会"。
  • 再次遇到，referenced=false：淘汰 — 解压缩并写入真实交换。

这防止了最近存储的页面被频繁淘汰（抖动），在不维护访问时间戳开销的情况下近似真正的 LRU 行为。

回写管道

shrink_memcg_cb() [LRU 遍历器]
  → zswap_writeback_entry(entry, swpentry)
      1. swap_cache_alloc_folio() → 分配新 folio
      2. 验证条目仍在 xarray 中 (并发失效检查)
      3. zswap_decompress(entry, folio) → 解压缩到 folio
      4. xa_erase() + 释放条目 → 从 zswap 移除
      5. __swap_writepage(folio) → 写入真实交换设备
      6. 计数 ZSWPWB 事件

Shrinker 校准

Shrinker 计数函数包含两个精密的反馈机制：

  • 磁盘换入保护：跟踪每个 lruvec 的 nr_disk_swapins。如果 zswap 过度收缩，换入时页面将从磁盘读取而非从 zswap — 此计数器检测到这一情况并降低未来的收缩激进程度。
  • 压缩率缩放：如果压缩效果好（后端较小），每释放一个字节需要的淘汰次数更少。如果压缩效果差，则需要更多次淘汰。

zswap 与内存控制组 (memcg v2) 深度集成：

Cgroup 文件用途

4. zsmalloc — 共享内存分配器

zram 和 zswap 都使用 zsmalloc (mm/zsmalloc.c) 来存储压缩后的页面数据。zsmalloc 是一个专为可变大小压缩对象设计的特化 slab 分配器，以最小的碎片化优化高密度填充。

关键属性

  • 255 个大小类，16 字节粒度，覆盖 32 B 到 PAGE_SIZE
  • ZSpages 将 1–N 个物理页面组合在一起；一个 zspage 内的所有对象属于相同大小类
  • 对象可跨越页面边界 — zs_obj_read_begin() 透明地将跨页对象复制到 local_copy 缓冲区
  • Handle 间接寻址 — handle 编码 PFN + 对象索引，允许 zspage 迁移用于整理而不会使外部引用失效
  • 最佳适配分配 — 优先从最满的 zspage 分配，最小化碎片化
  • 内存整理 — zs_compact() 将部分填充的 zspage 中的对象迁移以整合空闲页面，可由内存压力或 sysfs 触发
  • 典型开销 — 10–20% 内部碎片（对象向上取整到最近的大小类）

5. 压缩算法

压缩算法的选择是 CPU 与内存之间的根本权衡。zram 和 zswap 都独立压缩单个 4 KiB 页面（无跨页上下文），这限制了压缩率但实现了随机访问。

多重压缩策略 (zram)

zram 的 CONFIG_ZRAM_MULTI_COMP 启用分层方案：

  1. 热路径（写入）：快速算法（lzo-rle 或 lz4），以最小化延迟影响
  2. 冷路径（重压缩）：高密度算法（zstd），在后台应用于空闲页面，从不太可能被立即访问的页面中回收更多内存

这实现了两全其美：低延迟写入与冷数据的高压缩率。

6. zram 与 zswap — 如何选择

zram 最适用于

  • 无可用交换设备（嵌入式、无盘、仅 tmpfs）
  • Android / ChromeOS（作为默认方案）
  • 完全控制（sysfs 提供回写、重压缩、空闲追踪旋钮）
  • 非交换用途（压缩 /tmp、ramfs 替代方案）
  • 同时使用多种算法（热: lz4, 冷: zstd）
  • 可预测的延迟（除非启用回写，否则无磁盘 I/O）

zswap 最适用于

  • 有真实交换的服务器（增强现有交换基础设施）
  • 透明部署（无需用户空间更改，仅启用模块）
  • Cgroup 感知的工作负载（容器、systemd slices）
  • 自动淘汰（shrinker + LRU，无需手动回写）
  • 减少 SSD 磨损（减少对真实交换的写入）
  • 大内存服务器（20% 的 RAM 作为压缩缓存非常可观）

可以同时使用两者吗？技术上可以 — zswap 在真实交换设备前，加上一个单独的 zram 设备。但这种做法不常见，可能导致混乱的内存记账。实际上，请选择其一：无盘/嵌入式用 zram，有交换的服务器用 zswap。

LRU 反转问题

当 zram 与低优先级磁盘交换设备组合使用时，会出现一个微妙但关键的问题。该问题由 Chris Down（Meta 内核团队，2026 年 3 月）提出：

zram 的内存使用对 cgroup 记账也是不可见的 — 内核无法按 cgroup 追踪或限制 zram 设备消耗多少 RAM，使其不适用于需要内存隔离的容器工作负载。zswap 则按 objcg 追踪每个条目。

6.1 NVMe 作为后端存储 — 分层内存层次

zram 和 zswap 都可以使用 NVMe 设备来存储内存页面，但方式根本不同。这创建了一个三层内存层次：DRAM → 压缩 RAM → NVMe，其中压缩添加了一个新层级，弥合了 DRAM 与闪存之间的延迟鸿沟。

各自如何使用 NVMe

zswap + NVMe 交换（自动）

zswap 需要真实的交换设备。NVMe 分区或交换文件作为后端存储。内核自动处理一切：

# 设置
mkswap /dev/nvme0n1p2
swapon /dev/nvme0n1p2
echo 1 > /sys/module/zswap/parameters/enabled

# 流程 (完全自动):
# 页面被淘汰
#   → 在 RAM 中压缩 (zswap 池)
#   → 池满? LRU shrinker 淘汰最冷的
#     → 解压缩 → __swap_writepage()
#       → 写入 NVMe 交换分区
#
# 换入时:
#   → 先检查 zswap (~1 µs)
#   → 未命中? 从 NVMe 读取 (~50 µs)

zswap 将 NVMe 写入量减少 60–80%，并完全消除了仍在压缩池中页面的 NVMe 读取。

zram + NVMe 后端（手动）

zram 可以可选地将冷页面回写到 NVMe 后端设备（CONFIG_ZRAM_WRITEBACK）。这由用户触发：

# 设置
echo /dev/nvme0n1p3 > /sys/block/zram0/backing_dev

# 触发空闲页面回写
echo all > /sys/block/zram0/idle
sleep 3600   # 让页面变冷
echo "type=idle" > /sys/block/zram0/writeback

# 启用 compressed_wb=1 时, 压缩数据
# 直接写入 — 无需解压缩周期:
echo 1 > /sys/block/zram0/compressed_wb
echo "type=huge" > /sys/block/zram0/writeback

compressed_wb 选项将已压缩的数据写入 NVMe，节省回写路径上的 CPU。读回时，解压缩在高优先级工作队列中异步执行。

压缩回写 — 节省 NVMe 路径上的 CPU

zram 和 zswap 都支持将已压缩的数据写入后端设备，避免解压缩→写入→重压缩的循环：

普通回写压缩回写
写入路径 解压缩 (CPU) → 写入 PAGE_SIZE 到 NVMe 直接写入 comp_len 到 NVMe (无 CPU 解压缩)
读取路径 从 NVMe 读取 PAGE_SIZE → 完成 从 NVMe 读取 comp_len → 解压缩 (CPU)
NVMe 带宽 每页完整 4 KiB 每页 ~1.5–2 KiB (已压缩)
CPU 开销 写入时解压缩, 读取时无开销 写入时无开销, 读取时解压缩
最适用于 读密集型工作负载 (快速换入) 写密集型 / SSD 寿命敏感 (更少写入字节)
SSD 磨损减少：使用压缩回写后，每个淘汰的页面仅向 NVMe 写入约 1.5–2 KiB 而非 4 KiB。结合压缩缓存带来的 60–80% 流量减少，NVMe 总写入量可减少 85–95% — 大幅延长 SSD 使用寿命。

7. 实际部署案例

Android (zram)

Android 自 Android 4.4 (KitKat) 起将 zram 作为主要交换机制。在内存受限的设备（2–4 GB RAM）上，zram 通常通过压缩提供 1–2 GB 的有效额外内存。Android 的 lmkd（低内存杀手守护进程）与 zram 协同工作 — kswapd 将页面推送到 zram，当即使压缩交换也耗尽时 lmkd 杀死应用。

ChromeOS (zram)

ChromeOS 广泛使用 zram。swap_management 服务在启动时使用 LZO 压缩配置 zram。ChromeOS 还将 zram 用于标签页丢弃决策 — 标签页在被完全丢弃之前先压缩到 zram。

Fedora / Ubuntu (zram)

Fedora 33+ 通过 systemd-zram-generator 默认启用 zram，通常配置 RAM 的 50% 作为 zram 交换，使用 lzo-rle 压缩。Ubuntu 24.10+ 也跟进采用。这取代了默认桌面安装中的传统交换分区。

Meta / Facebook (zswap)

Meta 在其服务器集群中广泛使用 zswap。Johannes Weiner（Meta 工程师）是主要维护者。他们的工作负载受益于 zswap 与其容器基础设施的透明 cgroup 集成。最近的大部分 zswap 工作（memcg 感知 shrinker、改进的淘汰机制）都源于 Meta 的生产经验。

企业级 Linux（视情况而定）

RHEL/CentOS 通常提供两者作为内核模块。默认配置因部署场景而异：容器化工作负载倾向于 zswap；裸机数据库通常禁用两者，转而使用充足的 RAM 且不启用交换。

8. 学术背景与文献

奠基性工作

[1] Compressed Caching — Wilson, Kaplan, Smaragdakis (1999)

Paul R. Wilson, Scott F. Kaplan, Yannis Smaragdakis. "The Case for Compressed Caching in Virtual Memory Systems." USENIX Annual Technical Conference, 1999.

该奠基性论文确立了：用于淘汰页面的压缩内存缓存性能优于将相同 RAM 用作未压缩页面缓存。关键洞察：在 2:1 的压缩率下，将 1/8 的 RAM 用于压缩缓存实际上将可用内存翻倍，边际成本仅为 CPU 时间。论文的自适应模型 — 根据观察到的压缩率和访问模式动态调整压缩缓存大小 — 直接影响了 zswap 的设计。

[2] zcache 与 tmem (2010–2012)

Dan Magenheimer (Oracle). "Transcendent Memory" and "zcache" patches. Linux Plumbers Conference, 2010–2012.

提出了一种"超越内存"抽象，内核可以机会性地将页面存储在常规页面分配器管理之外的压缩缓存中。zcache 是原型；其经验教训（cleancache/frontswap API 的复杂性、集成的困难）直接塑造了 zswap 的更简单设计。

[3] macOS 中的内存压缩 (2013)

Apple Inc. "OS X Mavericks Core Technologies Overview." WWDC 2013.

Apple 的实现在消费级规模上验证了压缩内存。他们的方案使用 WKdm（一种针对内存页面优化的字级字典压缩器），并将压缩集成到 Mach VM 层。与 Linux 的 zram/zswap 分离不同，macOS 使用统一的压缩内存系统。

[4] Windows 10 中的内存压缩 (2015)

Microsoft. "Memory compression" in Windows 10 Memory Manager.

Windows 10 引入了由 System 进程管理的"压缩存储"。页面在写入页面文件之前在内存中压缩，概念上类似于 zswap。Windows 实现使用 XPRESS（霍夫曼变体）压缩。

压缩算法研究

[5] LZ4 — Collet (2011)

Yann Collet. "LZ4: Extremely fast compression algorithm." 2011.

LZ4 在现代 CPU 上实现 >500 MB/s 的压缩速度和约 1.2 GB/s 的解压速度，使其成为延迟关键的压缩交换的理想选择。其固定开销设计（无霍夫曼表）意味着无论输入熵如何都有一致的性能表现。

[6] Zstandard — Collet (2015)

Yann Collet. "Zstandard: Fast real-time compression algorithm." RFC 8478, 2018. Facebook Engineering Blog, 2015.

Zstd 提供接近 DEFLATE 的压缩率和接近 LZ4 的速度。其字典压缩模式对具有重复结构的内存页面特别有效。zram 的多重压缩功能利用 zstd 对冷页面进行后台重压缩。

系统研究

[7] 理解内存压缩的权衡

Various authors. Proceedings of Linux Plumbers Conference, 2019–2024.

来自生产部署的关键发现：

  • CPU 开销并非免费：在 CPU 密集型工作负载上，zram/zswap 压缩可能降低吞吐量 5–15%。收益取决于工作负载是内存密集还是 CPU 密集。
  • 压缩率差异很大：文本/代码页面可达 3–4 倍；已压缩或加密数据约为 ~1 倍（不可压缩）。zram 的"大页"检测避免了在不可压缩页面上浪费 CPU。
  • 碎片化很重要：zsmalloc 的 16 字节粒度意味着 10–20% 的已分配内存浪费在取整上。z3fold 和 zbud 分配器（早期 zswap 版本使用）碎片化更严重但代码更简单。

[8] Zpool 分配器比较

Seth Jennings. "zbud: A Low-Latency, Low-Overhead Compressed Page Cache" and related work on z3fold (Vitaly Wool). 2013–2016.

Linux 中使用过三种压缩页面分配器：

分配器设计每页最大对象数碎片化状态
zbud每页 2 个对象（伙伴系统）2高 (~50%)已弃用
z3fold每页 3 个对象3中等 (~33%)已弃用 (6.1.112)
zsmalloc每 zspage N 个对象多个低 (~15%)当前标准

zram 和 zswap 现在都专门使用 zsmalloc。zpool 抽象层在内核 6.18（2025 年 12 月）中从 zswap 中移除 — zswap 现在直接调用 zsmalloc。

[9] Cgroup 感知的内存压缩

Nhat Pham, Johannes Weiner. "zswap: memcg-aware compressed swap cache." LKML patches, 2023–2024.

为 zswap shrinker 添加 memcg 感知是由 Meta 的容器工作负载驱动的。如果没有 cgroup 感知的淘汰，单个容器的压缩页面可能垄断 zswap 池，降低所有容器的性能。memcg 感知的 shrinker 实现了每 cgroup 的 zswap 限制和公平淘汰。

移动端 / Android 研究

[10] SWAM — Samsung Research (MobiCom 2023)

Samsung Research. "SWAM: Revisiting Swap and OOMK for Improving Application Responsiveness on Mobile Devices." ACM MobiCom 2023. arXiv:2306.08345

集成自适应交换（内存或存储）、OOM 清理器和增强的 OOM 杀手。被杀死的应用减少 6.5 倍，启动速度加快 36%，响应速度加快 41%。在 Pixel 6 上使用 Linux 5.10 + AOSP 12 实现（修改 7,250 行代码）。

[11] Ariadne — CMU-SAFARI (HPCA 2025)

Yu Liang et al. "Ariadne: A Hotness-Aware and Size-Adaptive Compressed Swap Technique for Fast Application Relaunch." IEEE HPCA 2025. arXiv:2502.12826, GitHub

热度感知的数据组织 + 大小自适应压缩（热数据用小块，冷数据用大块）+ 主动解压缩。在 Pixel 7 上：重启延迟减少 50%，压缩/解压缩 CPU 开销减少 15%。

[12] ASAP — SNU-ARC (USENIX ATC 2021)

Sam Son et al. "ASAP: Fast Mobile Application Switch via Adaptive Prepaging." USENIX ATC 2021. PDF

通过为文件缓存页和匿名页结合足迹估算器与高效预分页，解决应用切换时的解压缩延迟问题。切换时间减少 22–28%。

[13] oaRAM (ACM TACO 2025)

"Object-Aware Memory Compression for Smartphones." ACM TACO, 2025. DOI

提出一种对象感知的压缩内存格式，保留引用信息以使 GC 线程能够解读压缩页面 — 解决了 zram 的 LZ4 格式与托管运行时（Java/Android）中垃圾回收之间的根本不兼容性。

硬件压缩

[14] IBM Memory Expansion Technology (MXT)

Tremaine et al. "IBM Memory Expansion Technology (MXT)." IBM J. Res. Dev. 45(2), 2001. IEEE

首个商用硬件内存压缩。在处理器总线和压缩主内存之间放置一个大型低延迟缓存，使用专用硬件执行实时压缩/解压缩。实现了 >2 倍的有效内存且性能损失可忽略不计。

[15] QZRAM (MDPI Applied Sciences 2023)

"QZRAM: A Transparent Kernel Memory Compression System with QAT Accelerator." MDPI Applied Sciences 13(18), 2023.

将 zram 压缩卸载到 Intel QAT 硬件。CPU 使用率降低 49.2%，有效内存比软件 zram 增加 1.66 倍。

[16] Compresso (MICRO 2018)

Choukse et al. "Compresso: Pragmatic Main Memory Compression." IEEE/ACM MICRO 2018. PDF

首个在未修改操作系统上运行的主内存压缩架构。平均 1.85 倍压缩，单核加速 24%，多核加速 27%。

早期 Linux 工作

[17] Cervera, Cortes, Becerra (USENIX ATC 1999)

"Improving Application Performance through Swap Compression." USENIX ATC 1999. PDF

最早的 Linux 特定压缩交换实现之一。使用约 1 MB 的缓存大小，性能比未压缩交换提升最高 2 倍。

[18] Tuduce 与 Gross (USENIX ATC 2005)

Irina Chihaia Tuduce, Thomas Gross. "Adaptive Main Memory Compression." USENIX ATC 2005. PDF

动态调整压缩与非压缩内存之间的分配比例，无需用户参与即可管理碎片化。直接影响了现代 zswap 中的自适应池管理。

未来方向

虚拟交换空间 (2026, 进行中)

由 Nhat Pham 主导（基于 Yosry Ahmed 的早期提案），虚拟交换空间用单一虚拟交换表替代每设备交换表。这将解除 zswap 对后端交换设备的依赖，使其能够像 zram 一样运行（纯内存模式），同时保留所有内核管理的优势（cgroup 感知、LRU、shrinker）。交换表重构已在内核 6.18 中合入。

硬件加速器的压缩批处理 (2024)

zswap_batch_compress() API 和 acomp_has_async_batching() crypto 接口使硬件压缩引擎（如 Intel IAA/QAT）能够在单次批处理操作中处理多个页面，在支持的硬件上显著提升吞吐量。

zpdesc 内存描述符 (2025)

新的 zpdesc 类型取代了 zsmalloc 中对原始 struct page 的使用，当前覆盖 struct folio。随着内核逐步摆脱单体的 struct page，这为未来的内存描述符优化奠定了基础。
