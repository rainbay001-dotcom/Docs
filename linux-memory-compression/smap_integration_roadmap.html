SMAP Integration Roadmap

Modern Linux memory-management features worth absorbing into SMAP — prioritized by value, effort, and strategic fit

SMAP today is a paddr-driven tiering daemon: ACTC + AF-bit access tracking, a per-process strategy engine, NUMA-node-based L1/L2 migration via migrate_pages(), and (as of 2026-04-17) NVMe swap-out via reclaim_pages(). That covers the basic job of three-tier placement. But the Linux memory subsystem has evolved substantially since SMAP was designed; a number of kernel features would let SMAP be smarter, more reactive, and less custom.

This document catalogs fifteen integration directions, explains the gap each fills, sketches the concrete integration, and proposes a phased roadmap. It is intentionally a survey, not a commitment — the goal is to make the tradeoffs explicit so we can pick correctly.

1. Framing: What SMAP Is and Isn't Seeing

Before surveying features, it helps to name the gaps. Every integration below fills one of these buckets:

Missing signals (input)

  • pressure How much the workload is actually suffering right now. SMAP sees memory levels (supply side), not stalls (demand side). → PSI.
  • generation How old/cold each page is in the kernel's own view. SMAP's AF-bit scanner redoes work the kernel already does. → MGLRU.
  • topology Which NUMA nodes are DRAM vs CXL vs PMEM, and their relative speed. SMAP hardcodes this. → HMAT / CDAT.
  • pattern Actual addresses accessed, with latency. AF-bit is binary; SPE samples are continuous. → ARM SPE / x86 IBS.
  • device Per-CXL-device access counts exposed in hardware. → CXL 3.0 DPA counters.

Missing levers (output)

  • compressed tier In-RAM compressed storage before going to disk. → zswap / zram.
  • scope Per-cgroup reclaim instead of per-process or global. → cgroup memory.reclaim.
  • throttle Prevent runaway swap consumption by one tenant. → memory.swap.{max,high}.
  • async I/O Explicit control over swap device I/O pipeline. → io_uring.
  • fault-time Make placement decisions at page-fault time, not post-hoc. → userfaultfd.

Missing framework integration

  • framework Custom kprobe-based symbol lookup is fragile across kernels. → eBPF introspection.
  • framework Strategy is compiled-in C; hard to experiment. → eBPF-loadable policy.
  • framework SMAP re-implements access tracking, filtering, quotas — DAMON already has all of this. → DAMON ops set.

Research / long-term

  • research Static thresholds are wrong for most workloads. → RL auto-tuning.
  • research Pool memory across machines via CXL fabric or RDMA. → RDMA cross-node.

2. Current Architecture Snapshot

Signals SMAP acts on today: ACTC counters, AF-bit scans, L2 watermarks, per-process cold-window counts. Levers: migrate_pages(), reclaim_pages(). Everything else below is a potential expansion of either set.

3. The Fifteen Directions

Each direction is presented with a consistent template: gap it fills, concrete integration, effort, risks / dependencies. Rough effort is for a single engineer: days, weeks, months.

3.1 signal PSI — Pressure Stall Information days

Gap: SMAP sees memory levels (free, used, watermark) but not stalls. A system can have plenty of free memory and still thrash; or low free memory and be idle. PSI reports wall-time fraction that tasks spent blocked on memory, IO, or CPU — the actual pain signal.

openEuler availability: Confirmed. kernel/sched/psi.c + CONFIG_PSI + openEuler extensions CONFIG_PSI_FINE_GRAINED (per-cause breakdown) and CONFIG_PSI_CGROUP_V1 (v1 hierarchies).

Files / APIs:

  • /proc/pressure/{cpu,memory,io,irq} — standard 2-line format (some/full × avg10/60/300/total).
  • /proc/pressure/stat — openEuler fine-grained: global_memory_reclaim, cgroup_memory_reclaim, swap, compact, cgroup_async_memory_reclaim, cpu_cfs_bandwidth, cpu_qos.
  • /sys/fs/cgroup/<path>/memory.pressure + pressure.stat — same files per-cgroup.
  • Trigger channel: write "some 150000 1000000" to the file, then poll(fd, POLLPRI) — kernel wakes you on threshold crossing (150 ms stall in any 1 s window).

Integration:

  1. New module plugins/smap/src/user/sense/psi_monitor.{h,c} — reads global + per-cgroup PSI each tick, plus optional trigger fd.
  2. Extend SceneContext with PSIMetrics. Scene classifier inputs PSI before falling back to current heuristics.
  3. Strategy tuning: HEAVY (some_avg10 > 10%) → lower cold threshold, bigger batch. UNSTABLE (rapid derivative) → skip swap-out this round.
  4. Closed-loop (stretch): sample PSI delta pre/post swap to validate whether eviction actually reduced pain.

Risks: PSI must be enabled at build (CONFIG_PSI=y, optionally psi=1 cmdline). Fine-grained files only exist with CONFIG_PSI_FINE_GRAINED=y. Graceful fallback required. Also: cgroup PSI needs v2 (or v1 + extension).

Why SMAP specifically: Differentiator. Neither DAMON nor damon_reclaim uses PSI today — they're watermark-driven. SMAP + PSI = measurably smarter than stock tooling.

3.2 signal MGLRU — Multi-Generational LRU weeks

Gap: SMAP's cold-window tracker redoes work MGLRU already does. MGLRU ages pages into generations (gen 0 = newest, gen N = oldest) via scheduler-integrated sampling; this is the kernel's canonical "how old is this page?" answer. Running two agers in parallel wastes CPU and can disagree.

openEuler availability: MGLRU merged upstream 6.1; OLK-6.6 should have it. Check /sys/kernel/mm/lru_gen/enabled.

Files / APIs:

  • /sys/kernel/mm/lru_gen/enabled — on/off.
  • /sys/kernel/mm/lru_gen/admin — query per-memcg generation stats.
  • /sys/kernel/debug/lru_gen — read: table of generation counts per memcg/node/zone. Write: force-age commands.
  • Tracepoints: trace_mglru_* via tracefs/events/mglru/.
  • Kernel-internal: struct lru_gen_folio, folio_lru_gen(folio).

Integration:

  1. User-space: read /sys/kernel/debug/lru_gen for the target memcg and skim gen N (oldest) pages as swap-out candidates. Skip SMAP's own scan.
  2. Kernel-side (advanced): read folio_lru_gen(folio) during do_swap_out's isolation check — only evict if gen ≥ threshold. Cheap in-line filter.
  3. Alignment: SMAP's Scene model can be informed by generation distribution — if most pages are in the newest 2 gens, there's nothing to evict.

Risks: MGLRU's internal ABI isn't stable; reading debug files is sanctioned but parsing is brittle. Kernel-side use means kallsyms lookup of folio_lru_gen-adjacent symbols.

Win: Potentially remove the cold-window tracker entirely, or at minimum run it at a lower frequency. The longer-term prize is cutting SMAP's per-tick CPU cost.

3.3 signal HMAT / CDAT — Tier Topology Discovery weeks

Gap: SMAP hardcodes NUMA0-3 = local DRAM, NUMA4 = remote CXL. This doesn't survive porting: different boards have different topologies. ACPI HMAT (Heterogeneous Memory Attribute Table) and its successor CDAT (Coherent Device Attribute Table) expose per-node performance characteristics directly.

Files / APIs:

  • /sys/devices/system/node/nodeN/access0/initiators/ — which CPUs are "close" to this memory.
  • /sys/devices/system/node/nodeN/access0/{read,write}_{latency,bandwidth} — quantitative performance.
  • /sys/bus/cxl/devices/*/ — CXL device inventory.
  • Kernel: include/linux/node.h — in-kernel access-class queries.

Integration:

  1. New module plugins/smap/src/user/topology/tier_discovery.{h,c} — walks sysfs at startup, classifies nodes into tiers by latency/bandwidth quartiles.
  2. Replace hardcoded GetNrLocalNuma(), is_numa_remote(), NotInAttrL2() with topology-driven predicates.
  3. Add a 3+ tier mode: if HMAT shows L1 DRAM → L2 CXL → L3 CXL (farther/slower), SMAP should plan multi-hop demotion.

Risks: Older platforms may not populate HMAT. Need fallback to the existing hardcoded layout when sysfs returns empty. CXL device discovery varies by kernel version.

Downstream benefit: Cleanliness of this work also enables #5 (CXL.mem counters) — both rely on robust CXL device enumeration.

3.4 signal SPE / PMU — Hardware Access Sampling months

Gap: AF-bit scanning is binary (accessed yes/no within a window) and requires a full PTE walk. ARM Statistical Profiling Extension (SPE) continuously samples load/store instructions with physical and virtual addresses, latency, and target tier — richer signal at lower CPU overhead.

Platform: ARMv8.2+ / Neoverse with SPE. Kunpeng 920 has it (check: /sys/bus/event_source/devices/arm_spe_0). x86 analog: IBS (AMD) or PEBS+PRECDIST (Intel).

Files / APIs:

  • perf_event_open(2) with PERF_TYPE_RAW, arm_spe_0 PMU type — installs SPE sampling.
  • Sample record includes PA (PHYS_ADDR), VA (DATA_SRC), latency, target memory level.
  • Integration precedents: HeMem (OSDI '21), Memtis (SOSP '23).

Integration:

  1. New kernel-side sampling module plugins/smap/src/kernel/spe_sampler.c — configures SPE, ring-buffer consumer, populates a per-PA access-count map alongside ACTC.
  2. User-space merges: ACTC (hardware counters, per-page, sparse) + SPE (statistical, high-granularity, cross-node) → richer hotness score.
  3. Latency-aware tier decisions: if SPE shows a page's loads have 90 ns latency (L2 CXL), but access rate justifies DRAM — promote.

Risks: SPE output volume is large; need efficient filtering. Only some chips have it. Overhead if sampling rate too high. Aarch64-specific.

Research angle: SMAP + SPE + PSI is a publishable combination — three independent signal sources fused for tiering decisions.

3.5 signal CXL.mem Hardware Counters weeks

Gap: CXL 3.0 devices expose Device Physical Address (DPA) access counters — the CXL hardware tells you which pages on the device are being accessed, with no software sampling cost. Complements ACTC (which counts at the CPU side).

Files / APIs:

  • /sys/bus/cxl/devices/memX/ — device handle.
  • /dev/cxl/memX — mailbox commands via CXL_MEM_QUERY_COMMANDS / CXL_MEM_SEND_COMMAND.
  • Commands of interest: GET_POISON_LIST, vendor-specific DPA heatmaps (device-dependent).
  • Kernel: drivers/cxl/ subsystem.

Integration:

  1. Poll DPA counters at SMAP's scan interval; translate DPA → host PA via the CXL region mapping.
  2. Feed into the existing per-PA hotness map alongside ACTC.
  3. Special case: pages hot on CXL but never accessed by CPU — accelerator-only access; keep on CXL.

Risks: Very hardware-dependent. Only viable on specific CXL 3.0+ vendors. Vendor-specific mailbox commands needed — not portable across CXL devices.

3.6 action zswap / zram — Compressed In-RAM Tier days

Gap: Today's tier hierarchy is DRAM → CXL → NVMe. Between CXL and NVMe there's a gap of ~100× latency. zswap (write-back cache to backing swap) or zram (pure compressed RAM block device) closes it: 2–4× compression ratio, microsecond-class latency. Cheap insurance before disk.

openEuler availability: Built in. CONFIG_ZSWAP, CONFIG_ZRAM standard.

Integration:

  1. Deploy-time: modprobe zram; echo $SIZE > /sys/block/zram0/disksize; swapon -p 10 /dev/zram0 + configure zswap as front-cache for the NVMe swap (echo 1 > /sys/module/zswap/parameters/enabled).
  2. SMAP code: zero change required — reclaim_pages() writes to the highest-priority swap device, which will be zram. NVMe becomes overflow.
  3. Optional: expose the zswap hit ratio via SDK for visibility.

Risks: Minimal. zram uses RAM, so overly-aggressive eviction can paradoxically increase memory pressure (compressed pages still consume RAM). Size zram carefully (10–20% of RAM is typical).

Why early: Easiest win. No code changes. Immediately improves swap latency profile.

3.7 action cgroup memory.reclaim — Scoped Reclaim days

Gap: SMAP's ioctl-based do_swap_out reclaims by-process. But production workloads are organized in cgroups. Kernel 6.0+ exposes memory.reclaim: write "1G" and kernel synchronously reclaims up to 1 GB from that cgroup.

Files / APIs:

  • /sys/fs/cgroup/<path>/memory.reclaim (cgroup v2).
  • Input formats: "1G", "100M", "1G type=anon", "1G type=file".
  • Synchronous: return value = bytes actually reclaimed.

Integration:

  1. New strategy variant reclaim_by_cgroup: instead of ioctl, write to cgroup memory.reclaim.
  2. Drives most reclaim without any kernel module changes. Keeps custom do_swap_out for per-page control cases.
  3. Pairs with PSI: when memcg PSI spikes, write memory.reclaim for that memcg.

Risks: The kernel picks which pages to reclaim — SMAP's cold-tracker selectivity is lost. This path is coarser than custom ioctl but zero-maintenance.

Architectural note: This is a fallback, not a replacement. Keep both paths; use this for bulk, ioctl for surgical.

3.8 action cgroup memory.swap — Per-Container Throttling days

Gap: No protection against one tenant consuming all swap bandwidth. memory.swap.max and memory.swap.high cap per-cgroup swap usage, throttling writes past the high threshold.

Files / APIs:

  • /sys/fs/cgroup/<path>/memory.swap.max — hard cap.
  • /sys/fs/cgroup/<path>/memory.swap.high — soft cap, throttles once crossed.
  • /sys/fs/cgroup/<path>/memory.swap.events — event counters (max, fail, high).

Integration:

  1. Expose via SMAP SDK: smap_set_swap_limit(pid, high_bytes, max_bytes).
  2. Admin config file maps scenes to limits: LIGHT → 50% of RAM, HEAVY → 200%, etc.
  3. Monitor memory.swap.events for throttle events — feed into Scene classifier.

Risks: None material. Pure configuration of existing kernel feature.

3.9 action io_uring — Async Swap I/O months

Gap: reclaim_pages() submits I/O through the kernel's generic swap path, which is synchronous-ish and opaque. For high-throughput swap-out (hundreds of MB/s), explicit async I/O with completion steering beats the default by 2–5× in published results (Option A design doc referenced this).

Files / APIs:

  • io_uring_setup/submit/wait — user-space submission of NVMe I/O.
  • IORING_OP_WRITE / IORING_OP_READ with fixed buffers.
  • Requires custom-swap (Option A) path: SMAP kernel module allocates swap slots and submits bios, user-space orchestrates via ring.

Integration:

  1. Revive feat/nvme-swap-kernel (Option A): kernel module has slot allocator + bio engine. Add io_uring shim in user-space to feed it batches.
  2. Ring-buffer interface between user daemon and kernel: user submits "swap out these PA[]" via ring; kernel does zero-copy bio.
  3. Completion ring delivers done/failed lists.

Risks: Significant work. Duplicates the kernel's swap path. Only worth it if profiling shows current swap I/O is the bottleneck.

Decision gate: Only pursue after Phase 1–2 ship and we have throughput numbers. If NVMe isn't saturated, io_uring doesn't help.

3.10 action userfaultfd — User-Space Fault Handling months

Gap: All SMAP decisions are post-hoc: observe access → decide → migrate. For predictable workloads (known access pattern), a fault-time decision is more efficient — decide placement as the page is first faulted in.

Files / APIs:

  • userfaultfd(2) syscall — get an fd representing page faults on a given range.
  • UFFDIO_REGISTER with UFFDIO_REGISTER_MODE_MISSING for faults, _WP for writes.
  • UFFDIO_COPY / UFFDIO_ZEROPAGE / UFFDIO_CONTINUE to resolve faults.

Integration:

  1. SMAP daemon offers "opt-in" mode: process registers address ranges; SMAP installs uffd.
  2. On missing fault: daemon materializes the page from NVMe / CXL / compute on the right tier based on access history.
  3. On write-protect fault (tracked cold regions): log as dirty, reset cold counter.

Risks: Fault-driven path must be fast (microseconds). User-space daemon with syscall overhead is marginal. Realistic win only for niche workloads where SMAP is consulted pre-fault.

Verdict: Skip unless a specific tenant asks for it. High effort, narrow applicability.

3.11 framework DAMON Ops Set — Upstream-Friendly Refactor months

Gap: SMAP re-implements: access tracking, cold scoring, filter framework, quotas, watermarks, action dispatch. DAMON already has all of this. The architectural right answer is SMAP becomes a DAMON operations set — DAMON_OPS_ACTC — sourcing access data from ACTC, letting DAMOS schemes handle the rest.

Files / APIs:

  • struct damon_operations — init / update / prepare_access_checks / check_accesses / apply_scheme / cleanup.
  • damon_register_ops() — register a new ops set.
  • Existing ops sets for reference: DAMON_OPS_VADDR, DAMON_OPS_PADDR, DAMON_OPS_FVADDR.

Integration:

  1. Implement struct damon_operations smap_ops. check_accesses reads ACTC hot counters; everything else reuses DAMON core.
  2. Delete: cold-window tracker (DAMOS's min_age replaces it), custom strategy engine (DAMOS schemes replace it), custom ioctl (DAMOS_PAGEOUT + DAMOS_DEMOTION replace it).
  3. Keep: ACTC driver, process management, Scene model (as custom DAMOS quota/watermark plugins).
  4. Upstream candidate: the DAMON_OPS_ACTC module could be submitted to linux-mm.

Risks: Large refactor. Breaks SDK API surface (SMAP's IPC becomes DAMON sysfs). Scene classification logic needs a new home. openEuler-specific DAMOS_DEMOTION action is non-upstream — may not be in all DAMON versions.

Payoff: Delete thousands of lines of duplicated plumbing. Get DAMON's filter framework, quotas, watermarks, upstream bug fixes for free. Strongest case for long-term viability.

3.12 framework eBPF — Introspection Replacement weeks

Gap: SMAP resolves non-exported symbols (migrate_pages, folio_isolate_lru, reclaim_pages) via kallsyms_lookup_name() + kprobe. This is fragile across kernel versions, KABI breakage, and CONFIG differences. eBPF iterators and kfuncs offer stable-ish APIs for most of the same needs.

APIs:

  • bpf_iter_task_vma, bpf_iter_folio_lru, bpf_iter_pfn — safe enumeration.
  • Memory kfuncs: bpf_folio_*, bpf_migrate_* (expanding set).
  • BTF-driven access to kernel structs — survives struct layout changes.

Integration:

  1. New component plugins/smap/src/bpf/*.bpf.c — ebpf programs for access-scanning and folio enumeration.
  2. Load at daemon startup via libbpf.
  3. Gradually replace kprobe lookups with BPF calls where kfuncs exist.

Risks: kfunc coverage is incomplete — some SMAP operations (e.g., direct call to migrate_pages) don't have kfunc equivalents yet. Hybrid approach needed.

Win: Portability. One SMAP build works across several kernel versions instead of requiring per-version symbol tables.

3.13 framework eBPF — Loadable Strategy Policy months

Gap: separate_strategy.c is compiled-in C with baked-in constants. Changing strategy requires rebuild + redeploy. A BPF-loadable policy would let admins load .bpf.o at runtime — experiment without rebuilds.

Integration:

  1. Define policy interface as a BPF struct_ops: classify_scene(metrics) → scene, pick_threshold(scene, age_dist) → threshold, decide_action(folio, context) → action.
  2. Hardcoded C policy remains as default; BPF override optional.
  3. Tie-in to BCC/libbpf tooling: smapctl load-policy my_policy.bpf.o.

Risks: BPF verifier constraints limit expressiveness. Needs careful API design so operators can write meaningful policies without escaping verifier limits.

Verdict: Worth it only if we expect third-party policy tuning. For a single internal product, keep C.

3.14 research RL-Based Auto-Tuning months

Gap: Every constant in SMAP (cold-window threshold, batch size, scan interval, scene transition hysteresis) is currently hand-tuned. Published work (Pond, Memtis, HeMem) shows thresholds that work well for one workload fail others by >2×. A learned controller beats static tuning.

Approach:

  • State: PSI metrics, access density histogram, swap latency distribution, tier occupancy, Scene.
  • Action: threshold, batch size, scan interval (discretized).
  • Reward: -alpha * p95_latency - beta * swap_write_bw + gamma * free_dram.
  • Algorithm: PPO or contextual bandit; training offline from traces, online fine-tuning.

Integration:

  1. Telemetry path: dump state/action/reward tuples to a file or sqlite.
  2. Offline training: Python with PyTorch / RLlib.
  3. Inference: export tiny model to onnx, run in SMAP daemon each tick (sub-ms).

Risks: Training data requires diverse workloads. Operators rarely trust black-box controllers — need explainability (why did you raise threshold?).

Publishable: Yes, if it demonstrably beats static. Pair with SPE signal (#4) for best chance.

3.15 research RDMA Cross-Node Migration months

Gap: SMAP manages within a single host. In CXL-fabric-connected clusters or RDMA-networked datacenters, cold pages could migrate to another machine's spare RAM rather than local swap. Dramatically higher throughput than NVMe.

Integration:

  1. New tier L2.5 (remote RAM via RDMA). Requires cluster-wide memory allocator (e.g., based on Carbink, AIFM).
  2. Swap-out path: for pages cold enough for NVMe but hot enough to not want 100µs latency → RDMA to remote peer.
  3. Swap-in path: low-latency fault handler issues RDMA read.

Risks: Needs cluster infrastructure. Consistency (host A's cold page is evicted by host B's memory pressure). Failure modes proliferate.

Verdict: Long-horizon research. Not a near-term SMAP feature.

5. Phased Roadmap

Phase 1 — Easy wins (Q2 2026, ~3 weeks)

  • #6 zswap/zram — deploy-time config; zero SMAP code.
  • #7 memory.reclaim — add as alternate reclaim path in strategy.
  • #8 memory.swap throttling — SDK surface; admin config.
  • #1 PSI (Level 1 & 2) — sensing + triggered mode, drive Scene classifier.

Outcome: SMAP uses modern cgroup/swap/pressure infrastructure; immediately measurable latency + tenant-isolation improvements.

Phase 2 — Signal richness (Q3 2026, ~6 weeks)

  • #2 MGLRU integration — replace or complement cold-window tracker.
  • #3 HMAT/CDAT topology discovery — remove hardcoded NUMA layout.
  • #12 eBPF introspection — replace kprobe-based lookups where possible.
  • #5 CXL.mem counters (if hw available).

Outcome: Cheaper per-tick cost, portable across boards, better signal fusion.

Phase 3 — Framework maturity (Q4 2026, ~10 weeks)

  • #9 io_uring (only if Phase 1–2 profiles show swap I/O is the bottleneck).
  • #11 DAMON ops set refactor — major cleanup; positions SMAP for upstream contribution.

Outcome: Architecturally clean. Maintenance cost drops. Upstream-contribution path open.

Phase 4 — Research (2027+)

  • #4 SPE-based access sampling — paper-worthy if paired with RL.
  • #14 RL auto-tuning — trained on Phase 1–3 telemetry.
  • #13 eBPF policy — only if a productization path demands it.
  • #15 RDMA / #10 userfaultfd — evaluate only on concrete workload asks.

Explicit non-goals

Items that are tempting but I recommend not pursuing unless a specific driver appears:

  • #10 userfaultfd — narrow use case, high effort, user-space fault paths are slow.
  • #13 eBPF policy — adds surface area without clear customer.
  • #15 RDMA cross-node — cluster-level problem, not a host-level feature.
