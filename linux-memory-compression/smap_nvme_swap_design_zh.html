1. 背景与问题陈述

1.1 SMAP 现状: 仅内存层级迁移

ubturbo 中的 SMAP (Smart Memory Adaptive Platform) 管理两级内存层次结构:

层级介质延迟容量SMAP 角色
L1 本地 NUMADRAM (DDR5)~80–120 ns基础容量热页驻留于此
L2 远端 NUMACXL / PMEM / UB 互连~200–500 ns扩展池冷页降级至此

迁移通过内核的 migrate_pages() 系统调用实现, 在 plugins/smap/src/tiering/smap_migrate_wrapper.c 中通过 kprobes 进行封装。 plugins/smap/src/user/strategy/separate_strategy.c 中的策略引擎按访问频率 (硬件 ACTC 计数器或软件 AF 位扫描) 对页面排序, 选择冷的本地页进行降级, 热的远端页进行提升。

manage.h:95-99 中定义的三种迁移方向:

typedef enum {
    DEMOTE,    // 冷页 L1 → L2
    PROMOTE,   // 热页 L2 → L1
    SWAP,      // 双向交换 (冷 L1 ↔ 热 L2)
} MigrateDirection;

1.2 缺口: 没有通向持久化存储的路径

当 L1 和 L2 内存同时面临压力时, SMAP 没有泄压阀。 ubturbo_smap_urgent_migrate_out() 函数只能在 NUMA 节点之间降级 — 无法溢出到块设备。 如果远端 NUMA 也已满, 系统将触发 OOM。

当前限制: SMAP 的 MigrateDirection 枚举没有到磁盘的 SWAP_OUT 方向。 oom_migrate.h:FindPidMigrateSize() 只能为 NUMA 降级寻找候选页, 不支持磁盘 swap。 整个内核模块 (smap_migrate_pages.c) 完全基于 migrate_pages() 工作 — 从不发起块 I/O。

1.3 为什么需要 NVMe Swap

  • 内存超分: 虚拟机密度场景中, 虚拟机内存总量超过物理 DRAM + CXL 容量
  • 消除冷尾巴: 在多个扫描窗口中访问频率为零的页面正在占用昂贵的内存
  • 成本效率: NVMe 每 GB 成本约为 CXL 内存的 1/10, 约为 DRAM 的 1/50
  • 优雅降级: 与其 OOM 杀死虚拟机, 不如以 ~10–100 μs 的延迟将最冷的页面溢出到 NVMe

带延迟信息的三级层次结构提案:

层级介质延迟SMAP 操作
L1本地 DRAM~100 ns保留热页
L2远端 NUMA (CXL)~300 ns降级温页
L3NVMe SSD~10–100 μs换出最冷页面 (新增)

3. 详细设计

3.1 策略层: 三级决策引擎

当前 separate_strategy.c 中的策略做二元决策: 在 L1 和 L2 之间降级或提升。 我们将其扩展为三级决策:

SWAP_OUT 的判定条件

L2 (远端 NUMA) 中的页面满足以下条件时可被换出:

  1. 零频率持续 cold_window_threshold 个连续扫描窗口 (默认: 3)
  2. 不在白名单中 (排除 pinned、reserved 或内核页面)
  3. L2 使用率超过 l2_swap_watermark (默认: 80%)
  4. NVMe swap 空间可用 (通过 swap map 位图检查)

此设计保留了 SMAP 已有的 L1↔L2 行为, 仅将 L3 层级作为进一步的兜底。 页面严格向下流动: L1 → L2 → L3, 提升则走 L3 → L1 (跳过 L2, 以避免缺页时的额外迁移延迟)。

3.2 内核模块: swap_out / swap_in 路径

3.2.1 Swap-Out 路径

/* smap_swap.c 中的新函数 */
int smap_swap_out(struct folio **folios, unsigned int nr_folios,
                   struct smap_swap_device *sdev)
{
    for (int i = 0; i < nr_folios; i++) {
        /* 1. 在 swap map 中分配槽位 */
        slot = smap_swap_map_alloc(sdev);

        /* 2. 锁定 folio, 解除 PTE 映射, 替换为 swap 条目 */
        folio_lock(folios[i]);
        try_to_unmap(folios[i], TTU_SWAP_OUT);
        set_smap_swap_pte(folios[i], sdev, slot);

        /* 3. 提交异步写 bio 到 NVMe */
        bio = bio_alloc(sdev->bdev, 1, REQ_OP_WRITE, GFP_KERNEL);
        bio->bi_iter.bi_sector = slot_to_sector(slot);
        bio_add_folio(bio, folios[i], folio_size(folios[i]), 0);
        submit_bio(bio);

        /* 4. 完成回调中: 释放 folio */
    }
}

3.2.2 Swap-In 路径

/* 由 swap PTE 上的缺页异常触发 */
int smap_swap_in(struct vm_fault *vmf,
                  struct smap_swap_device *sdev, unsigned long slot)
{
    /* 1. 在 L1 (本地 DRAM) 上分配新 folio */
    folio = folio_alloc(GFP_HIGHUSER_MOVABLE | __GFP_THISNODE, 0);

    /* 2. 提交异步读 bio 从 NVMe 读取 */
    bio = bio_alloc(sdev->bdev, 1, REQ_OP_READ, GFP_KERNEL);
    bio->bi_iter.bi_sector = slot_to_sector(slot);
    bio_add_folio(bio, folio, folio_size(folio), 0);
    submit_bio_wait(bio); /* 缺页路径同步等待 */

    /* 3. 安装新 PTE 指向 L1 folio */
    set_pte_at(vmf->vma->vm_mm, vmf->address,
               vmf->pte, mk_pte(&folio->page, vmf->vma->vm_page_prot));

    /* 4. 释放 swap 槽位 */
    smap_swap_map_free(sdev, slot);
}
设计选择: userfaultfd 与内核缺页处理。 Swap-in 有两种方案: (a) 在 SMAP 用户空间注册 userfaultfd 处理器来拦截缺页并发起读取; 或 (b) 通过 SMAP 内核模块挂钩到内核的缺页路径。 推荐方案 (b) 以获得更低延迟 — 避免每次缺页的用户态-内核态往返。 SMAP 模块已有字符设备 (/dev/smap_mig_device) 和 kprobe 基础设施来挂钩内核函数。

3.3 后端设备管理

Swap Map (位图分配器)

一个简单的位图跟踪 NVMe 设备上哪些 4 KiB (或用于大页的 2 MiB) 槽位正在使用:

struct smap_swap_device {
    struct block_device *bdev;      /* NVMe 块设备 */
    unsigned long *bitmap;           /* 槽位分配位图 */
    unsigned long nr_slots;          /* 总槽位数 */
    atomic_long_t nr_used;           /* 已分配计数 */
    spinlock_t lock;                 /* 保护位图 */
    sector_t start_sector;           /* 分区起始扇区 */
    unsigned int slot_size;          /* PAGE_SIZE 或 HPAGE_SIZE */
};

/* 槽位分配: 查找第一个零位 */
unsigned long smap_swap_map_alloc(struct smap_swap_device *sdev) {
    unsigned long slot;
    spin_lock(&sdev->lock);
    slot = find_first_zero_bit(sdev->bitmap, sdev->nr_slots);
    if (slot < sdev->nr_slots) {
        set_bit(slot, sdev->bitmap);
        atomic_long_inc(&sdev->nr_used);
    }
    spin_unlock(&sdev->lock);
    return slot;
}

设备配置

NVMe 后端设备通过新的 SMAP 配置项进行配置:

# /opt/ubturbo/conf/smap/swap.config
swap_device=/dev/nvme0n1p2
swap_size_mb=65536          # 64 GB swap 分区
slot_size=4096              # 4K 槽位 (或 2097152 用于 2M 大页槽位)
l2_swap_watermark=80        # L2 使用超过 80% 时开始换出
cold_window_threshold=3     # 页面 freq=0 须持续 3 次连续扫描

enable_compressed_swap=0    # 可选: 写入 NVMe 前先压缩

6. 运行流程

7. 与内核 Swap 子系统的集成

方案 A: 自定义 Swap (推荐)

SMAP 管理自己的 swap map 和块 I/O, 完全绕过内核的 swap_info_struct / swap_cache。这可以完全控制:

  • 槽位分配策略 (顺序 vs. 分散, 用于 NVMe 磨损均衡)
  • 压缩 swap (可选 LZ4/zstd 压缩后再写入)
  • 每进程 swap 限制 (内核 swap 不提供此功能)
  • 大页感知的槽位大小 (VM 工作负载使用 2 MiB 槽位)

代价: 必须小心处理 PTE 操作并挂钩到缺页路径。 SMAP 已经使用 kprobes 进行 migrate_pages() 解析, 因此扩展此机制来挂钩 do_swap_page() 或注册自定义 swap_ops 是可行的。

方案 B: 内核 swapon 集成

替代方案: 通过 swapon /dev/nvme0n1p2 将 NVMe 设备配置为标准 Linux swap 分区, 让 SMAP 对冷页触发 madvise(MADV_PAGEOUT), 通过内核的标准 swap 路径推送它们。

方面方案 A: 自定义 Swap方案 B: 内核 swapon
实现工作量高 (bio, PTE, 缺页钩子)低 (madvise + swapon)
每进程记账完全控制有限 (仅 cgroup)
大页支持原生 2M 槽位swap 时拆分为 4K
压缩 swap自定义 LZ4/zstd使用 zswap 作为前端
Swap-in 延迟优化 (直接 bio)标准 (经过 VFS)
维护负担须跟踪内核 ABI 变更内核维护
建议: 先用方案 B (内核 swapon + madvise) 做快速原型, 然后评估延迟和大页限制是否足以证明迁移到方案 A 的必要性。 方案 B 可以完全在 SMAP 用户空间实现, 无需修改内核模块。

方案 B 实现草案

/* 在 SwapOutStrategy() 中 — 仅用户空间, 无内核变更 */
static int SwapOutViaMadvise(ProcessAttr *process, ActcData *coldPages,
                             uint64_t count)
{
    for (uint64_t i = 0; i < count; i++) {
        void *vaddr = phys_to_virt_for_pid(process->pid, coldPages[i].addr);
        /* MADV_PAGEOUT 告知内核立即回收该页面 */
        process_madvise(pidfd, &iov, 1, MADV_PAGEOUT, 0);
    }
    process->swapAccounting.nr_swapped_out += count;
}

这使用了 process_madvise(2) (Linux 5.10+), 它允许一个进程对另一个进程的内存提出建议 — 这正是 SMAP 用户空间所需要的。结合对 NVMe 分区的标准 swapon, 内核会自动处理所有 I/O、PTE 操作和 swap-in 缺页处理。

8. 设计权衡与替代方案

8.1 何时换出 vs. 何时压缩

SMAP 可以选择在写入 NVMe 之前压缩页面 (使用我们配套文档中记录的 zram/zswap 压缩基础设施)。这会增加 ~1–10 μs 的 CPU 时间, 但可以将有效 NVMe 容量翻倍。

方式延迟 (写入)NVMe 容量CPU 开销
直接 swap (未压缩)~10–50 μs1×极小
压缩 swap (LZ4)~15–55 μs~2–3×适中
压缩 swap (zstd)~30–100 μs~3–4×较高

8.2 LRU 倒置风险

一个已知问题 (由 Chris Down / Meta 在 zswap/zram 中记录): 如果 SMAP 将一个页面换出到 NVMe, 然后它被缺页取回到 L1, 但一个"更温"的页面仍停留在 L2, 内存层次就发生了倒置。 我们的设计通过以下方式缓解此问题:

  • 仅换出在多个扫描窗口中持续零频率的页面
  • 仅从 L2 换出 (永不直接从 L1 到 L3)
  • 缺页时直接换入到 L1 (该页被访问, 说明它现在是热的)
  • 跟踪每进程的 swap_in_faults — 如果 swap-in 频率过高, 则降低换出激进程度

8.3 大页处理

SMAP 主要针对 VM 工作负载处理 2M 大页。换出 2M 页面有不同的权衡:

  • 方案 A (自定义): 2M NVMe 槽位 — 如果页面大部分为零则浪费空间, 但避免 THP 拆分
  • 方案 B (内核): 内核在 swap 前将 THP 拆分为 4K — swap-in 后丢失大页
  • 混合方案: 压缩 2M 页面; 如果压缩比 > 50%, 压缩后写入 NVMe; 否则拆分为 4K

8.4 磨损均衡

顺序槽位分配会在 NVMe 上造成热点。缓解措施:

  • 每个分配纪元使用随机起始偏移
  • 依赖 NVMe FTL 的内部磨损均衡 (现代驱动器处理得很好)
  • 监控 /sys/block/nvmeXnY/device/wear_leveling_count, 接近寿命终点时告警
