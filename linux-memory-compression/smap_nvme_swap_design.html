1. Background & Problem Statement

1.1 SMAP Today: Memory-Tier-Only Migration

SMAP (Smart Memory Adaptive Platform) in ubturbo manages a two-level memory hierarchy:

LevelMediumLatencyCapacitySMAP Role
L1 Local NUMADRAM (DDR5)~80–120 nsBase capacityHot pages reside here
L2 Remote NUMACXL / PMEM / UB interconnect~200–500 nsExtended poolCold pages demoted here

Migration is performed by the kernel’s migrate_pages() syscall, wrapped via kprobes in plugins/smap/src/tiering/smap_migrate_wrapper.c. The strategy engine in plugins/smap/src/user/strategy/separate_strategy.c sorts pages by access frequency (hardware ACTC counters or software AF-bit scanning) and selects cold local pages for demotion and hot remote pages for promotion.

The three migration directions defined in manage.h:95-99:

typedef enum {
    DEMOTE,    // Cold L1 → L2
    PROMOTE,   // Hot L2 → L1
    SWAP,      // Bidirectional exchange (cold L1 ↔ hot L2)
} MigrateDirection;

1.2 The Gap: No Path to Persistent Storage

When both L1 and L2 memory are under pressure, SMAP has no escape valve. The ubturbo_smap_urgent_migrate_out() function only demotes within NUMA — it cannot spill to a block device. If remote NUMA is also full, the system hits OOM.

Current limitation: SMAP’s MigrateDirection enum has no SWAP_OUT to disk. The oom_migrate.h:FindPidMigrateSize() only finds candidates for NUMA demotion, not for disk swap. The entire kernel module (smap_migrate_pages.c) works exclusively with migrate_pages() — it never issues block I/O.

1.3 Why NVMe Swap Matters

  • Memory overcommit: VM density scenarios where aggregate VM memory exceeds physical DRAM + CXL capacity
  • Cold tail elimination: Pages with zero access frequency across multiple scan windows are occupying expensive memory
  • Cost efficiency: NVMe is ~10× cheaper per GB than CXL memory, ~50× cheaper than DRAM
  • Graceful degradation: Instead of OOM-killing VMs, spill coldest pages to NVMe at ~10–100 μs latency

The proposed three-tier hierarchy with latency profile:

TierMediumLatencySMAP Action
L1Local DRAM~100 nsKeep hot pages
L2Remote NUMA (CXL)~300 nsDemote warm pages
L3NVMe SSD~10–100 μsSwap out coldest pages (NEW)

3. Detailed Design

3.1 Strategy Layer: Three-Tier Decision Engine

The current strategy in separate_strategy.c makes a binary decision: demote or promote between L1 and L2. We extend this to a three-tier decision:

Decision Criteria for SWAP_OUT

A page in L2 (remote NUMA) qualifies for swap-out when:

  1. Zero frequency across cold_window_threshold consecutive scan windows (default: 3)
  2. Not white-listed (pinned, reserved, or kernel pages excluded)
  3. L2 utilization exceeds l2_swap_watermark (default: 80%)
  4. NVMe swap space available (checked via swap map bitmap)

This preserves SMAP’s existing L1↔L2 behavior and only adds the L3 tier as a further backstop. Pages flow strictly downhill: L1 → L2 → L3, with promotion going L3 → L1 (skipping L2 to avoid extra migration latency on fault).

3.2 Kernel Module: swap_out / swap_in Paths

3.2.1 Swap-Out Path

/* New function in smap_swap.c */
int smap_swap_out(struct folio **folios, unsigned int nr_folios,
                   struct smap_swap_device *sdev)
{
    for (int i = 0; i < nr_folios; i++) {
        /* 1. Allocate slot in swap map */
        slot = smap_swap_map_alloc(sdev);

        /* 2. Lock folio, unmap PTEs, replace with swap entry */
        folio_lock(folios[i]);
        try_to_unmap(folios[i], TTU_SWAP_OUT);
        set_smap_swap_pte(folios[i], sdev, slot);

        /* 3. Submit async write bio to NVMe */
        bio = bio_alloc(sdev->bdev, 1, REQ_OP_WRITE, GFP_KERNEL);
        bio->bi_iter.bi_sector = slot_to_sector(slot);
        bio_add_folio(bio, folios[i], folio_size(folios[i]), 0);
        submit_bio(bio);

        /* 4. On completion callback: free the folio */
    }
}

3.2.2 Swap-In Path

/* Triggered by page fault on swap PTE */
int smap_swap_in(struct vm_fault *vmf,
                  struct smap_swap_device *sdev, unsigned long slot)
{
    /* 1. Allocate new folio on L1 (local DRAM) */
    folio = folio_alloc(GFP_HIGHUSER_MOVABLE | __GFP_THISNODE, 0);

    /* 2. Submit async read bio from NVMe */
    bio = bio_alloc(sdev->bdev, 1, REQ_OP_READ, GFP_KERNEL);
    bio->bi_iter.bi_sector = slot_to_sector(slot);
    bio_add_folio(bio, folio, folio_size(folio), 0);
    submit_bio_wait(bio); /* sync for fault path */

    /* 3. Install new PTE pointing to L1 folio */
    set_pte_at(vmf->vma->vm_mm, vmf->address,
               vmf->pte, mk_pte(&folio->page, vmf->vma->vm_page_prot));

    /* 4. Free swap slot */
    smap_swap_map_free(sdev, slot);
}
Design choice: userfaultfd vs kernel page fault. Two approaches for swap-in: (a) register a userfaultfd handler in SMAP user-space that intercepts faults and issues reads, or (b) hook into the kernel’s fault path via the SMAP kernel module. We recommend (b) for lower latency — avoiding user-kernel round-trip on every fault. The SMAP module already has a character device (/dev/smap_mig_device) and kprobe infrastructure to hook kernel functions.

3.3 Backing Device Management

Swap Map (Bitmap Allocator)

A simple bitmap tracks which 4 KiB (or 2 MiB for hugepage) slots on the NVMe device are in use:

struct smap_swap_device {
    struct block_device *bdev;      /* NVMe block device */
    unsigned long *bitmap;           /* slot allocation bitmap */
    unsigned long nr_slots;          /* total slots */
    atomic_long_t nr_used;           /* allocated count */
    spinlock_t lock;                 /* protects bitmap */
    sector_t start_sector;           /* partition start */
    unsigned int slot_size;          /* PAGE_SIZE or HPAGE_SIZE */
};

/* Slot allocation: find first zero bit */
unsigned long smap_swap_map_alloc(struct smap_swap_device *sdev) {
    unsigned long slot;
    spin_lock(&sdev->lock);
    slot = find_first_zero_bit(sdev->bitmap, sdev->nr_slots);
    if (slot < sdev->nr_slots) {
        set_bit(slot, sdev->bitmap);
        atomic_long_inc(&sdev->nr_used);
    }
    spin_unlock(&sdev->lock);
    return slot;
}

Device Configuration

The NVMe backing device is configured via a new SMAP config entry:

# /opt/ubturbo/conf/smap/swap.config
swap_device=/dev/nvme0n1p2
swap_size_mb=65536          # 64 GB swap partition
slot_size=4096              # 4K slots (or 2097152 for 2M hugepage slots)
l2_swap_watermark=80        # start swapping when L2 > 80% used
cold_window_threshold=3     # pages must be freq=0 for 3 consecutive scans
max_swap_out_per_cycle=1024 # max pages to swap per scan period
enable_compressed_swap=0    # optional: compress before writing to NVMe

6. Operational Flow

7. Integration with Kernel Swap Subsystem

Option A: Custom Swap (Recommended)

SMAP manages its own swap map and block I/O, bypassing the kernel’s swap_info_struct / swap_cache entirely. This gives full control over:

  • Slot allocation strategy (sequential vs. scattered for NVMe wear leveling)
  • Compressed swap (optional LZ4/zstd compression before write)
  • Per-process swap limits (not available in kernel swap)
  • Hugepage-aware slot sizing (2 MiB slots for VM workloads)

Trade-off: must handle PTE manipulation carefully and hook into the fault path. SMAP already uses kprobes for migrate_pages() resolution, so extending this to hook do_swap_page() or register a custom swap_ops is feasible.

Option B: Kernel swapon Integration

Alternative: configure the NVMe device as a standard Linux swap partition via swapon /dev/nvme0n1p2 and let SMAP trigger madvise(MADV_PAGEOUT) on cold pages to push them through the kernel’s standard swap path.

AspectOption A: Custom SwapOption B: Kernel swapon
Implementation effortHigh (bio, PTE, fault hook)Low (madvise + swapon)
Per-process accountingFull controlLimited (cgroup only)
Hugepage supportNative 2M slotsSplit to 4K on swap
Compressed swapCustom LZ4/zstdUse zswap as frontend
Swap-in latencyOptimized (direct bio)Standard (through VFS)
Maintenance burdenMust track kernel ABI changesKernel-maintained
Recommendation: Start with Option B (kernel swapon + madvise) for a quick prototype, then evaluate whether the latency and hugepage limitations justify moving to Option A. Option B can be implemented entirely in SMAP user-space with no kernel module changes.

Option B Implementation Sketch

/* In SwapOutStrategy() — user-space only, no kernel changes */
static int SwapOutViaMadvise(ProcessAttr *process, ActcData *coldPages,
                             uint64_t count)
{
    for (uint64_t i = 0; i < count; i++) {
        void *vaddr = phys_to_virt_for_pid(process->pid, coldPages[i].addr);
        /* MADV_PAGEOUT tells kernel to reclaim this page immediately */
        process_madvise(pidfd, &iov, 1, MADV_PAGEOUT, 0);
    }
    process->swapAccounting.nr_swapped_out += count;
}

This uses process_madvise(2) (Linux 5.10+) which allows one process to advise another’s memory — exactly what SMAP user-space needs. Combined with a standard swapon on the NVMe partition, the kernel handles all the I/O, PTE manipulation, and swap-in fault handling automatically.

8. Design Tradeoffs & Alternatives

8.1 When to Swap vs. When to Compress

SMAP could optionally compress pages before writing to NVMe (using the zram/zswap compression infrastructure documented in our companion document). This adds ~1–10 μs CPU time but can double effective NVMe capacity.

ApproachLatency (write)NVMe CapacityCPU Cost
Direct swap (uncompressed)~10–50 μs1×Minimal
Compressed swap (LZ4)~15–55 μs~2–3×Moderate
Compressed swap (zstd)~30–100 μs~3–4×High

8.2 LRU Inversion Risk

A known problem (documented by Chris Down / Meta for zswap): if SMAP swaps a page to NVMe and it gets faulted back to L1, but a “warmer” page is still sitting in L2, the memory hierarchy is inverted. Our design mitigates this by:

  • Only swapping pages with sustained zero frequency across multiple scan windows
  • Swapping from L2 only (never directly from L1 to L3)
  • Faulting directly to L1 (the page was accessed, so it’s now hot)
  • Tracking swap_in_faults per process — if swap-in rate is high, reduce swap-out aggressiveness

8.3 Hugepage Handling

SMAP primarily works with 2M hugepages for VM workloads. Swapping 2M pages has different tradeoffs:

  • Option A (custom): 2M NVMe slots — wastes space if page is mostly zero, but avoids THP splitting
  • Option B (kernel): Kernel splits THP to 4K before swapping — loses hugepage on swap-in
  • Hybrid: Compress the 2M page; if compression ratio > 50%, write compressed to NVMe; else split to 4K

8.4 Wear Leveling

Sequential slot allocation causes hot spots on NVMe. Mitigations:

  • Randomized starting offset per allocation epoch
  • Rely on NVMe FTL’s internal wear leveling (modern drives handle this well)
  • Monitor /sys/block/nvmeXnY/device/wear_leveling_count and alert if approaching end-of-life
