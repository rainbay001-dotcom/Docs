SMAP Architecture Deep Dive

Smart Memory Adaptive Platform — Huawei ubturbo Memory Tiering Subsystem

SMAP is the core memory tiering plugin within Huawei's ubturbo framework, managing page migration between local DRAM (L1) and remote CXL/PMEM memory (L2) NUMA nodes. It uses hardware ACTC counters and software AF-bit page table scanning to detect page hotness, applies frequency-based strategies to classify hot/cold pages, and performs multi-threaded kernel-level page migration via migrate_pages().

1. System Overview & Architecture

SMAP operates across three layers: a public SDK (C API with IPC), a user-space daemon (strategy + process management), and kernel modules (access tracking drivers + tiering migration engine). The ubturbo framework provides IPC transport, lifecycle management, and logging infrastructure.

The system manages up to 4 local NUMA nodes (L1, IDs 0–3) and 18 remote NUMA nodes (L2, IDs 4–21), supporting up to 300 processes (4K mode) or 100 VMs (2M mode) simultaneously.

2. IPC & SDK Layer

SMAP exposes a C SDK with 17 public API functions. The SDK communicates with the SMAP daemon via ubturbo's IPC framework. On the server side, TurboModuleSmap loads libsmap.so at startup via dlopen() and registers 19 IPC service handlers.

3. Process Management

The ProcessManager is a global singleton managing all tracked processes. Each process is represented by a ProcessAttribute struct (30+ fields) in a singly-linked list. The manager tracks device file descriptors, NUMA topology, thread contexts, and remote NUMA capacity accounting.

Process Lifecycle

  1. Add: ProcessAddManage() — validates PID, detects VM via IsQemuTask() (checks /proc/[pid]/comm), sets L1/L2 NUMA nodes, allocates ProcessAttr, appends to linked list
  2. Track: Registers process with kernel access tracking driver via ioctl, starts AF-bit scanning
  3. Migrate: Periodic ScanMigrateWork() runs strategy, builds migration message, issues ioctl
  4. Remove: RemoveManagedProcess() — stops scanning, frees actcData buffers, unlinks from list
  5. Invalidation: CheckAndRemoveInvalidProcess() — removes processes whose PIDs no longer exist

4. Access Tracking & Hotness Detection

SMAP uses two complementary mechanisms to detect page access frequency:

  • Software AF-bit scanning: Walks page tables, reads and clears the hardware Accessed Flag (AF) bit on PTEs, increments per-page frequency counters
  • Hardware ACTC counters: Reads Access Counter Table (ACTC) values from ACPI SRAT-derived memory regions (Huawei-specific hardware)
submit_one_work(ap) schedules delayed_work with ap->scan_time ms delay (50ms unit, range 5–200ms) 2. work_func() runs in workqueue context: walks pages, reads AF bits, updates freq counters 3. Reschedules itself after completion — continuous scanning loop until process removed or SMAP stopped

AP State Machine

The access_pid_struct tracks a global state bitmask controlling what operation is in progress:

5. Strategy Engine

The strategy engine decides which pages to migrate and in which direction. It takes the per-page frequency data from the access tracking layer, sorts pages by frequency, and uses a binary search algorithm to find the optimal hot/cold threshold.

6. Scene Classification

The scene classifier analyzes workload behavior over a sliding window of PageInfo snapshots to adaptively tune scanning intervals and migration aggressiveness. It maintains a PAGE_INFO_DEPTH=8 ring buffer of historical page statistics.

7. Kernel Tiering Engine

The kernel tiering engine is a loadable kernel module (smap_tiering.ko) that provides the actual page migration capability. It creates a character device (/dev/smap_mig_device), receives migration commands via ioctl, and performs multi-threaded page migration using the kernel's migrate_pages() API.

8. End-to-End Migration Flow

This section traces the complete path from periodic timer firing to physical page movement.

OOM Emergency Migration

When local memory is exhausted, ubturbo_smap_urgent_migrate_out(size) triggers an emergency path:

  1. FindPidMigrateSize() identifies processes with the most remote-eligible pages
  2. Bypasses normal strategy — migrates all possible cold pages immediately
  3. Process state set to PROC_MOVE during emergency migration
  4. Does not wait for frequency data — uses last known scan results

Config Binary Layout (/dev/shm/smap_config)

SmapConfigHeader (8 bytes):
| ver (1B) | runMode (1B) | headerLen (2B) | totalLen (4B) |

NUMA Payload Section:
| numaCount (4B) | NumaPayload[numaCount] |
  NumaPayload (12B each): int16 local, int16 remote, uint32 size_MB

Process Payload Section:
| procCount (4B) | ProcessPayload[procCount] |
  ProcessPayload: pid, scanType, type, state, migrateMode,
                  numaNodes, scanTime, duration,
                  count, migrate_params[]

Recovery Sequence

  1. RecoverFromConfig() reads /dev/shm/smap_config at startup
  2. Restores run mode (WATERLINE or MEM_POOL)
  3. Restores NUMA capacity allocations
  4. Re-adds tracked processes with their migration parameters
  5. Validated against config version SMAP_CONFIG_VER=1

Initialization Sequence

smap_process_symbols() — kprobe lookup for migrate_pages, isolate_folio_to_list, putback_movable_pages 2. init_migrate() — Register char device /dev/smap_mig_device, class, device node 3. Access tracking driver init — Create per-node /dev/smap_nodeN devices, setup workqueues ProcessManager Initialization (ProcessManagerInit) 1. Detect page size (4K/64K) via sysconf → set g_pageSizeNormal, g_pageSizeHuge=2M 2. Open kernel device FDs: /dev/smap_mig_device, /dev/smap_node[0-21], /dev/smap_access_device 3. RemoteNumaInfoInit() — zero 4x18 privateSize + 18 sharedSize matrices, init lock 4. InitThread(manager, period, ScanMigrateWork) — Start periodic migration thread
