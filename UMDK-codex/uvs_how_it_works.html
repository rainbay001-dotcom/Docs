UVS: topology and route selection for UnifiedBus

In this UMDK checkout, UVS is the user-visible topology and route-selection layer packaged as libtpsa.so. Applications and higher-level UMDK components use its APIs to create aggregate devices, publish topology, and translate bonding EIDs into physical EID/path choices before URMA or UMQ creates transport resources.

Short Answer

UVS is the part of the UMDK/URMA stack that owns topology visibility and route/path selection for UnifiedBus. It is not the fast datapath and it is not the UDMA hardware driver. It is a control-plane library/API layer that lets software say: "given source and destination bonding EIDs, which concrete primary or port EIDs should I use for RTP, CTP, or UTP?"

The local source names the user-space library target tpsa, installs libtpsa.so, and exposes public headers uvs_api.h and uvs_types.h. The logs identify the component as libuvs.

What UVS Does

  • Maintains a topology model made of nodes, aggregate devices, UEs, primary EIDs, port EIDs, and links.
  • Pushes topology into kernel ubcore and ubagg.
  • Creates and deletes aggregate devices through /dev/ubagg.
  • Answers route queries from bonding EID pairs.
  • Answers path-set queries with concrete source/destination EIDs and port identifiers.

What UVS Does Not Do

  • It does not post work requests or move payload data.
  • It does not directly create TP resources in UDMA.
  • It does not replace liburma; it complements URMA by choosing paths before URMA setup.
  • It is not the same thing as kernel ubcore, although it uses ubcore ioctls.

Topology Model

UVS topology is built around struct urma_topo_node in user space and the matching struct ubcore_topo_node in the kernel. A node records:

  • type: topology type. The code handles FULLMESH_1D and CLOS.
  • super_node_id and node_id: logical placement of this host/node.
  • is_current: marks the local node in the topology map.
  • links[IODIE_NUM][PORT_NUM]: peer node, peer die, and peer port for direct links.
  • agg_devs[DEV_NUM]: aggregate devices indexed by device id.

Each aggregate device contains an aggregate EID plus per-die UE records. Each UE has one primary_eid and an array of port_eid values. UVS translates aggregate or bonding EIDs into these physical EIDs when callers ask for routes or paths.

Command Flow

Setting topology

  1. Management software calls uvs_set_topo_info(topo_buf, node_size, node_num).
  2. tpsa_api.c checks that node_size == sizeof(struct urma_topo_node).
  3. UVS writes the topology to /dev/ubagg with uvs_ubagg_ioctl_set_topo().
  4. UVS writes the same topology to /dev/ubcore/ubcore with uvs_ubcore_ioctl_set_topo().
  5. Kernel ubcore_cmd_set_topo() creates or updates the global ubcore_topo_map.
  6. Kernel code calls ubcore_create_jetty_rsrc() for primary EIDs on the current node.
  7. Kernel code calls ubmgr_notify_set_topo() so the manager side can react to the new topology.

Querying a path set

  1. A caller such as UMQ calls uvs_get_path_set(src_bonding_eid, dst_bonding_eid, tp_type, multi_path, out).
  2. libtpsa.so validates the EIDs and TP type, then opens /dev/ubcore/ubcore.
  3. It sends UVS_CMD_GET_TOPO_PATH_EID as a TLV payload.
  4. Kernel ubcore_cmd_get_path_set() calls ubcore_get_path_set().
  5. ubcore_get_path_set() finds source and destination aggregate devices in the global topology map.
  6. The kernel fills ubcore_path_set with selected physical EIDs and port IDs.
  7. User space receives uvs_path_set_t. For FULLMESH_1D, user space currently filters the returned paths based on multi_path.

Route and Path Semantics

The naming is easy to confuse. uvs_get_route_list() returns route EIDs in a compact list. uvs_get_path_set() returns richer path records: topology type, source/destination node IDs, chip/die counts, source and destination port identifiers, and physical source/destination EIDs for each path.

Query Input Output Typical use
uvs_get_route_list() Source/destination bonding EIDs and route flags. uvs_route_list_t containing source/destination EID pairs. Find EIDs for TP setup or diagnostics.
uvs_get_path_set() Source/destination bonding EIDs, TP type, and multi_path. uvs_path_set_t with port IDs, node IDs, topo type, and EID pairs. UMQ route selection and topology-aware transport setup.

Topology-specific behavior

  • For FULLMESH_1D, the kernel first adds primary paths, then port paths. User-space UVS filters the result for multi_path because the kernel currently returns all EIDs.
  • For CLOS, multi_path == true and RTP currently returns an empty path set.
  • For CLOS, multi_path == true and CTP uses primary EID path logic.
  • For non-multipath CTP or RTP, the kernel uses port path logic.

Relationship to URMA, UMQ, and TP Creation

UVS sits before the normal URMA link setup path. It answers "which EIDs should this connection use?" URMA and UDMA then use those EIDs to create and activate transport resources. 

topology provider / management system
  -> uvs_set_topo_info()
  -> kernel ubcore global topo map
  -> uvs_get_path_set() or uvs_get_route_list()
  -> selected local/peer EIDs
  -> URMA/UMQ connection setup
  -> UDMA get_tp_list / active_tp
  -> normal UB datapath

UMQ uses this relationship directly. The UMQ UB implementation dynamically loads libtpsa.so, resolves uvs_get_path_set, converts UMQ TP types to UVS TP types, then copies the returned path set into UMQ route records.

Why this matters for first-link latency: if the topology is already known, UVS can provide peer/local EID pairs before the first transport is created. That makes it a natural source of warmup candidates for a TP-list prefetch design. If topology has not been pushed yet, UVS route/path queries either fail or return no useful path, so a warmup pass based on UVS will silently have nothing meaningful to prefetch unless the caller logs that condition.

Operational Notes

  • The CMake target is tpsa, installed as /usr/lib64/libtpsa.so.
  • The public headers are installed under /usr/include/ub/umdk/urma.
  • UVS logs use the URMA|libuvs tag.
  • The rsyslog config routes URMA|libuvs messages to /var/log/umdk/tpsa/%programname%.log.
  • The environment variable UVS_LOG_LEVEL controls library log verbosity.
  • urma_admin still shows legacy EID commands marked "only for uvs"; these are not the same as the normal control-plane path.

Failure Modes to Check

Symptom Likely cause Where to check
uvs_get_route_list() or uvs_get_path_set() fails. Global topology map has not been set or bonding EID is not present. ubcore_get_global_topo_map(), ubcore_get_topo_info_by_agg_eid().
Warmup code based on UVS schedules zero routes. uvs_set_topo_info() has not run yet, or topology was pushed after driver probe. UVS logs, kernel ubcore_cmd_set_topo(), route/path return counts.
Route exists but TP creation is still slow. UVS only selected EIDs; first get_tp_list still performs control-queue TP allocation. UDMA get_tp_list and active_tp timing logs.
FULLMESH_1D returns unexpected extra paths. The kernel returns all EIDs and user-space UVS applies a temporary filter based on multi_path. uvs_filter_path_set() in tpsa_api.c.

How to Read the Code

  1. Start with uvs_api.h to understand public objects: uvs_route_t, uvs_route_list_t, uvs_path_set_t, and urma_topo_node.
  2. Read tpsa_api.c to see API validation and the call into ioctl helpers.
  3. Read uvs_cmd_tlv.c to see how each command maps to a TLV payload.
  4. Read uvs_ubagg_ioctl.c to see which device node each operation uses.
  5. Read kernel ubcore_uvs_cmd.c to see command dispatch and topology creation.
  6. Read kernel ubcore_topo_info.c to understand route/path selection behavior.
  7. Read UMQ umq_symbol_private.c and umq_ub_impl.c for a real consumer of uvs_get_path_set().
