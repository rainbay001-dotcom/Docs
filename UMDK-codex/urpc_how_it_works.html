UMDK / URPC Framework

URPC Architecture and Runtime Workflow

URPC is the UMDK user-space remote-procedure-call framework. It gives an application a channel, queue, function, memory-segment, poll, and callback model, while hiding the concrete UB transport behind provider and queue operation tables.

In the checked source tree, the framework control plane is IP based for server listen and attach, while the data plane maps URPC queues to URMA Jetty resources over UB. This document focuses on the framework path under src/urpc/framework; UMQ is a separate queue subsystem in the same URPC tree and is covered by the companion UMQ HTML.

Overall Architecture

What URPC Is

URPC is not a serialization framework like gRPC plus protobuf. The caller prepares request and response memory in urpc_sge_t arrays, places a URPC protocol header in the first SGE, and chooses how completion should be observed. The framework tracks channel state, queue state, request IDs, server function IDs, remote queue imports, memory segment tokens, and transport completion events.

Control Plane

Creates sessions, exchanges server and queue metadata, attaches or detaches servers, and drives queue add, remove, pair, and unpair workflows through task objects.

Data Plane

Formats REQ/RSP protocol headers, selects local and remote queues, posts Jetty send/read work, polls completions, and reports events back to the application.

Provider Layer

Maps URPC generic queues to concrete transport resources. In this checkout, the main framework backend is Jetty over URMA UB.

Initialization

urpc_init() is the framework entry point. The implementation in cp.c validates the config, ignores SIGPIPE, prepares async events and thread context, initializes optional DFX Unix control sockets, initializes resource-release and queue-ID allocators, then calls provider_init() for the configured transports.

Validate role, feature flags, and transport info. Features include timeout, keepalive, multi-EID, hardware UB offload, function-info exchange, multiplexing, and stats behavior.
Initialize process services. Async event context, thread context, DFX, resource release, queue ID allocator, and manager modules are set up.
Initialize providers. provider_init() walks configured trans_info[] entries and calls provider vtables registered by constructors.
Initialize role-specific state. Depending on role, server, client, or server-client state is initialized, then URPC state becomes URPC_STATE_INIT.

For the Jetty provider, provider initialization selects an URMA device by device name, IP, or EID. In multi-EID mode, the provider can enable all EIDs for a device name. The provider then opens the URMA context, records device attributes, and prepares memory registration or import state for UB transport.

Server Startup and Attach

Server startup and client attach are control-plane operations. urpc_server_start() currently accepts IPv4 or IPv6 listen configuration. SERVER_TYPE_UB is present in the type system, but the listen path in this checkout returns unsupported for UB server listen config.

urpc_channel_server_attach(), urpc_channel_server_refresh(), queue add/remove, and queue pair/unpair all construct a callback context, create a task through task_manager_client_task_create(), then either wait for its semaphore or return a task ID for nonblocking mode.

Channel and Queue Setup

A channel is the RPC session object. It does not send by itself; it owns queue references and request tracking. The actual data path goes through a local queue and an imported or materialized remote queue.

Create a channel. urpc_channel_create() allocates urpc_channel_info_t, assigns an ID, initializes list heads, locks, server-node state, request-entry table state, and provider binding.
Create local queues. urpc_queue_create(QUEUE_TRANS_MODE_JETTY, cfg) allocates a queue ID, calls ops->create_local_queue(), initializes per-queue slabs, and optionally hooks timeout notification state.
Attach to a server. The client exchanges channel, queue, and function metadata over the IP control plane. If function-info exchange is enabled, server function name-to-ID mappings are copied into the client channel table.
Add queues to the channel. urpc_channel_queue_add() adds local queues by handle or remote queues by queue ID through control-plane tasks.
Pair queues. urpc_channel_queue_pair() validates local/remote roles and status, then runs a pair task. In the Jetty backend, pairing eventually calls urma_bind_jetty() to advise the local Jetty of the remote target Jetty.

The channel can hold multiple local and remote queues. If an application does not pin a queue in urpc_call_option_t, the framework can choose from the channel queue lists.

Jetty Transport Mapping

The framework transport abstraction is deliberately thin. The queue code registers g_urpc_send_recv_ops with mode = QUEUE_TRANS_MODE_JETTY. That vtable points to Jetty implementations for local queue creation, remote queue creation and import, send, read, receive post, completion poll, interrupt wait, queue bind, and queue status modify.

Function IDs and Dispatch

Server-side functions are registered by name with urpc_func_register(). The implementation initializes a function module, allocates a private method ID, combines it with the configured device class and subclass, and inserts the handler into both ID and name hash maps. 

function id layout:

device_class: 12 bits
sub_class:    12 bits
private bit:   1 bit
method:       23 bits

server:
  urpc_func_register(name, handler) -> full func_id

client:
  attach or refresh -> function info exchange
  urpc_func_id_get(channel, name) -> server func_id copied into channel table

On the server, urpc_func_exec() looks up synchronous handlers by method ID and invokes the registered function. urpc_func_async_exec() does the same for async handlers and passes the request context plus queue handle to the application, so the application can call urpc_func_return() later.

Data-Plane Call Flow

Client calls urpc_func_call(). The function validates SGEs, gets the channel, selects a local queue and remote queue, allocates a tx_ctx_t, and, unless early-response mode is used, allocates a req_entry_t.
URPC fills the request header. urpc_req_fill_basic_info() and urpc_req_fill_req_info_without_dma() write protocol version, type, function ID, request total size, request ID, client channel, and function_defined.
URPC posts transport send. The local queue vtable converts the SGEs to URMA SGEs and posts URMA_OPC_SEND to the imported target Jetty.
Server polls. urpc_func_poll() drains notify events, then calls queue->ops->poll(). RX completions are decoded by process_rx_msg().
Server reports the request. For URPC_MSG_REQ, the server creates req_ctx_t, creates a quick-reply remote queue from source queue info, and reports POLL_EVENT_REQ_RECVED.
Application executes. The server can call urpc_func_exec() for synchronous handlers, or the async handler can keep req_ctx_t and return later.
Server responds. urpc_func_return() fills the response header, optionally encrypts, then posts another queue send to the client's remote queue.
Client polls response. Client-side poll sees URPC_MSG_RSP or URPC_MSG_ACK_AND_RSP, parses request ID, looks up req_entry_t, reports the completion event or releases the waiting urpc_func_poll_wait() caller, and frees the request entry.

Protocol Headers

Header Used For Key Fields

urpc_keepalive_head_t Keepalive manager messages. Keepalive version, response bit, status, local queue ID, server manager channel, reserved/security bytes.
urpc_ctl_head_t Control-plane task messages. Version, opcode, error code, encryption/keepalive flags, channel, data size, control opcode, task ID.
urpc_security_exthdr_t Data-plane encryption extension header. AES IV and authentication tag fields.

Polling and Event Model

URPC is poll-driven. The application can poll one queue by passing a queue handle in urpc_poll_option_t, or poll every local queue in a channel by setting the queue handle to URPC_INVALID_HANDLE. The framework first reports queued abort or timeout notifications, then polls transport completions, then processes read-cache work.

TX Completions

TX completion records come from the JFS JFC. The queue backend converts them into TX_SEND queue messages. The datapath then reports send success, send error, response-sent, read-return, or internal-message cleanup depending on tx_ctx_t.

RX Completions

RX completion records come from the JFR JFC. The datapath parses the first SGE as a URPC header, validates protocol version and type, then emits request-received, request-response, combined ACK-response, or error events.

Interrupt mode is available through urpc_queue_interrupt_fd_get() and the Jetty JFCE path. In that mode, a caller can integrate the queue with an event loop by waiting for the underlying interrupt file descriptor, then polling URPC completions.

Memory and Remote Read

URPC requires memory registration for data-plane SGEs. A memory handle stores provider-specific target segment handles, so the Jetty backend can pick the handle matching the queue provider index. The API also exposes memory segment registration, token query, remote-access enable or disable, and remote read through urpc_ref_read().

Local buffer ownership stays with the application. URPC uses SGEs and registered memory handles; it does not own application serialization or payload layout.
Send uses local target segment handles. send_recv_send() converts each non-data-zone SGE to an URMA SGE using mem_h and provider index.
Remote read uses tokens. urpc_ref_read() obtains the real remote queue for the request context, builds source and destination SGEs, looks up the imported target segment by server channel and token, then posts URMA_OPC_READ.

Security

urpc_ssl_config_set() configures TLS-PSK for the control plane after urpc_init(). The security documentation says TLS 1.2 and TLS 1.3 are supported, and enabling SSL can read /dev/random, which may block if the system entropy pool is low.

The data-plane path has extension hooks for encryption and decryption keyed by function_defined. The protocol includes a security extension header with IV and tag fields, and the datapath calls extension operations around request and response processing when SSL or function-defined extensions require it.

DFX, Admin, and Observability

URPC has several operational hooks. Queue stats and error stats are maintained in the queue layer. A Unix-domain DFX server is initialized when unix_domain_file_path is configured. urpc_admin connects to a process-local socket path and can send administrative requests. The framework also exposes async event retrieval, async-event file descriptor access, task cancellation, performance-recorder registration, and log configuration.

How URPC Relates To UMQ

The URPC source tree contains both framework and umq. The framework path described here does not call the public UMQ API directly for normal RPC transport. Instead, it uses queue_ops_t and provider_ops_t under src/urpc/framework/core/queue. The concrete backend found in this checkout is a Jetty send/receive provider implemented under framework/core/queue/jetty.

UMQ is still part of the broader URPC component area. It provides a separate message-queue API and separate UB, IPC, and UBMM transport modes. For this repository snapshot, treat UMQ as a neighboring subsystem, not the direct data-plane object behind urpc_func_call().

Typical API Sequence

server:
  urpc_init(&cfg)
  urpc_func_register(&handler_info, &func_id)
  urpc_server_start(&control_plane_cfg)
  qh = urpc_queue_create(QUEUE_TRANS_MODE_JETTY, &qcfg)
  urpc_queue_rx_post(qh, rx_sges, rx_sge_num)
  loop:
    n = urpc_func_poll(server_chid_or_channel, &poll_opt, msgs, max)
    for each POLL_EVENT_REQ_RECVED:
      urpc_func_exec(func_id, args, args_num, &rsps, &rsps_num)
      urpc_func_return(qh, req_ctx, &return_wr, &return_opt)

client:
  urpc_init(&cfg)
  chid = urpc_channel_create()
  urpc_channel_server_attach(chid, &server_host, &connect_opt)
  qh = urpc_queue_create(QUEUE_TRANS_MODE_JETTY, &qcfg)
  urpc_channel_queue_add(chid, qh, local_attr, &connect_opt)
  remote_qh = urpc_channel_queue_query(chid, &query_attr)
  urpc_channel_queue_add(chid, remote_qh, remote_attr, &connect_opt)
  urpc_channel_queue_pair(chid, qh, remote_qh, &connect_opt)
  req_h = urpc_func_call(chid, &call_wr, &call_opt)
  n = urpc_func_poll(chid, &poll_opt, msgs, max)
  or urpc_func_poll_wait(chid, req_h, &poll_opt, &rsp_msg, timeout)

Practical Notes

  • The first SGE passed to urpc_func_call() must be valid and large enough for urpc_req_head_t.
  • The framework does not define payload serialization. The application must agree on payload layout, endianness, and versioning.
  • ACK-only mode exists in the protocol, but the normal urpc_func_call() implementation logs that ACK call mode is not supported.
  • Control-plane listen configuration currently supports IPv4 and IPv6, not UB listen mode.
  • Queue creation is per transport mode. In this checkout, the framework queue mode to focus on is QUEUE_TRANS_MODE_JETTY.
  • When timeout or keepalive features are enabled, more manager state is created: notify tables, timer pools, manage channels, and keepalive probe tasks.
  • Nonblocking control-plane APIs return task IDs; completions should be consumed through async events.
