UMQ Architecture and Runtime Workflow

This document explains how Unified Message Queue (UMQ) works in the local UMDK source tree. The focus is the actual implementation: the public facade, transport plugin boundary, buffer ownership model, UB/IPC/UBMM backends, control-plane bind flow, and data-plane send/receive paths.

Source: /Volumes/KernelDev/umdk Observed branch: master Observed commit: 2d99e487 Generated: 2026-05-10

Executive Summary

UMQ is a C message-queue and asynchronous I/O layer under URPC. It exposes a single public API but does not implement every transport directly in the top-level library. Instead, libumq.so is a common facade: it validates calls, owns generic handles, initializes common buffer/DFX/flow-control state, and dispatches to transport-specific operation tables.

The important runtime split is:

  • src/urpc/umq/umq_api.c implements lifecycle, bind, buffers, base queue operations, interrupts, async events, device/route APIs, and global dispatch.
  • src/urpc/umq/umq_pro_api.c implements the enhanced post/poll API and completion-event wrappers.
  • src/urpc/umq/umq_ub implements the UB backend over URMA/UVS resources.
  • src/urpc/umq/umq_ipc implements local IPC over POSIX shared memory, message rings, shared qbuf pools, and futex-style notification.
  • src/urpc/umq/umq_ubmm implements UB memory-mode transport by combining OB memory export/import, shared-memory qbuf mechanics, and a UB control/notify path.

Core Runtime Model

UMQ has a facade-plus-plugin model. A public handle is a generic umq_t wrapper. The backend handle is stored inside it as umqh_tp.

Generic handle

umq_t
  mode        -> selected umq_trans_mode_t
  tp_ops      -> base/control operation table
  pro_tp_ops  -> post/poll operation table
  dfx_tp_ops  -> stats/perf operation table
  umqh_tp     -> backend-specific handle

umq_create allocates this object and returns it as a uint64_t. Most public calls cast the handle back to umq_t *, validate the ops table, and call the backend with umq->umqh_tp.

Transport framework table

g_umq_fws[] maps each enabled umq_trans_mode_t to a shared object and symbol names. For example:

  • UMQ_TRANS_MODE_UB -> libumq_ub.so, umq_ub_ops_get, umq_pro_ub_ops_get, umq_ub_dfx_ops_get
  • UMQ_TRANS_MODE_IPC -> libumq_ipc.so, umq_ipc_ops_get, umq_pro_ipc_ops_get, umq_ipc_dfx_ops_get
  • UMQ_TRANS_MODE_UBMM -> libumq_ubmm.so, umq_ubmm_ops_get, umq_pro_ubmm_ops_get, umq_ubmm_dfx_ops_get

Transport ABI Tables

Ops table Representative functions Meaning
umq_ops_t umq_tp_init, umq_tp_create, umq_tp_bind, umq_tp_buf_alloc, umq_tp_enqueue, umq_tp_dequeue, umq_tp_wait_interrupt, umq_tp_get_topo Base lifecycle, control plane, base queue I/O, interrupt, device, route, and memory-pool state operations.
umq_pro_ops_t umq_tp_post, umq_tp_poll, umq_tp_interrupt_fd_get, umq_tp_get_cq_event Enhanced asynchronous API for explicit TX/RX post and completion polling.

Initialization Path

umq_init is the process-wide setup call. It validates config, records which transport modes are enabled, initializes common resources, loads transport libraries, and calls each backend's init function.

1. Validate config Reject duplicate init, null cfg, too many trans_info entries, or oversized headroom.
2. Configure common state Seed token-policy random state, configure qbuf block size, initialize config lock, and copy filtered config.
3. Enable modes Walk cfg->trans_info[] and mark the matching framework entries enabled.
4. Start optional helpers If flow control is enabled, initialize URPC thread context, timing wheel, manage thread, and listen thread.
5. Load backends For each enabled mode, dlopen the shared object, resolve ops symbols, load backend dynamic symbols, set logs, and call umq_tp_init.
Feature model. umq_init_cfg_t.feature selects base API, pro API, token policy, stats, perf, flow control, and remote memory access. The feature flags determine which code paths are meaningful after initialization.

IB, IB_PLUS, UCP Listed in public enum/framework table, but local backend source directories were not present. Treat as mode placeholders unless corresponding backend libraries are installed separately.

Create and Bind

UMQ separates queue creation from peer connection. Creation makes a local queue object and backend resources. Binding imports or maps the peer-side resources described by the peer's bind-info blob. 

App A                                        App B
-----                                        -----
umq_init(cfg)                                umq_init(cfg)
umq_create(option) -> umqh_a                 umq_create(option) -> umqh_b
umq_bind_info_get(umqh_a, local_a)           umq_bind_info_get(umqh_b, local_b)
             out-of-band exchange of local_a and local_b
umq_bind(umqh_a, local_b)                    umq_bind(umqh_b, local_a)
             queues are ready for base enqueue/dequeue or pro post/poll

The out-of-band exchange is intentionally outside UMQ. The application must carry the bind-info bytes over an existing control channel. In UB mode with token policy enabled, bind info carries token material, so the existing UMDK docs explicitly recommend a secure channel such as TLS.

Create Options

umq_create_option_t chooses the transport mode, device selection, queue name, optional RX/TX sizes, RX/TX depths, polling vs interrupt mode, TP mode, TP type, shared receive queue, used ports, priority, and main/sub UMQ flags.

Queue mode

UMQ_MODE_POLLING Callers poll/dequeue directly.

UMQ_MODE_INTERRUPT Backend event objects can be armed, waited on, and acknowledged.

TP mode and type

RCRMUM map to URMA transport modes.

RTPCTPUTP map to URMA/UVS TP types. Route lookup also converts UMQ TP type to UVS TP type.

Buffer Model

UMQ data moves through umq_buf_t. A qbuf is both metadata and a pointer to payload. Multiple qbufs can be chained with qbuf_next to represent fragments or batches.

Field How it is used
qbuf_next Links buffers for fragmented messages and batch operations.
umqh Identifies the owning queue or UMQ_INVALID_HANDLE for global/common pool buffers.
total_data_size Total payload size for the request, stored on the first fragment.
data_size and buf_data Valid payload size and payload pointer for each fragment.
mempool_id, token_id, token_value Internal tracking for pool ownership and transport memory-registration/import behavior.
status and io_direction Completion status and TX/RX direction returned to the application after poll/dequeue.
qbuf_ext Extension bytes. The pro API treats this as umq_buf_pro_t for opcode, flags, immediate data, remote SGE, and user context.

Allocation Decision

umq_buf_alloc(size, count, umqh, option)
  if umqh == UMQ_INVALID_HANDLE:
      allocate from common qbuf pool or huge qbuf pool
  else:
      delegate to backend umq_tp_buf_alloc()
          UB: common qbuf / huge qbuf variants
          IPC: queue-local shared-memory qbuf pool
          UBMM/UBMM_PLUS: shared-memory qbuf pool over exported/imported memory

The common qbuf pool has TLS caches and global pools. The source shows defaults such as a 64-buffer batch count for fetching/returning, a 2048 TLS maximum, dynamic expansion slots, and shrink/expand logic. UB registers common and huge qbuf memory as URMA target segments so those buffers can be used in send/write/read work requests.

Split vs Combine

Mode Layout Why it matters
UMQ_BUF_SPLIT Metadata and payload are separate. Better when payload pages should be managed/aligned separately.
UMQ_BUF_COMBINE Metadata and payload are contiguous. Better for locality and simpler direct payload access through data or buf_data.

Base API Data Path

The base API is the simpler queue model. Applications allocate a qbuf, fill it, enqueue it, notify if the backend needs it, then the peer dequeues and frees the returned qbuf. 

Sender:
  qbuf = umq_buf_alloc(size, 1, umqh, NULL)
  write payload to qbuf->buf_data
  umq_enqueue(umqh, qbuf, &bad_qbuf)
  umq_notify(umqh)             // backend-specific; no-op for plain UB, meaningful for IPC/UBMM

Receiver:
  qbuf = umq_dequeue(umqh)
  read payload from qbuf->buf_data
  umq_buf_free(qbuf)

What Happens Inside UB Base Enqueue

  1. The common facade calls umq_ub_enqueue_impl through umq_tp_enqueue.
  2. The UB backend checks that the queue is bound and polls TX completions opportunistically.
  3. It converts the qbuf chain into URMA SGEs and urma_jfs_wr_t send work requests.
  4. It posts the WR chain with urma_post_jetty_send_wr on the IO Jetty.
  5. It increments TX outstanding counters and packet stats.

What Happens Inside UB Base Dequeue

  1. The common facade calls umq_ub_dequeue_impl through umq_tp_dequeue.
  2. The backend polls the receive JFC with urma_poll_jfc.
  3. It resolves completion user_ctx back to the original qbuf, fills status, io_direction, and actual payload length.
  4. It replenishes RX buffers through umq_ub_fill_rx_buffer for the base API path.
  5. The received qbuf is returned to the application.
Base API RX behavior. UB base mode prefills receive buffers during bind. In contrast, pro API users are expected to post RX buffers explicitly with umq_post(..., UMQ_IO_RX, ...).

Pro API Data Path

The pro API exposes lower-level asynchronous behavior. Applications explicitly post TX or RX qbufs and explicitly poll TX, RX, or both completion queues. 

Receiver:
  rx = umq_buf_alloc(rx_size, rx_depth, umqh, NULL)
  umq_post(umqh, rx, UMQ_IO_RX, &bad_rx)

Sender:
  tx = umq_buf_alloc(tx_size, 1, umqh, NULL)
  ((umq_buf_pro_t *)tx->qbuf_ext)->opcode = UMQ_OPC_SEND
  umq_post(umqh, tx, UMQ_IO_TX, &bad_tx)
  umq_poll(umqh, UMQ_IO_TX, tx_done, max)

Receiver:
  umq_poll(umqh, UMQ_IO_RX, rx_done, max)
  process rx_done[i]
  umq_buf_reset(rx_done[i])
  umq_post(umqh, rx_done[i], UMQ_IO_RX, &bad_rx)

Pro Opcodes

Opcode Meaning Notes
UMQ_OPC_SEND Message send. Consumes a peer receive buffer and returns TX/RX completions.
UMQ_OPC_SEND_IMM Message send with immediate data. Used by user payloads and internal control messages.
UMQ_OPC_WRITE, UMQ_OPC_WRITE_IMM Remote memory write, optionally with immediate data. Requires remote segment/import state to be valid.
UMQ_OPC_READ Remote memory read. Used by the UBMM_PLUS rendezvous path for large/global buffers.
UMQ_OPC_NOP No-op work request. Reserved/utility operation.

Pro Poll Semantics

umq_poll can poll UMQ_IO_TX, UMQ_IO_RX, or UMQ_IO_ALL. The common wrapper requires at least two output slots for UMQ_IO_ALL. UB splits an UMQ_IO_ALL poll by first polling TX up to half of the caller's budget, then RX for the remaining budget.

UB Backend Deep Dive

The UB backend is the most complete transport in this checkout. It turns UMQ queues into URMA objects and uses URMA/UVS symbols loaded at runtime.

UB Context

umq_ub_ctx_t stores the per-device/EID state: lock policy, feature flags, flow-control config, urma_context_t, device attributes, registered target segments, imported remote segment tracking, trans_info, notify address, Jetty lookup table, and RX-consumed counters.

UB Queue

ub_queue_t is the backend queue object returned as umqh_tp. It owns or references:

  • IO and flow-control urma_jetty_t * entries.
  • Receive contexts: jfr_ctx_t, including JFR, JFR JFC, optional JFCE, RX-buffer tracking, and credit pool.
  • Send completion queues: jfs_jfc[] and optional send JFCE.
  • Bind context with imported remote target Jetty and remote target segments.
  • Queue settings: TP mode/type, RX/TX sizes, RX/TX depths, priority, ports, queue state, flow control, and UMQ context.

UB Resource Creation

  1. umq_ub_create_impl selects the device context by option->dev_info.
  2. It validates queue parameters and converts UMQ TP mode/type to URMA values.
  3. It creates or shares the receive context. Shared RQ mode points a sub-queue to another queue's JFR context.
  4. It creates send JFC/JFCE resources when needed.
  5. It creates the IO Jetty with urma_create_jetty, using a shared JFR.
  6. It allocates a small notify qbuf and creates flow-control resources when enabled.
  7. It records the queue in the device's umq_ctx_jetty_table so completions/events can find the queue.

UB Bind Info

UB bind info is serialized as TLV-style content and includes version info, device info, queue info, and optional flow-control info. Important payload includes local target segment information, transport mode, Jetty IDs, order type, target type, TP mode, TP type, token, notify buffer address, RX/TX depth, RX buffer size, queue state, PID, and namespace.

UB Bind Execution

  1. Deserialize peer bind info.
  2. Validate compatibility against the local queue.
  3. Import the remote IO Jetty with URMA. If flow control is enabled, import the remote flow-control Jetty too.
  4. In RC mode, bind the local Jetty to the imported remote target Jetty.
  5. For base API mode, prefill RX buffers after the Jetty connection is established.
  6. Store remote PID/namespace and import or track peer memory segment state.
  7. Initialize remote RX size and send initial flow-control credit request when enabled.
  8. Set queue state to QUEUE_STATE_READY.

Route Lookup

umq_get_route_list delegates to the backend umq_tp_get_topo. UB converts UMQ_TP_TYPE_RTP/CTP/UTP to UVS_RTP/UVS_CTP/UVS_UTP and calls uvs_get_path_set with source and destination bonding EIDs. The result is copied into umq_route_list_t with source/destination EIDs, source/destination port IDs, topology type, chip/die counts, and source/destination node IDs.

Route lookup detail. In this source, umq_ub_get_route_list_impl rejects TP types greater than or equal to UMQ_TP_TYPE_UTP, which means the current condition appears to accept RTP and CTP but not UTP. Check this before relying on UTP route lookup behavior.

IPC Backend Deep Dive

IPC mode is a local-process transport. It does not use URMA. It creates a shared-memory segment, builds a message ring inside it, and uses shared-memory qbuf pools so one process can pass buffer offsets to another.

IPC Create

  1. umq_ipc_ctx_init_impl initializes an ID allocator and shared qbuf subsystem.
  2. umq_ipc_create_impl validates queue mode, allocates umq_ipc_info_t, and fills local ring parameters from umq_create_option_t.
  3. The owner side creates a POSIX shared-memory object with shm_open, sizes it with ftruncate, and maps it with mmap.
  4. It creates a msg_ring inside the mapped memory.
  5. It initializes a local shared-memory qbuf pool in the data area after the transmit/ring region.

IPC Bind and Data Movement

IPC bind info contains the shared-memory name, total size, ring sizes, qbuf-pool data size, headroom, and buffer mode. The peer maps the same shared-memory object, creates a non-owner msg_ring, and initializes a remote shared-memory qbuf pool view.

IPC enqueue only accepts qbufs allocated from the local shared-memory pool. It converts qbufs to offsets with umq_shm_qbuf_enqueue and posts offsets into the local msg_ring. The peer dequeues by polling the remote message ring, resolving offsets through umq_shm_qbuf_dequeue, and returning a qbuf pointer in its own mapping.

IPC pro post/poll wrappers are present, but their implementation currently returns failure in umq_ipc_impl.c. The usable IPC data path in this source is the base enqueue/dequeue path.

IPC Notify and Interrupts

IPC notify uses the message-ring header: it sets cq_event_flag, increments pending_events, and wakes the peer with a futex. Interrupt wait/rearm/ack are implemented on top of these shared counters.

UBMM Backend Deep Dive

UBMM combines shared-memory style queueing with UB memory and control-plane support. It initializes an internal UB context, exports local OB memory, imports peer OB memory, then uses message-ring/qbuf mechanisms over imported memory.

UBMM Init and Create

  1. umq_ubmm_ctx_init_impl scans trans_info[] for UBMM or UBMM_PLUS entries.
  2. It initializes an underlying UB context once and stores the local CNA and UBMM EID from umq_init_cfg_t.
  3. It allocates UMQ I/O buffer memory and initializes the qbuf pool.
  4. umq_ubmm_create_impl creates an underlying UB queue with umq_ub_create_impl.
  5. It exports a local OB memory region, creates a local msg_ring, initializes a local shared qbuf pool in the exported memory, and allocates a notify qbuf.

UBMM Bind

UBMM bind info prepends UBMM-specific memory export data to the underlying UB bind info. It carries token ID, UBA, exported size, CNA, notify address, ring sizes, qbuf-pool parameters, and peer EID. Bind first calls umq_ub_bind_impl for the UB queue, then imports the peer OB memory with obmem_import_memory, creates a remote msg_ring, and initializes a remote qbuf pool view.

UBMM_PLUS Data Path

UMQ_TRANS_MODE_UBMM_PLUS implements base enqueue/dequeue:

  1. If a qbuf belongs to the UBMM queue, enqueue writes qbuf offsets to the local message ring.
  2. If a qbuf is from a global pool, the backend creates a small rendezvous qbuf containing reference-SGE information, records the original qbuf by message ID, and sends that descriptor through the shared-memory ring.
  3. The peer dequeues from the remote ring. If the dequeued item is rendezvous data, it issues a UB read through the underlying UB queue to fetch the actual data.
  4. umq_ubmm_notify_impl sends a UB immediate write to the peer notify buffer so the peer can wake up.
Plain UBMM caveat. In this source, the plain UMQ_TRANS_MODE_UBMM ops table has base enqueue/dequeue stubs that return failure/null, and pro post/poll stubs return failure. UBMM_PLUS is the path with implemented base data movement.

Flow Control

UMQ flow control is mainly implemented in the UB backend. It protects senders from overrunning the peer's receive buffers by granting and returning credits.

When UMQ_FEATURE_ENABLE_FLOW_CONTROL is set, each UB queue can have an IO Jetty and a flow-control Jetty. The flow-control path sends internal immediate messages for initial credit, credit requests, credit replies, credit returns, and acknowledgments. The sender checks available remote RX window before posting work requests. The receiver updates credit as RX buffers are posted and completions are consumed.

Config field Meaning
initial_credit Initial amount of receive capacity requested from the peer.
max_credits_request Upper bound for credit request size.
use_atomic_window Uses atomic maintenance for counters in concurrent send/receive scenarios.
credit_multiple Adaptive multiplier for credit request growth/shrink.
return_ratio Ratio controlling when held credit is returned.
min_reserved_credit Low-water threshold for requesting more peer credit.
timeout_ms Idle timeout after which held credit may be actively returned.

Interrupts, Events, and DFX

Queue interrupts

Public calls umq_rearm_interrupt, umq_wait_interrupt, and umq_ack_interrupt dispatch to backend base ops. Pro calls umq_interrupt_fd_get and umq_get_cq_event dispatch through pro ops.

UB maps these to URMA JFC/JFCE behavior. IPC maps interrupt-style notification to shared message-ring counters and futex wake/wait. UBMM delegates to the underlying UB queue for interrupt handling.

Async abnormal events

Device async-event APIs are transport-level calls: umq_async_event_fd_get, umq_get_async_event, and umq_ack_async_event. UB implements these through URMA async-event operations. Applications must acknowledge abnormal events before deleting affected objects.

DFX APIs

umq_dfx_api.c exposes flow-control stats, qbuf-pool stats, UMQ info, I/O packet stats, global perf, and transport perf start/stop/info calls. These either use common perf state or dispatch to umq_dfx_ops_t.

Device and route APIs

umq_dev_info_get, umq_dev_info_list_get, and umq_get_route_list are transport-backed. UB implements device info through URMA device/EID discovery and routes through UVS path-set lookup.

How to Choose the API Path

Use case Recommended UMQ path Reason
Simple message queue over UB or IPC Base API: umq_enqueue, umq_dequeue, umq_notify Less application-side queue management. UB base mode handles RX buffer refill automatically.
Explicit completion handling and RX buffer ownership Pro API: umq_post, umq_poll Application controls TX/RX posting and completion polling.
RDMA-like read/write/send operations over UB Pro API over UMQ_TRANS_MODE_UB or UB_PLUS umq_buf_pro_t carries opcode, flags, immediate data, remote SGE, and user context.
Local same-host IPC UMQ_TRANS_MODE_IPC base enqueue/dequeue Uses shared memory and offsets, avoiding network/URMA setup.
Large shared-memory style data with UB wakeup/rendezvous UMQ_TRANS_MODE_UBMM_PLUS base enqueue/dequeue Combines OB-memory import/export with message rings and UB reads for rendezvous buffers.
Latency and health inspection DFX APIs and async-event APIs Flow-control stats, packet stats, qbuf stats, queue info, and abnormal-event reporting are exposed separately from the data path.

Important Caveats from the Current Source

  • The public enum lists several transport modes, but local source implements UB, UB_PLUS, IPC, UBMM, and UBMM_PLUS. IB/UCP backend source was not present in this checkout.
  • IPC pro post/poll functions currently route to implementation stubs that return failure. IPC's implemented data path is base enqueue/dequeue.
  • Plain UBMM base and pro paths are mostly stubs in the current source. UBMM_PLUS has the implemented base enqueue/dequeue path.
  • UB base API prefills and refills RX buffers. UB pro API does not; the application must post RX buffers.
  • Bind info is application-transported. UMQ does not provide the out-of-band channel.
  • When UB token policy is enabled, bind info contains token-sensitive data and should be exchanged securely.
  • Route lookup depends on UVS topology availability. If UVS has no route/path data, umq_get_route_list cannot synthesize it.
