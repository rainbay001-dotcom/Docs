UMDK Repository Map

What Is In The UMDK Repo

UMDK is a user-space development kit for Lingqu UnifiedBus memory-semantic communication. The repo contains the base URMA stack, higher-level RPC and message-queue libraries, socket-compatible acceleration, distributed lock libraries, CAM Ascend communication operators, examples, tools, tests, packaging, and bilingual documentation.

This overview is based on the local checkout at /Volumes/KernelDev/umdk. The main CMake/RPM product path starts at src/CMakeLists.txt and builds URMA, ULOCK/DLOCK, URPC/UMQ, and USOCK/UMS. CAM is present in the same repository, but it has its own build scripts under build/cam and produces Ascend operator and Python artifacts.

Main Build Split

The repo has two practical build families. The main CMake/RPM family is driven by cmake ./src from umdk.spec. It can build all standard components or selected components using RPM conditionals such as --with urma, --with urpc, --with dlock, and --with ums. CAM is built separately using scripts under build/cam.

Main CMake/RPM Path

src/CMakeLists.txt adds urma, ulock, urpc, and usock based on BUILD_ALL and component switches. The RPM spec wraps this with dependencies, install paths, package names, and kernel-version options.

CAM Path

src/cam and build/cam target Ascend operator packaging. CAM produces an operator .run package and a Python wheel, and requires Ascend/CANN/Torch/Torch-NPU-style environment setup.

URMA

URMA is the base user-space memory-semantic API. It exposes application-facing APIs for remote memory and transport resources, plus a provider ABI for hardware providers. It is the foundation used by URPC, UMQ, DLOCK, and parts of UMS/CAM integration.

Subtree Contents
src/urma/common Shared utility code: lists, hash maps, bitmaps, strings, timing, barriers, and general helpers.
src/urma/lib/urma/core liburma.so core API implementation: devices, provider loading, control/data path APIs, command TLVs, formatting, logging, and performance hooks.
src/urma/lib/urma/core/include Public URMA headers such as urma_api.h, urma_provider.h, urma_types.h, and urma_opcode.h.
src/urma/lib/uvs UVS/TPSA user-space control library. Builds libtpsa.so and provides UVS APIs, private APIs, ioctl, TLV, UBAGG ioctl, and logging code.
src/urma/lib/urma/bond UB aggregation and bonding library. Handles multipath context, datapath conversion and scheduling, topology info, health check, link recovery, netlink, and segment handling.
src/urma/hw/udma User-space UDMA provider. Contains provider ops and resource code for buffers, Jetty, JFS, JFR, JFC, segments, TIDs, DBs, and control queue TP handling.
src/urma/tools urma_admin, urma_ping, and urma_perftest.
src/urma/examples urma_sample and example README.

URPC and UMQ

URPC is the RPC framework. UMQ is a neighboring message-queue subsystem in the same source root. The framework builds liburpc_framework.so; UMQ builds base and backend libraries such as libumq.so, libumq_buf.so, libumq_ub.so, and libumq_ipc.so. The source tree also contains a umq_ubmm backend.

Subtree Contents
src/urpc/include/framework Public URPC framework API, types, and errno headers.
src/urpc/framework URPC framework implementation: channel and queue core, control plane, data plane, managers, protocol, function registry, and framework logging.
src/urpc/include/umq Public UMQ base/pro APIs, DFX APIs, backend API headers, and transport-layer vtables.
src/urpc/umq UMQ implementation: front-end API dispatch, qbuf pools, DFX/perf, IPC backend, UB backend, UBMM backend, message rings, ID generator, and logging.
src/urpc/util URPC common utilities: bitmap, command queue, hash map, ID generator, epoll helper, socket helper, slab/pool, timer, thread, TLV, locks, and logs.
src/urpc/tools urpc_admin, urpc_framework_perftest, and umq_perftest.
src/urpc/examples URPC framework example, UMQ example, and UMQ connection setup tool.
src/urpc/third_party Bazel-related third-party wiring for OpenSSL, URMA, and zlib.

The URPC and UMQ HTML deep dives in this docs directory go deeper into these two subtrees. This overview keeps the focus on repository contents and boundaries.

ULOCK / DLOCK

The src/ulock/dlock subtree implements distributed lock and distributed object libraries on top of URMA-style connectivity. The release notes describe client management, server management, distributed lock operations, distributed object operations, heartbeat, failure recovery, secure transmission, and DFX.

Library Source Role
libdlockm.so src/ulock/dlock/lib/common Common descriptor, connection, cipher, Jetty manager, TCP/TLS, URMA context, logging, and utility code.
libdlockc.so src/ulock/dlock/lib/client Client APIs and wrappers for lock and object operations.
libdlocks.so src/ulock/dlock/lib/server Primary server, server manager, client entries, lock/object memory, server nodes, and wrappers.
Public headers src/ulock/dlock/lib/include dlock_client_api.h, dlock_server_api.h, dlock_types.h, and log header.
Examples src/ulock/dlock/examples Client, object-client, and primary-server examples.

USOCK / UMS

USOCK is the socket-compatible feature area. The concrete subtree in this checkout is UMS. It contains a kernel module, a preload library, an activation wrapper, an admin tool, and a user-space agent for secure token exchange.

Subtree or Artifact Purpose
src/usock/ums/kmod/ums UMS kernel module source: connection management, device integration, TX/RX IO, DFX, DIM, namespace helpers, and socket operation hooks for accept, bind, close, connect, listen, and release.
src/usock/ums/tools/ums-preload.c Builds libums-preload.so, used by the transparent replacement path.
src/usock/ums/tools/ums_run Wrapper tool for enabling UMS without modifying an application's code path.
src/usock/ums/tools/ums_admin UMS admin CLI.
src/usock/ums/ums_agent User-space daemon with config, systemd unit, logging, and token-exchange logic. The RPM spec describes it as a TLS 1.3 token exchange daemon between UMS kernel modules.

CAM

CAM stands for Communication Acceleration for Matrix on Ascend NPU. The local README describes EP communication, fused DeepMoE, KVCache transfer, RL weights transfer, and AFD communication as its feature areas. The current source tree has communication operator kernels, host tiling code, pybind bindings, PyTorch extension helpers, examples, and Catlass as a third-party submodule.

Subtree Contents
src/cam/comm_operator/ascend_kernels Ascend operator host and kernel code for dispatch layout, fused DeepMoE, MoE combine/dispatch normal, A2, SHMEM variants, notify dispatch, and shared utilities.
src/cam/comm_operator/pybind Python bindings for CAM operations plus PyTorch extension bridge code.
src/cam/examples Python examples and run scripts for fused DeepMoE and MoE dispatch/combine modes.
src/cam/third_party/catlass Third-party Catlass submodule.
build/cam Build scripts that generate CAM operator .run and Python .whl artifacts.

CAM is not in the top-level src/CMakeLists.txt component list. Treat it as a separate product path that requires an Ascend/CANN/PyTorch environment, not as part of the standard UMDK RPM CMake path.

How Components Depend On Each Other

  • URMA is the base memory-semantic transport library. Other UMDK libraries mostly build on it or interact with the same UB kernel stack.
  • UDMA provider is the user-space hardware provider under URMA. Its matching kernel provider and ubcore/uburma pieces live outside this repo in the kernel stack.
  • UVS/TPSA provides topology/session/control support used by the URMA/UB ecosystem.
  • URPC framework links against URMA and common URMA utilities, and exposes RPC semantics on top of URMA/UB resources.
  • UMQ provides message-queue APIs and backends, with UB, IPC, and UBMM source paths.
  • DLOCK links against URMA and URMA common utilities for distributed lock and object services.
  • UMS is a socket-compatible path with both kernel and user-space pieces.
  • CAM is separate from the main RPM path and targets Ascend operator/Python deployment.
