1. The original code — mask_fn helper

This is the pure function under analysis. It takes four input arrays (already-computed) and one compile-time constant, and returns a Boolean mask. No I/O, no tl.load / tl.store, no tl.program_id — just elementwise arithmetic.

@triton.jit
def mask_fn(q_attn_arg, k_attn_arg, q_offset, k_offset, TYPE: tl.constexpr):
    if TYPE == 1:
        triu_causal = (q_offset[:, None] <= k_offset[None, :])
        return ((triu_causal & ((q_attn_arg[:, None] == k_attn_arg[None, :]) |
                                (k_attn_arg[None, :] == 0))) |
                (q_offset[:, None] == k_offset[None, :]))
    if TYPE == 2:
        # ... different mask shape, not covered here ...
        pass
Bool variant = TYPE == 1: produces an upper-triangular causal mask with two side-channel exceptions (matching attention groups and a "zero-group" wildcard). All operations stay in i1 (boolean) until the very end of the surrounding kernel.

2. The kernel that calls mask_fn

Camodel needs a runnable kernel surface (loads + stores). The launcher wraps mask_fn and supplies real GM pointers:

@triton.jit
def mask_kernel(
    q_attn_ptr, k_attn_ptr, q_off_ptr, k_off_ptr, out_ptr,
    M: tl.constexpr, N: tl.constexpr, TYPE: tl.constexpr,
):
    q_off  = tl.load(q_off_ptr  + tl.arange(0, M))         # shape: (M,)
    k_off  = tl.load(k_off_ptr  + tl.arange(0, N))         # shape: (N,)
    q_attn = tl.load(q_attn_ptr + tl.arange(0, M))         # shape: (M,)
    k_attn = tl.load(k_attn_ptr + tl.arange(0, N))         # shape: (N,)

    mask   = mask_fn(q_attn, k_attn, q_off, k_off, TYPE=TYPE)  # shape: (M, N), i1
    out    = tl.where(mask, 1, 0).to(tl.int8)              # shape: (M, N), i8

    tl.store(out_ptr + tl.arange(0, M)[:, None] * N
                  + tl.arange(0, N)[None, :],
             out)
  • Four input vectors get loaded (each of length M or N).
  • mask_fn is inlined; it broadcasts the four vectors against each other to produce an M×N mask.
  • tl.where(mask, 1, 0) converts i1 → i32; .to(tl.int8) narrows to 8-bit before storing back to GM.

3. What is mask_fn actually computing?

An attention mask matrix answering, for each pair (query i, key j): "can query i attend to key j ?"

The full Boolean expression (TYPE == 1)

mask[i,j] =
   (triu_causal[i,j]
   AND ( q_attn==k_attn[i,j] OR k_attn==0[j] ))
 OR q_off==k_off[i,j]

Three rules a position must satisfy

  • Rule A (causal): q_off[i] ≤ k_off[j] — queries can only see keys at-or-after themselves.
  • Rule B (group match): q_attn[i] == k_attn[j] — same "attention group" — OR k_attn[j] == 0, the key is in the wildcard group "0".
  • Rule C (diagonal backstop): q_off[i] == k_off[j] — a query can always attend to its own position, regardless of A and B.

In PTO ISA: vreg V (i32) after VSEL.b32 V, V1, V0, P3 (select V1=1 or V0=0 per lane using mask P3), then narrowed to i8 by the store.

The bool-variant name comes from the fact that all comparisons and combinations stay in the i1 predicate domain until the very last tl.where — no intermediate i32 ↔ f16 ↔ i1 round-trips. The cast variant (TYPE-dependent path, not shown) does the same logic but in i32 arithmetic, producing far more vector ops.

Observations

  • Top row (i=0): query "sees" 3 of 4 keys. j=3 is blocked because q_attn[0]=1 ≠ k_attn[3]=2 and j=3 isn't a wildcard and 0≠3.
  • Diagonal stays on regardless: position (2,2) gets a T only from the diagonal backstop — neither triu·group nor wildcard would fire (group 2 wants to match key with attn=2, but column 2 has k_attn=1).
  • The wildcard column j=0 would be lit for every row in isolation, but the AND with triu_causal kills it for rows i ≥ 1 (since i ≤ j fails when j=0). Only the diagonal OR can rescue (i=0, j=0).
Why this mask in practice? Block-sparse attention with per-token "groups" and a global wildcard key — common in retrieval-augmented and sliding-window attention. Bool variant is the fast path because the entire compute stays in predicate-register arithmetic on the AIV.

12. How it maps to A5 (351x) hardware — one VLOOP iter

Each row of the M×N mask becomes one iteration of a 35-instruction hardware loop body. The 5 Triton-level comparisons collapse into 4 VCMPs + 3 predicate combiners + 1 select:

; Pass A: compute one row of the bool-mask matrix (32 i32 lanes per VL)
VLDI V2,[S6],#0,#0,#0          ; load k_offset pattern
VLDI V3,[S69],#16,#0,#1        ; load q_offset[i] (broadcast)
VLDI V4,[S68],#1,#3,#1         ; load q_attn[i] (broadcast)
VLDI V5,[S10],#0,#0,#0         ; load k_attn vector
VLDAS ULD0,[S12]               ; load q==k positional info
VADDS.s32 V2,V2,S8,P1          ; k_offset += pid*BLOCK   (launcher's arange+stride)
VCMP.LE.s32 P5,V3,V2,P1        ; P5 = (q_off[:,None] <= k_off[None,:])   ← triu_causal
VCMP.EQ.s32 P4,V4,V5,P1        ; P4 = (q_attn[:,None] == k_attn[None,:]) ← group match
VLDUI V6,ULD0,[S70],#0
MOVVP.b32 P3,V6,#0             ; P3 = (k_attn == 0)                       ← wildcard
PXOR P3,P3,P2,P1               ; (no-op for bool; sign-flip helper)
POR P3,P4,P3,P1                ; P3 = (group match) | (wildcard)         ← inner OR
PAND P3,P5,P3,P1               ; P3 = triu & (above)                     ← AND
VCMP.EQ.s32 P4,V3,V2,P1        ; P4 = (q_off == k_off)                    ← diagonal
POR P3,P3,P4,P1                ; P3 = (...) | diag                        ← top-level OR
VSEL.b32 V2,V1,V0,P3           ; V2 = where(P3, 1, 0)                     ← tl.where
VSTI V2,[S67],#16,#2,P1,#1     ; store row back to UB (later MTE3 → GM)
One row = one VL. With VL = 32 i32 lanes per RegTensor, an N=32 mask row fits exactly. M iterations of this loop produce the full M×N output. Predicate registers P3/P4/P5 carry the boolean intermediates — they never touch a vreg until the final VSEL. See a5_aiv_vector_parallelism.html §5 for the dual-issue back-end details.

13. Summary

What we walked through

  1. The mask_fn source — a 5-line Boolean expression over four input vectors.
  2. The mask_kernel wrapper that supplies GM pointers and converts i1 → i8 on store.
  3. The three semantic rules: causal (triu), group match + wildcard, diagonal backstop.
  4. A 4×4 example walked through the four elementwise comparison matrices.
  5. How they combine via the OR / AND / OR tree into the final mask.
  6. How the bool path stays in P-regs all the way to VSEL, mapping cleanly to ~15 PTO ISA instructions per row.

Key insights

  • Predicate-register friendly: every intermediate is a boolean. No i1 → i32 → fp16 round-trips like the cast variant.
  • Single VL per row: at M = N = 32, the entire row is one VLDI/VCMP issue, not a long sub-VL loop.
  • Diagonal OR is load-bearing: without it, queries whose group doesn't match any visible key would never self-attend. Row 2 in our example shows this.
  • Hardware mapping is 1-to-1: each Triton op becomes one or two PTO instructions; the compiler's job is mostly scheduling, not lowering.
