Ascend rvec — Register Vector ops

What the rvec prefix means in BiSheng / HiVM IR graphs and AscendC compiler dumps, and why it appeared with the A5/351x architecture.

TL;DR. rvec = Register Vector. It marks operations in the A5/351x Regbase vector model that work on the new RegTensor SIMD register (256 B / 2048 b wide), as opposed to the older Membase ops that operated directly on the Unified Buffer (UB). When you see rvecld, rvecst, rvecex, etc. in a graph, you are looking at the new register-stationary side of the pipeline introduced when the A5/351x switched away from UB-direct vector compute.

1. Why the prefix exists — the Membase → Regbase pivot

The 220x generation (910C) ran a Membase vector unit: the vector ALU consumed UB data directly, with the data path looking like UB → Vector ALU → UB. There were no architecturally visible vector registers; alignment was handled implicitly by the UB bank logic and the MTE engines.

The 351x (A5) generation switched to a Regbase vector unit. The vector ALU now consumes and produces RegTensor values, with the data path looking like UB → RegTensor → Vector ALU → RegTensor → UB. This change unlocked SIMT-style scheduling, register-stationary fused operations, and finer instruction-issue granularity, but it also created a new family of operations distinct from the legacy Membase ops.

The IR / ISA naming convention reflects this split:

PrefixOperates onEraExamples
v* UB-resident data, no register intermediary 220x (910C) Membase legacy vadd, vld, vst, vmul
rvec* RegTensor-resident data (or UB ↔ RegTensor staging) 351x (A5) Regbase native rvecld, rvecst, rvecadd, rvecex
Reading guidance. If a graph or compiler dump mixes v* and rvec* ops, the kernel is bridging Membase and Regbase paths — typically because some operations (MTE2 from GM, FixPipe to GM) are still memory-staged while compute happens in RegTensors. Concentrated rvec* clusters mean the compiler kept the data in registers across multiple ops, which is the win the Regbase architecture is designed to enable.

2. Decoding common rvec mnemonics

MnemonicExpansionWhat it doesNotes
rvecld Register Vector Load UB → RegTensor Backed by UnalignRegForLoad when source isn’t 256 B-aligned. Auto-stride via AddrReg.
rvecst Register Vector Store RegTensor → UB Backed by UnalignRegForStore when destination isn’t 256 B-aligned. Pairs with the vstu/vstus/vstur ISA-level unaligned-store family.
rvecex Register Vector Execute / Extract / Extension In-register transformation between RegTensors, no UB round-trip Could be a permute, a cast, an extract-subvector, a fused arithmetic, or a type-conversion op — the exact opcode depends on the dialect variant. No memory side-effect.
rvecadd / rvecmul / etc. Register Vector arithmetic RegTensor op RegTensor → RegTensor The native arithmetic surface of the Regbase ALU.
rvecmask Register Vector with mask Element-wise op gated by MaskReg MaskReg is VL/8 wide (256-bit on A5); 1 bit per element of the RegTensor.

3. The supporting register classes

Regbase introduces a small, named register file that rvec* ops consume:

Register classWidthRole
RegTensor256 B (VL = 2048 b)Holds vector operands; vector ALU reads/writes these.
MaskRegVL/8 bPer-element participation mask.
UnalignRegForLoad(buffer)Stages misaligned UB → RegTensor transfers.
UnalignRegForStore(buffer)Stages misaligned RegTensor → UB transfers.
AddrRegscalarAddress offset register with auto-stride increment per loop iteration.

The UnalignReg* registers are the alignment-merge buffers (the Ascend equivalent of PowerPC AltiVec’s lvsl/vperm pattern, Tensilica’s valign, or SPARC VIS’s alignaddr/faligndata). They exist because the vector ALU expects 256 B-aligned RegTensor operands, but UB data may start at any 32 B boundary — loads from an unaligned UB address need two adjacent aligned 256 B reads stitched together by a shift-merge.

4. Why this matters for reading a kernel graph

When inspecting a BiSheng / HiVM compiler dump, the density and sequencing of rvec* ops tells you a lot about how well the kernel is using A5’s Regbase model:

  • Long rvec* chains with no intervening rvecst: register-stationary compute — the compiler kept intermediates in RegTensor, which is the optimal path. Avoids UB read-write traffic.
  • Frequent rvecld ↔ rvecst oscillation: the kernel is spilling to UB between every operation. Either the compiler couldn’t keep things in registers (register pressure), or it’s a Membase-style kernel that hasn’t been ported.
  • Mixed v* and rvec*: legacy code path or a transition kernel. Common at MTE boundaries where UB-resident data has to come in via Membase-style paths before being lifted into RegTensors.
  • Heavy UnalignReg* use (visible as boundary metadata on rvecld/rvecst): the source/destination addresses aren’t 256 B-aligned. Aligning the upstream MTE descriptor or padding the tile would eliminate the alignment-merge cost.

If you see rvec* ops in a graph for a 220x kernel, double-check the target arch. The compiler will sometimes emit Regbase-shaped IR for analysis purposes even when the final lowering target doesn’t support it; the ISA-level lowering should reject this. Most production 220x compilation paths stay on v* mnemonics end-to-end.
