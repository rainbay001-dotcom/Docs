3. Two-pass body structure

What's the same in both passes

  • Same 4 predicate computations (A, B, C, F).
  • Same OR/AND/OR boolean tree.
  • Same VSEL.b32 → VSTI sequence to materialize and write the row.
  • Pass B reuses V4 (q_attn) from pass A — load is elided in lines 23–25.

What's different

  • Buffer addresses (S6 / S69 / S68 / S10 / S12 / S67 → S14 / S66 / — / S16 / S18 / S65).
  • S70 is reset between passes (line 14, line 9 for next iter).

Likely interpretation

  • Each iter produces one query row × two K-block-mask outputs (key-block unroll factor 2). The row index i is invariant across both passes within one iter; only the K-side advances.
  • Pass A and pass B operate on different key-blocks (different k_offset / k_attn ranges, different result buffers [S67] vs [S65]) — this is the "2 K-block unroll" pattern. Opt 9 / 10 / 15 would extend it further by packing both K blocks into the upper-32 lanes of one vreg.
  • Pinned by V4 not advancing: V4 is loaded only at line 6 (brc_b32 q_attn[i]) and there is no VLDI V4 at lines 23–25, so pass B reuses pass A's V4. Same query row ⇒ same q-side data; different K block ⇒ different k-side data; therefore same output row, two K-block-halves.

4. Pre-loop setup (lines 1–3)

LineInstructionWhat it doesWhy
1 PSET.b32 P1,#8 Set predicate P1 = the VL64 mask token (#8 ⇒ "first 64 i32 lanes valid"). Tells every subsequent vector op which lanes are live. P1 is the global "this many lanes" mask.
2 PSET.b32 P2,#15 Set predicate P2 = the ALLF token (#15 ⇒ "all 64 i32 lanes FALSE", i.e. all-zeros). Holds an all-zeros reference for the PXOR no-op on line 16 (X XOR 0 = X). Never changes value across the entire loop.
3 VLOOPV2_V310 S3,#35,#1,#1 Hardware loop: execute the next 35 instructions S3 times. Fields: S3 = runtime iter count (set by pre-loop code we don't see), #35 = body length, #1 = innermost layer, #1 = terminal at this layer. The _V310 suffix is the chip-revision encoding.
P1 and P2 are loop-invariant. They get set once before the VLOOP and never touched again across all 35 × S3 instructions. The same applies to V0 and V1 (the "1" and "0" sources for VSEL) if they are set elsewhere pre-loop — but we don't see their definitions in this snippet (Opt 6 target).

VLOOPV2 — the heart of the loop

VLOOPV2 is a hardware-managed loop: the AIV's RVECLP pipe maintains the iter counter internally, no per-iter conditional-branch is issued. The 35-insn body executes in a back-to-back stream with no front-end fetch overhead between iterations. This is why VLOOP-based kernels hit higher RVECEX utilization than equivalent Triton-emitted scalar loops.

6. Why q doesn't advance between pass A and pass B — per-block view

The two passes don't process two elements of the same row one-by-one — they process two key-blocks of the same row in parallel within each pass. Elements within a row land in different vreg lanes, not in different instructions. That's why q is invariant across the pass boundary.

Broadcast view — how q_attn[:, None] meets two different k_attn blocks

q_attn[:, None] is a column-broadcast: same scalar across each row. k_attn[None, :] is a row-broadcast: same row across all output rows. With 2-K-block unroll, pass A and pass B see the same q_attn[:, None] column but two different k_attn[None, :] rows (one per K block).

What "element-by-element" actually means here

Pass A's VCMP.EQ.s32 P4,V4,V5,P1 at line 12 is one instruction that produces 32 results at once:

P4_A[0] = (V4[0] == V5[0])  ; q_attn[2] == k_attn_A[0]
P4_A[1] = (V4[1] == V5[1])  ; q_attn[2] == k_attn_A[1]
P4_A[2] = (V4[2] == V5[2])  ; q_attn[2] == k_attn_A[2]
…
P4_A[31]= (V4[31] == V5[31]) ; q_attn[2] == k_attn_A[31]

All 32 lane comparisons happen in parallel (one VCMP issue, one cycle of issue, ~8 cycles of latency). The element index j varies across lanes, not across instructions.

Because V4 is brc_b32-loaded (#dist=3), all 32 lanes of V4 hold the same scalar q_attn[2] — that scalar gets compared against each of V5's 32 different lane values.

Per-iter advance accounting

  • V4 (q_attn): S68 += 4 B per iter (one i32 scalar). Brc_b32 broadcasts it to all 32 lanes. Pass B reuses what pass A loaded.
  • V3 (q_offset): S69 += 512 B per iter (full pre-replicated vreg). Pass B should reuse pass A's V3 but currently doesn't — that's Opt 11.
  • V5 / V2 (K-side): no per-iter advance (#p=0). The pass switch uses different base addresses (S10 vs S16, S6 vs S14), not a post-increment.

8. Tiny example for q_offset / k_offset — same pattern, two VCMPs per pass

Same blocks-first structure as slide 6, but now for the offset predicates. Two key differences from q_attn / k_attn:

  • V3 uses full-VL load (#dist=0), not brc_b32. The launcher pre-replicates each row's q_offset[i] across all VL lanes in a UB tile, then V3 loads that whole row per iter. Post-increment is +512 B per iter (16 alignment-units × 32 B) — much larger than V4's +4 B.
  • Two VCMPs per pass use V3: VCMP.LE for A = triu_causal (line 11 / 28) and VCMP.EQ for F = diagonal (line 19 / 35). So V3 is consumed four times per iter (twice per pass × 2 passes), loaded only once — high leverage for Opts 4 / 11 / 12.

Step 0 — what [:, None] and [None, :] actually mean

  • MTE3 (UB → GM DMA)
  • SIMD VF (where VLOOP body fires)
  • SIMT VF (idle in mask_kernel)

Within one launch, MTE2/MTE3/SIMD-VF already overlap intra-iter (this is what assembler.as's unroll-by-2 over key blocks achieves). But across back-to-back launches, launch N+1's pre-loop MTE2 prefetch can be issued during launch N's VLOOP compute — hiding ~700 cyc of operand load under the previous launch's compute. The current launcher serialises launches.

35. Opt 22 — Align operand pointers to 256 B Tier B ~−50–100 cyc / launch

§6.19 • ND-DMA Cache alignment • applies to: launcher (no kernel change)

The mechanism

Between L2 Cache and UB sits the ND-DMA Cache, which accelerates non-aligned and strided GM accesses. Its cacheline granularity is 256 B. If an operand pointer (q_attn_ptr, k_attn_ptr, q_offset_ptr, k_offset_ptr) lands at a non-256 B offset, the 32-element operand spans two cachelines and MTE2 pays two cacheline accesses instead of one. Aligning at allocation time eliminates the cross-line split.

Where misalignment happens

  • Triton tensor allocator already 256 B-aligns large tensors
  • Slices of larger tensors (e.g. q_attn for a specific head/batch) may not land at 256 B
  • Each misaligned operand costs ~tens of cycles per launch — 4 operands ⇒ up to ~100 cyc

36. Opt 23 — C-buffer 32 B alignment Tier A −160 ops

§6.20 • Eliminate unaligned-load preamble • applies to: assembler.as γ baseline (CANN 9.0.0 uses α so doesn't have this problem)

The mechanism — C is at an unaligned offset

; CURRENT: 4-instruction unaligned-load preamble per iter
VLDAS ULD0,[S12]      ; setup unaligned addr ctx
SMOV.b32 S70,S12        ; copy base addr
VLDUI V6,ULD0,[S70],#0 ; load via offset
SMOV.b32 S70,S18        ; pass-B addr reset

The 4-step dance is needed because the C-buffer in UB starts at a non-32 B offset. VLDAS sets up an "unaligned context" register (ULD0); VLDUI uses that context for a cross-bank read.

The fix — align C-buffer to 32 B

Pure launcher-side change: round the C-buffer's UB starting address up to the next 32 B boundary. Then a vanilla VLDI works.

; OPTIMIZED: 1 instruction replaces 4
VLDI V6,[S12],#0,#0,#0   ; aligned load
