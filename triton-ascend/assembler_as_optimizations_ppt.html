2. Two-pass body structure

What's the same in both passes

  • Same 4 predicate computations (A, B, C, F).
  • Same OR/AND/OR boolean tree.
  • Same VSEL.b32 → VSTI sequence to materialize and write the row.
  • Pass B reuses V4 (q_attn) from pass A — load is elided in lines 23–25.

What's different

  • Buffer addresses (S6 / S69 / S68 / S10 / S12 / S67 → S14 / S66 / — / S16 / S18 / S65).
  • S70 is reset between passes (line 14, line 9 for next iter).

Likely interpretation

  • Each iter produces two output rows simultaneously (unroll factor 2).
  • Pass A and pass B operate on different key-blocks (different k_offset / k_attn ranges) — implementing the "2 row-tile pack" pattern that Opt 9 tries to take further.

3. Pre-loop setup (lines 1–3)

LineInstructionWhat it doesWhy
1 PSET.b32 P1,#8 Set predicate P1 = the VL64 mask token (#8 ⇒ "first 64 i32 lanes valid"). Tells every subsequent vector op which lanes are live. P1 is the global "this many lanes" mask.
2 PSET.b32 P2,#15 Set predicate P2 = the ALLF token (#15 ⇒ "all 64 i32 lanes FALSE", i.e. all-zeros). Holds an all-zeros reference for the PXOR no-op on line 16 (X XOR 0 = X). Never changes value across the entire loop.
3 VLOOPV2_V310 S3,#35,#1,#1 Hardware loop: execute the next 35 instructions S3 times. Fields: S3 = runtime iter count (set by pre-loop code we don't see), #35 = body length, #1 = innermost layer, #1 = terminal at this layer. The _V310 suffix is the chip-revision encoding.
P1 and P2 are loop-invariant. They get set once before the VLOOP and never touched again across all 35 × S3 instructions. The same applies to V0 and V1 (the "1" and "0" sources for VSEL) if they are set elsewhere pre-loop — but we don't see their definitions in this snippet (Opt 6 target).

VLOOPV2 — the heart of the loop

VLOOPV2 is a hardware-managed loop: the AIV's RVECLP pipe maintains the iter counter internally, no per-iter conditional-branch is issued. The 35-insn body executes in a back-to-back stream with no front-end fetch overhead between iterations. This is why VLOOP-based kernels hit higher RVECEX utilization than equivalent Triton-emitted scalar loops.

4, 10 VLDI V2,[S6] + VADDS V2,V2,S8,P1 k_offset = arange(0, BLOCK_N) + pid_n * BLOCK_N — the launcher reconstructs k_offset on-chip from a constant arange + per-block scalar.

6 VLDI V4,[S68],#1,#3,#1 q_attn[i] — brc_b32 broadcast (#dist=3): read 1 i32 (4 B) and replicate to all 64 lanes. Per-iter post-increment of 1 × 4 B = +4 B on S68.

21 VSEL.b32 V2,V1,V0,P3 tl.where(mask, V1, V0) where V1 holds 1's, V0 holds 0's.

5. Loop-invariants exposed — what the body needlessly recomputes

Inside the VLOOP, certain instructions produce bit-identical results every iteration. They're loop-invariants the compiler failed to hoist. These are the highest-leverage optimization targets.

 4  VLDI V2,[S6],#0,#0,#0           ; #p=0 ⇒ same V2 every iter
 5  VLDI V3,[S69],#16,#0,#1             ; per-iter — real work
 6  VLDI V4,[S68],#1,#3,#1              ; per-iter — real work
 7  VLDI V5,[S10],#0,#0,#0          ; #p=0 ⇒ same V5 every iter
 8  VLDAS ULD0,[S12]                  ; same ULD0 every iter
 9  SMOV.b32 S70,S12                   ; same S70 every iter
10  VADDS.s32 V2,V2,S8,P1                ; S8 const ⇒ same V2 every iter
11  VCMP.LE.s32 P5,V3,V2,P1               ; depends on V3[i] — real
12  VCMP.EQ.s32 P4,V4,V5,P1               ; depends on V4[i] — real
13  VLDUI V6,ULD0,[S70],#0           ; same C every iter
14  SMOV.b32 S70,S18                  ; pass-B reset, but invariant
15  MOVVP.b32 P3,V6,#0                ; C produced every iter
16  PXOR P3,P3,P2,P1                 ; XOR with all-ones = no-op
…

Why these are invariant

  • #p=0 VLDIs (lines 4, 7): k_offset / k_attn have no per-row data — they're full-row vectors that broadcast across the matrix.
  • VADDS (line 10): S8 = pid * BLOCK_N is set pre-loop; never changes across iterations.
  • VLDAS+VLDUI+MOVVP (lines 8, 13, 15): produce C[j] = (k_attn[j] == 0), which is row-invariant (column-only).
  • PXOR (line 16): P2 = ALLF, so P3 XOR ALLF = ~P3 — but the subsequent POR with another always-recomputed C makes it functionally a no-op. Pure dead code.
11 of 22 lines in pass A are invariant. Every iter wastes RVECLD bandwidth and RVECEX cycles redoing setup. Opts 1–6 + 11–12 are entirely about hoisting these out of the loop.

7. Tier-A loop-invariant hoists (Opts 1–6, 11–12)

#OptimizationSavingMechanism
1 §6.6 Inline-recompute C −34 ops Skip the UB-buffered C; compute it fresh each iter as VCMP.EQ V_kattn, V_zero. Trades pre-loop VCMP+vstore plus per-iter VLDUI+MOVVP (2 ops/iter) for one in-body VCMP — net −1 op/iter + −2 pre-loop, −34 over 32 iters.
2 §6.6 Hold C in a P-reg −65 ops Compute C once pre-loop into P6; body reads P6 directly. Eliminates all 5 C-setup insns per iter.
3 §6.7 brc_b32 stream q_offset −16 ops Switch V3 load from full-VL+per-iter to brc_b32 mode like V4 — one scalar advance per iter, not a full vreg.
4 §6.8 Hoist V2 (VLDI + VADDS) −62 ops V2 = k_offset+pid·BLOCK is constant across iters; move out of VLOOP entirely. Also saves 8 KB UB read bandwidth.
5 §6.8 Hoist V5 (k_attn) −31 ops Same reasoning as Opt 4 but for V5. With #p=0, V5 is bit-identical every iter.
6 §6.8 Hoist V0/V1 + scalar setup ~−160 ops V0, V1 (constants 0 and 1 used by VSEL) and S70 setup are also invariant. Big total because each is a few ops × 32 iters.
11 §6.10 Share V3 across passes −48 ops Pass A's VLDI V3,[S69] and pass B's VLDI V3,[S66] load from different addresses that hold identical q_offset data (both passes process the same query row). Reuse pass-A's V3 in pass B — same trick already applied to V4 (q_attn).
12 §6.11 Hoist F = q_off == k_off (packed) −32 to −64 ops F[i,j] = (q_off[i] == k_off[j]) does vary with the row i, so it isn't loop-invariant per row — but the whole 1024-bit tile of F is precomputable once. Hoist to a packed predicate (ppack pair of _b32 regs) or UB bitvector, then read per-row. Eliminates the 2 in-body VCMPs (lines 19 + 35). CANN 9.0.0 already does this; assembler.as doesn't.
Combined Tier-A hoists ≈ −450 ops across the 32-iter VLOOP. For RVECEX-bound work this is ~−350 cyc (at the empirical 1.28 ops/cyc rate). The CANN 9.0.0 build of mask_kernel already applies Opts 4, 5, 6, 12; assembler.as is from an older / different compiler that didn't.

8. Tier-A μarch alignment (Opts 13, 14, 22, 23)

#OptimizationSavingMechanism
13 §6.12 Bank-aware UB layout 0–10 cyc/iter Place V3, V5, C-buf so they live in different bank-groups → simultaneous reads on UB's 2R/group budget without conflicts.
14 §6.12 Co-locate C-reload + result-VSTI 0–5 cyc/iter Aligning C-bytes and output-row addresses so that VSTI write doesn't bank-conflict with the next iter's C-load.
22 §6.19 Align operand pointers to 256 B ~−50-100 cyc/launch Allows ND-DMA Cache to coalesce GM bursts without padding lanes → MTE2 setup gets shorter and L2-line-friendly.
23 §6.20 C-buffer 32 B alignment −160 ops If C-buffer starts at a 32 B boundary, replace the VLDAS+SMOV+VLDUI 3-op unaligned-load sequence with a single VLDI. Eliminates ~5 ops per iter × 32.
These come from reading atlas_ascendc_best_practices_10_00021 + atlas_ascendc_10_00065 — the UB bank-port and ND-DMA Cache rules. None of them require any kernel-code change in mask_fn; they're pure launcher-side fixes (buffer allocation alignment / placement).

Why Opt 23 specifically is a big win for assembler.as

Lines 8, 13, 14, 15 (VLDAS+VLDUI+SMOV+MOVVP) exist only because the C-buffer wasn't aligned to 32 B. With launcher-side alignment, the entire 4-instruction unaligned-load preamble per iter collapses to a single VLDI V6, [S12], #0, #0, #0 followed by the already-present MOVVP. 3 ops/iter × 32 iters = ~96 ops, plus the SMOV scalar work, brings total saving to ~160.

9. Tier-A cast-variant collapse (Opts 18, 19, 20)

These apply to the cast variant of mask_fn (TYPE=2 path or any version with explicit .to(tl.int32) conversions), not directly to the bool assembler.as. But they're important because they show how the cast variant could converge back to the bool path's efficiency.

#OptimizationSavingWhat it removes
18 §6.16 Eliminate i1 → f16 → i32 widenings −192 ops The cast variant lowers tl.where(bool, ...) via float intermediates. Six VCASTs per iter × 32 = 192 wasted ops.
19 §6.16 Skip i32→f32→i1→f16→i8 final chain −96 ops End-of-row narrowing goes through 4 intermediate types. Direct VSEL.b32 → b8 narrow does the same in one step.
20 §6.16 Full migration to P-reg algebra converges to bool Keep all booleans in P-regs (as Opts 2, 12 already do). The cast variant becomes structurally identical to the bool kernel.
Cast → bool convergence. Once Opts 18 + 19 + 20 apply, the cast variant has the same op count and same critical path as bool. The reason it's slower today is purely the intermediate-type widening; the logical work is identical.

Where the widenings come from

# cast variant (TYPE=2-like, schematic):
mask_i1   = (q_off <= k_off)
mask_f16  = mask_i1.to(tl.float16)        # ← VCAST i1 → f16
mask_i32  = mask_f16.to(tl.int32)         # ← VCAST f16 → i32
combined  = mask_i32 & group_i32                # ← vector AND in i32
…
out_i1    = combined.to(tl.int1)          # ← VCAST i32 → i1 (back where we started!)
out_f16   = out_i1.to(tl.float16)
out_i8    = out_f16.to(tl.int8)            # ← 3-step narrow for the store

None of these intermediates carry meaningful information that isn't in mask_i1. The compiler shouldn't need this many VCASTs — but the cast lowering path inserted them conservatively.

11. Deep dive — Opt 2: Hold C in a P-reg −65 ops

Current body (γ baseline — camodel-style aligned spill)

; pre-loop: 2 ops (compute C once, spill to UB byte buffer)
VCMP.EQ.s32 P_C,V_kattn,V_zero,P1
; vstore P_C → UB byte buffer at S_C

; per iter: 2 ops (reload C from UB, narrow i8→i1)
VLDUI V6,ULD0,[S70],#0       ; load C-bytes
MOVVP.b32 P3,V6,#0            ; V6 → P3

The unaligned-load preamble (VLDAS + SMOV) in assembler.as lines 8–9 is a separate overhead handled by Opt 23 — it doesn't affect the Opt 2 accounting.

Optimized (β — hold C in a P-reg)

; pre-loop: 1 op (compute C once into a preserved P-reg)
VCMP.EQ.s32 P6,V5,V_zero,P1      ; P6 = (k_attn == 0)

; per iter: 0 C-related ops — P6 is read directly
POR P3,P4,P6,P1             ; B | C, reads P6 directly

Cost is one P-register held live across the loop body (P6). The kernel already uses P1–P5; A5 has at least 8 P-regs, so reserving a 6th is trivial.

Why it requires P-reg preservation

P-regs (P0..P7) are preserved across VLOOP iterations by hardware contract — the VLOOP wrapper doesn't reset them. This is what makes P1 = VL64 and P2 = ALLF stick across all 35 × S3 instructions. Same mechanism applies to a "C in P6" strategy: set once pre-loop, read N times in body.

Why hivmc-a5 didn't do this. The C → P6 path requires recognizing that the "C-bytes" buffer in UB is the boolean result of a single pre-loop VCMP. The compiler instead chose to materialize C as i8 bytes in UB, then re-materialize as i1 in body — over 32 iters this is a much more expensive choice.

12. Deep dive — Opt 4: Hoist V2 (VLDI + VADDS) −62 ops

What V2 contains

V2 holds k_offset[None,:], computed as arange(0, 32) + pid_n × BLOCK_N. Both inputs are constant: arange(0, 32) is a fixed UB pattern, pid_n × BLOCK_N is the scalar S8 assigned pre-loop.

; CURRENT body, lines 4 + 10 run every iter:
VLDI V2,[S6],#0,#0,#0        ; load arange(0, 32)
…
VADDS.s32 V2,V2,S8,P1            ; V2 += S8 (constant)

; Result: V2 = [pid*BLOCK, pid*BLOCK+1, …, pid*BLOCK+31]
; This value is identical for iter 0, 1, 2, …, 31

Hoist out of the loop

; PRE-LOOP (2 ops, once per kernel):
VLDI  V2,[S6],#0,#0,#0
VADDS.s32 V2,V2,S8,P1

; BODY: V2 just sits there, used by VCMP.LE and VCMP.EQ
VCMP.LE.s32 P5,V3,V2,P1
VCMP.EQ.s32 P4,V3,V2,P1

Pipe-level impact

VLDI hits RVECLD; VADDS hits RVECEX. Hoisting saves:

  • RVECLD: −32 ops (large headroom anyway — pipe is 18% utilized).
  • RVECEX: −32 ops × 7.5 cyc/op pipelined ≈ −25 cyc on the critical path.
  • UB read ports: 32 × 8 reads = 256 fewer port-cycles.

13. Deep dive — Opt 23: C-buffer 32 B alignment −160 ops

The problem — C is at an unaligned offset

; CURRENT: lines 8, 9, 13, 14 — 4-instruction
; unaligned-load preamble per iter
VLDAS ULD0,[S12]      ; setup unaligned addr
SMOV.b32 S70,S12        ; copy base addr
VLDUI V6,ULD0,[S70],#0 ; load using offset
SMOV.b32 S70,S18        ; (pass-B addr reset)

The 4-step dance is needed because the C-buffer in UB starts at an offset that isn't 32-byte aligned. VLDAS sets up an "unaligned context" register (ULD0) holding the base+shift, and VLDUI uses that context to do a cross-bank read.

The fix — align the C-buffer to 32 B

Pure launcher-side change: when allocating C-buf in UB, round the starting address up to the next 32 B boundary. Then a vanilla VLDI works.

; OPTIMIZED: 1 instruction
VLDI V6,[S12],#0,#0,#0  ; aligned load, no helper

Cost-benefit ledger (matches walkthrough §6.20)

Ops eliminatedPer passPer VLOOP iterTotal (T=32)
VLDAS (unaligned ctx setup)1264
SMOV (addr snapshot for VLDUI)1264
SMOV (mid-iter reset between passes)—132

T = 32 in the walkthrough's accounting counts each VLDAS/SMOV occurrence across both passes (16 VLOOP iters × 2 passes = 32 occurrences).

Pipe-level breakdown of saved ops

  • RVECLD: −64 VLDAS ops (these run on the load pipe)
  • RVECSU (scalar): −96 SMOV ops (64 addr-snapshots + 32 mid-iter resets)
  • VLDUI is a 1:1 swap to VLDI — same op count, but aligned VLDI is faster per op (~10 cyc vs ~10-12 cyc), so a small additional latency win.
  • Both pipes have lots of slack in the current trace (RVECLD 18%, RVECSU 2% utilized) — so the cycle saving on the critical path is ~−40-60 cyc, not the full −160.
Zero compiler change needed. Opt 23 is a pure launcher fix: just bump the C-buffer base address in UB. No hivmc-a5 update, no rewriting of mask_fn. The cleanest "free win" of the 23.

14. Deep dive — Opt 21: Cross-launch prefetch −700 cyc / launch-pair

The setup — 4 AIV instruction queues, run concurrently

The AIV has 4 independent instruction queues:

  • MTE2 (GM → UB DMA)
  • MTE3 (UB → GM DMA)
  • SIMD VF (the VLOOP body fires here)
  • SIMT VF (idle in mask_kernel)

They issue concurrently — an MTE2 burst can run while the SIMD-VF queue executes a vector op.

Current single-launch timeline

launch N:  | MTE2 prefetch (~700 cyc) | VLOOP (~1400 cyc) | MTE3 |
                                        ╰─ AIV idle here? No — but cross-launch is

With cross-launch prefetch

launch N:    | MTE2 prefetch | VLOOP body                | MTE3 |
launch N+1:                  | MTE2 prefetch              | VLOOP body | MTE3 |
                              ╰──── overlap region (~700 cyc) ────╯

Why the AIV can do this

MTE2 and SIMD-VF queues are dispatched independently. A SIMD-VF op doesn't block an MTE2 op (unless they hit the same UB bank-group, which is what Opt 13 alignment prevents). So issuing launch N+1's MTE2 setup while launch N's VLOOP is executing is structurally free.

The catch

  • Requires double-buffering of UB regions: launch N owns one slot, launch N+1's prefetch goes to another slot.
  • Requires compiler-emitted MTE2 issue from outside the kernel: today the kernel sees one launch at a time; cross-launch is a launcher-level pipeline.
  • This is the AscendC TQue + BUFFER_NUM=2 pattern (see ascend_mte_data_movement.md §2).
